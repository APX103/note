A2A Agent Platform Design

A modular microservices platform for managing A2A-protocol-compatible agents, built with go-zero and a modern SaaS-style static frontend.

Goals

  • Agent-agnostic: Any service implementing the A2A protocol can register and participate.
  • Group isolation: Each agent belongs to one group. Messages are scoped to groups at the Gateway layer.
  • Dual-mode tasks: Users can initiate tasks from the web UI, and agents can call each other autonomously via A2A. Both modes share unified trace tracking.
  • Full-chain tracing: Every task generates a trace_id that flows through Gateway → Queue → Agent call → Response. Frontend renders a DAG visualization of the message graph.
  • Observable: Prometheus metrics + Grafana dashboards for agent health, message throughput, task latency, and queue depth.
  • User-friendly: Modern SaaS-style UI (Vue3 + TailwindCSS), simple agent onboarding, clear documentation.

Three go-zero services, one embedded Vue3 SPA, and a Prometheus/Grafana observability layer.

Services

Gateway Service (go-zero API Server)

The single HTTP entry point for both the frontend and agents.

Agent Registration

  • Agent POSTs to /api/v1/agents/register with name, endpoint, group, auth_token.
  • Gateway fetches /.well-known/agent.json from the endpoint, validates the Agent Card, and stores it.
  • Heartbeat monitor probes each agent at a configurable interval (default 30s). On repeated failure, agent status flips to offline.

A2A Proxy

  • Agent-to-agent calls arrive as A2A JSON-RPC at /a2a/{agent_name}.
  • Gateway resolves the target, enforces group isolation (sender group must match target group), generates trace_id, stores the message, and publishes to go-queue.
  • SSE proxy at /a2a/{agent_name}/tasks/{taskId} streams long-running task updates.

Auth & Group Filter

  • Every message dispatch checks that sender.group_id == target.group_id.
  • Cross-group messages are rejected with HTTP 403.

Prometheus Endpoint

  • Exposes /metrics for Prometheus scraping.
  • go-zero's built-in PrometheusHandler covers HTTP metrics; custom middleware emits A2A-specific counters and histograms.

Message Service (go-zero RPC Server)

Consumes from go-queue and handles message delivery.

Message Processing

  • Consumes events from topic a2a.messages.{group_id}.
  • Forwards the A2A request to the target agent's endpoint via HTTP/SSE.
  • On response: stores the message in MySQL, publishes the response to the queue for async propagation.
  • On failure: retries with exponential backoff (1s initial, 2x factor, 60s max, 5 attempts).

Message Storage

  • All messages (request, response, delegate, notify) are persisted to the messages table with trace_id for trace reconstruction.

Platform Service (go-zero RPC Server)

Manages agent metadata, groups, tasks, and trace queries.

Agent Management

CRUD for agents and groups. Group membership controls message scope.

Task Lifecycle

Tracks tasks from submitted → working → completed/failed. Supports user-initiated and agent-initiated tasks.

Trace Query

Given a trace_id, fetches all messages from the messages table ordered by created_at, constructs DAG nodes and edges via parent_message_id, and returns the graph structure for frontend visualization.

Frontend

Vue3 single-page application, built with Vite, styled with TailwindCSS. Embedded as static files in the go-zero binary.

Pages

  1. Dashboard — Overview cards (online agents, tasks today, avg latency, failures) + recent task list + quick actions.
  2. Agent Management — Agent list with health status, registration form, group assignment, Agent Card viewer.
  3. Task Center — Create task (select target agent/group, input content, set priority), task list with filters, task detail with SSE streaming output.
  4. Message Flow — Per-task DAG visualization (D3.js), message timeline, artifact viewer, group scope indicator.
  5. Group Management — Group CRUD, member list, message retention settings, activity summary.
  6. Platform Settings — Prometheus endpoint config, Grafana links, message retention policies, health check intervals, access tokens.

Tech Stack

  • Vue 3 (Composition API)
  • Vite (build tool)
  • TailwindCSS (styling)
  • D3.js or vis.js (DAG graph visualization)
  • Pinia (state management)

Core Data Flows

Agent Registration

  1. Agent starts up, POSTs to /api/v1/agents/register with name, endpoint, group, auth_token.
  2. Gateway validates auth_token.
  3. Gateway fetches {endpoint}/.well-known/agent.json — the Agent Card.
  4. Gateway parses and validates the Agent Card (name, description, skills, url, authentication).
  5. Gateway stores agent record + Agent Card in MySQL, registers in Redis active set.
  6. Gateway starts heartbeat monitor (default 30s interval), probing {endpoint}/.well-known/agent.json.
  7. Returns success — agent is now discoverable and routable.

Message Dispatch

Mode A — User initiates from Web UI:

  1. User submits task via /api/v1/tasks with target group, target agent, content, priority.
  2. Gateway resolves target agent from Redis/MySQL.

Mode B — Agent-to-Agent A2A call:

  1. Agent sends A2A JSON-RPC to /a2a/{agent_name}.
  2. Gateway parses the request and resolves the target.

Common path (both modes):

  1. Gateway generates trace_id (UUID), propagated via HTTP header.
  2. Gateway enforces group isolation: sender.group_id must equal target.group_id. Returns 403 on mismatch.
  3. Gateway stores task + initial message in MySQL.
  4. Gateway publishes to go-queue topic a2a.messages.{group_id}.
  5. Message Service consumes from queue, forwards the request to the target agent via A2A HTTP/SSE.
  6. Message Service stores the response in MySQL, emits trace events to Prometheus.

Trace Query & DAG Visualization

  1. Frontend requests GET /api/v1/traces/{trace_id}/graph.
  2. Platform Service queries all messages with that trace_id, ordered by created_at.
  3. Platform Service constructs DAG: each message is a node, parent_message_id defines edges.
  4. Frontend renders the graph with D3.js/vis.js — interactive nodes with timestamps, latency annotations, and status colors.

Deployment

Single docker-compose.yaml brings up all services:

  • MySQL 8
  • Redis 7
  • go-queue (kafka or rabbitmq, selectable via config; docker-compose defaults to kafka)
  • Prometheus
  • Grafana (with pre-built dashboards)
  • Gateway Service (port 8080)
  • Message Service (internal RPC)
  • Platform Service (internal RPC)

The Vue3 frontend is built (npm run build) and the static output is embedded into the Gateway service binary. Users access everything through a single port.

Agent Onboarding

Agents built with Google ADK can use to_a2a() or adk api_server --a2a to expose an A2A-compatible endpoint. Agents built with any other framework just need to implement:

  1. GET /.well-known/agent.json — return a valid Agent Card.
  2. POST / — accept JSON-RPC 2.0 tasks/send requests and return task responses.
  3. GET /tasks/{taskId} — support SSE streaming for long-running tasks.

The platform provides a quick-start guide with curl examples and a sample Agent Card.
