📖 A2A 协议调研

什么是 A2A？

A2A（Agent-to-Agent） 是 Google 推出的开放协议，旨在让不同厂商、不同框架构建的 AI Agent 能够相互发现、协作完成任务。它不替代 MCP（Model Context Protocol），而是与 MCP 互补：MCP 解决 Agent 与工具/资源的连接，A2A 解决 Agent 与 Agent 的连接。

核心定位：A2A 是 "Agent 的 HTTP" —— 标准化、无状态、可扩展，任何实现了 A2A 接口的 Agent 都可以被其他 A2A 客户端调用，无需了解对方内部实现。

协议核心概念

AgentCard
Agent 的"名片"，包含 name、description、version、capabilities、skills、supportedInterfaces。客户端通过 GET /.well-known/agent.json 获取，实现零配置发现。
Task
任务状态机：SUBMITTED → WORKING → INPUT_REQUIRED / COMPLETED / FAILED。每个 Task 有唯一 ID，支持通过 context_id 关联会话上下文。
Message
消息载体，包含 role（user / agent）和 parts（text、file、data 等）。A2A 支持多模态，但文本是最小公共子集。
Streaming
通过 SSE 实时推送 Task 状态变更、Artifact 产出和 Message 响应，避免长轮询，提升交互体验。

🎯 为什么做这个平台

痛点分析

❌ Agent 孤岛
各团队用不同框架（LangChain、LlamaIndex、自研）造 Agent，彼此无法通信。想让一个 Agent 调用另一个 Agent，只能写硬编码 HTTP 调用，耦合严重。
❌ 缺乏统一管控
Agent 跑在不同进程、不同机器，没有统一的注册中心、没有健康检查、没有状态视图。运维靠人肉和日志 grep。
❌ 调试困难
Agent 间通信是黑盒：消息发出去有没有到？对方怎么处理？返回了什么？中间有没有重试？出了问题只能加 print。
❌ 异构 LLM 难以协同
有的 Agent 用 GPT-4，有的用 Claude，有的用内网模型。没有统一协议把它们串成流水线，各模型优势无法互补。

平台价值主张

标准化互通
所有 Agent 无论底层 LLM 是什么，只要实现 A2A 接口就能互相发现、发送消息、协作完成任务。
统一管控面
Host 作为控制平面，提供 Agent 注册、健康状态、能力查询、任务追踪，一个页面看清全局。
全链路可观测
从消息发出到最终响应，每个状态变更都记录为 Trace 事件，时间轴可视化，排障从小时降到分钟。
异构兼容
helloworld（通用 OpenAI）、bridge（外部服务适配）、claude_code（本地 CLI）三类 Agent 共存，证明平台对异构场景的包容性。

关键设计点

  • AgentCard 驱动：不依赖人工填写 YAML 的 skills，而是从 AgentCard 动态解析。即使 Agent 升级了能力，Host 重新连接后自动同步。
  • 指数退避重试：连接失败时 1s → 2s → 4s 重试 3 次，避免抖动；最终失败写入 DB 并标记 status=error，UI 可感知。
  • 运行时动态增删：除了启动时从配置加载，还支持 POST /api/agents {url} 动态接入外部 Agent，DELETE /api/agents/{name} 下线。
  • 状态一致性：内存中的 Registry._connections 与 DB 中的 agents 表保持同步。列表查询时交叉校验，内存缺失则标记 disconnected。

MessageBus 核心机制

流式事件解析
A2A SDK 返回的 SSE 包含 message / task / statusUpdate / artifactUpdate 四种 payload。MessageBus 实时区分并转换为统一事件格式推送给前端。
上下文续传
同一 context_id 的多次消息自动关联到同一 Task，Agent 侧据此恢复对话历史，实现真正的多轮会话。
统一工具定义
HOST_TOOLS 在 MessageBus 中单点维护，helloworld 与 bridge Agent 通过 function calling 调用，MCP 客户端通过 SSE 调用，行为完全一致。
降级与兜底
若 SSE 未直接返回 message，依次 fallback 到 artifact text → status message → "(empty)"，确保前端永远有可见反馈，不卡住。

版本兼容保障

旧架构中存在 Version mismatch: actual='0.3', expected='1.0' 的错误，根因是 Relay 层透传时丢掉了 A2A-Version Header。我们的解决方案：

  • Proxy 自动补全：Host Proxy 在转发请求时，若检测到缺失 A2A-Version，自动注入 A2A-Version: 1.0。
  • Agent 开启兼容：所有 Agent 的 JSON-RPC 路由启用 enable_v0_3_compat=True，即使收到 0.3 请求也能优雅处理。
  • 移除 Relay 层：旧架构中 relay 与 proxy 并存导致路径混乱，新架构统一走 proxy，流量单一、排障简单。

🔍 巧妙的 Trace 方案

Trace 是本平台区别于普通 A2A 演示项目的核心竞争力。我们将"黑盒通信"变为"白盒可观测"。

为什么需要 Trace？

  • Agent 间消息发了没收到？是网络问题还是解析问题？
  • LLM 调用了工具，但返回结果是空的，工具执行成功了吗？
  • 一个用户请求触发了 3 个 Agent 的链式调用，时间花在哪一步？

Trace 事件模型

事件类型触发时机记录内容
sendMessageBus 发起请求消息内容、message_id、context_id
task_created本地 Task 创建成功server_task_id、context_id
task_update收到 statusUpdate / task 事件状态变更（WORKING → RESPONDED 等）
artifact收到 artifactUpdateartifact 文本内容
response收到 message 或 fallback 完成最终回复文本、来源标记
error异常捕获错误码、异常信息

Trace 时间轴可视化

[14:32:01] SEND: 问 hermes 内网状态
[14:32:01] TASK_CREATED: task_id=abc-123 context_id=ctx-456
[14:32:02] STATE: WORKING | Processing request...
[14:32:03] STATE: WORKING | [tool_call] send_to_agent({"agent_name":"hermes",...})
[14:32:05] STATE: WORKING | [tool_result] send_to_agent: 内网服务正常...
[14:32:06] RESPONSE (message): hermes 报告内网所有服务运行正常...

设计亮点

双维度查询
支持按 task_id（单次请求）和 context_id（整个会话）查询。前者看一次交互，后者看多轮对话全貌。
零侵入采集
Trace 逻辑内嵌在 MessageBus 和 Proxy 中，Agent 侧无需修改代码即可被完整追踪。
持久化到 SQL
Trace 事件实时写入数据库，页面刷新不丢失。支持历史回溯，便于审计与问题复盘。
前后端联动
UI 收到 SSE 事件的同时，后端自动写入 Trace；用户无需手动刷新，时间轴随消息流实时生长。
Trace 的隐藏价值：不仅是调试工具，更是 Agent 协作的"审计日志"。未来可基于此做性能分析（哪一步最慢）、异常告警（错误率突增）、甚至自动优化（重试慢路径）。

📦 Agent SDK 设计

agents/ 目录下的实现不仅是演示（demo），更是一套可复用的 Agent SDK。开发者可以基于这些模板快速构建自己的 A2A Agent。

三种 Agent 类型

类型适用场景核心能力
helloworld 通用 LLM Agent 基于 OpenAI SDK，支持 function calling、多轮历史、工具调用闭环。是最简单的 A2A Agent 模板。
bridge 外部服务桥接 适配非标准 OpenAI 端点（如 OpenClaw、内网模型），支持自定义 Header、模型名、超时时间。
claude_code 原生 CLI 集成 spawn 本地 claude 进程，解析 stream-json 输出，支持会话续传、预算控制、权限跳过。

SDK 复用模式

  • 统一骨架：所有 Agent 共享同一套 __main__.py 模式 —— 解析配置 → 构建 AgentCard → 创建 JSON-RPC 路由 → 启动 Uvicorn。启动后自动调用 agents/common/registration.py 向 Host 注册。
  • 只改 Executor：开发者只需实现 AgentExecutor.execute() 方法，处理输入、调用 LLM、返回事件，即可拥有一个完整的 A2A Agent。
  • 共享注册 SDK：agents/common/registration.py 提供 register_with_host() 异步函数，封装了注册、冲突退避重试、幂等校验等逻辑，所有 Agent 类型均可复用。
  • 配置即部署：当前通过 config.yaml 声明实例；未来将迁移到数据库存储 Agent 配置，支持运行时动态注册与热更新，无需重启 Host。
演进方向：当前 Agent 配置走 YAML 是为了快速迭代和版本控制。随着平台成熟，配置将迁移到 SQL 数据库，Host 提供管理界面进行 CRUD，实现真正的 Agent 生命周期管理。

🔌 Agent 接入方式

平台支持多种 Agent 接入方式，无论是本地启动的 Agent、远端部署的服务，还是第三方异构系统，都可以灵活接入 Host 控制平面。

三种接入模式

接入方式适用场景核心机制
静态配置 Host 与 Agent 同网段 / 开发调试 Host 启动时读取 config.yaml 中的 agents 列表，主动拉取 /.well-known/agent.json 建立连接。适合固定基础设施。
远程自注册 Agent 动态启动 / 跨网段 / 容器化部署 Agent 启动后通过 agents/common/registration.py 向 Host POST /api/agents/register 主动上报自身信息。Host 自动完成 AgentCard 拉取、健康检查、连接池初始化。
手动动态添加 临时接入外部 Agent / 第三方服务 通过 Web UI 或 API POST /api/agents {url} 手动添加外部 Agent URL。Host 即时拉取 AgentCard 并尝试连接，支持运行时动态增删，无需重启。

接入方式对比

Host 主动 vs Agent 主动
静态配置是 Host 主动发现（pull 模式），远程自注册是 Agent 主动上报（push 模式）。前者适合已知拓扑，后者适合弹性扩缩容场景。
异构兼容
三种方式对 Agent 的技术栈零要求：只要是 A2A 协议兼容的服务（JSON-RPC over HTTP + AgentCard），无论用 Python、Node.js 还是 Go 实现，均可接入。
状态一致性
无论通过哪种方式接入，Registry 都会将 Agent 信息持久化到 SQL DB，并与内存连接池交叉校验。即使 Host 重启，已注册的 Agent 信息不丢失。
混合部署
生产环境中可同时使用三种方式：核心 Agent 走静态配置保证稳定性，弹性 Agent 走远程自注册，临时测试 Agent 走手动添加，互不冲突。
设计原则：Agent 接入方式的选择只影响"如何被发现"，不影响"如何通信"。一旦接入完成，所有 Agent 在 MessageBus 和 Trace 层面完全平等，统一走 A2A 协议交互。

📡 远程注册 SDK

agents/common/registration.py 是所有 Agent 共享的自注册客户端 SDK。它封装了注册、冲突处理、退避重试等复杂逻辑，开发者只需一行调用即可让 Agent 自动加入 Host 集群。

核心 API

async def register_with_host(
    host_url: str,      # Host 发现 API 地址，如 http://127.0.0.1:7860
    name: str,          # Agent 唯一标识名
    agent_type: str,    # Agent 类型：helloworld / bridge / claude_code
    port: int,          # Agent 本地监听端口
    skills: list[dict], # 技能列表（会被序列化到 AgentCard）
    url: str = "",      # 显式指定 URL（NAT/代理场景必填）
    secret: str = "",   # 幂等重注册密钥
    max_retries: int = 5,
    retry_interval: float = 3.0,
) -> bool

Host 注册端点行为

Agent 调用 SDK 后，实际向 Host POST /api/agents/register 发送以下 JSON：

{
  "name": "agent-a",
  "type": "helloworld",
  "port": 10001,
  "skills": [...],
  "url": "",           // 留空时 Host 自动从请求 IP 推导
  "secret": "agent-a-8ac73c6f7c16fa1f"
}

关键设计点

  • URL 自动推导：若 Agent 未显式提供 url，Host 从 HTTP 请求的 X-Forwarded-For 或 request.client.host 推导 IP，拼接端口生成 http://{ip}:{port}。支持反向代理和容器网络。
  • 幂等重注册：若 Agent 重启后重新注册，提供相同的 secret 且与 Host 记录匹配时，Host 视为同一 Agent 的重新上线，直接返回 re_registered: true，无冲突。
  • 冲突退避重试：若名称已被其他 Agent 占用（且无匹配的 secret），SDK 返回 409。此时 SDK 执行指数退避重试（间隔 3s → 6s → 9s...，最多 5 次），绝不自动 DELETE 其他 Agent，避免误删生产服务。
  • 异步无阻塞：Agent 的 __main__.py 在 Starlette lifespan 中通过 asyncio.create_task(_delayed_register(...)) 延迟 1.5s 后发起注册，确保 HTTP 服务已启动完毕。

使用示例（helloworld Agent）

from agents.common.registration import register_with_host

async def _delayed_register(host_url: str, instance) -> None:
    await asyncio.sleep(1.5)
    await register_with_host(
        host_url=host_url,
        name=instance.name,
        agent_type=instance.type,
        port=instance.port,
        skills=instance.skills,
        url=instance.url or "",
        secret=instance.secret,
    )

@asynccontextmanager
async def lifespan(app):
    if host_url:
        asyncio.create_task(_delayed_register(host_url, instance))
    yield

🚀 未来扩展方向

平台已完成从 0 到 1 的验证，以下是从 1 到 N 的确定规划：

Agent 自动注册机制 ✅
Agent 启动时通过 agents/common/registration.py 主动向 Host 发送注册请求（POST /api/agents/register），Host 自动完成 AgentCard 拉取、健康检查、连接池初始化。 支持幂等重注册（secret 校验）、冲突退避重试、URL 自动推导（含 X-Forwarded-For 支持）。三类 Agent（helloworld / bridge / claude_code）均已集成。
Agent 管理后台（管理员权限）
将配置从 YAML 迁移到 SQL 数据库后，提供 Web 管理页面：新增/编辑 Agent 配置（URL、LLM 参数、skills）、批量启停、查看连接日志、设置访问权限。 引入 RBAC：普通用户只能聊天和查看，管理员可操作 Agent 生命周期。
稳定性与高可用设计
  • 连接池健康检查：定时 ping Agent 的 AgentCard 端点，自动标记失活并告警。
  • 熔断与降级：某 Agent 连续失败时熔断，快速返回降级提示，避免拖垮 Host。
  • 水平扩展：Host 无状态化（除 SQL DB），可部署多实例 + 负载均衡；DB 可迁移至 PostgreSQL 主从。
  • 消息队列缓冲：高并发场景下，引入 Redis / RabbitMQ 缓冲 A2A 请求，削峰填谷。
监控与可观测体系
将现有 Trace 数据升级为完整监控体系：
  • Metrics：QPS、P99 延迟、Agent 错误率、LLM Token 消耗，对接 Prometheus + Grafana。
  • Alerting：Agent 离线、错误率突增、Task 长时间未完成时自动告警（飞书/钉钉/邮件）。
  • 链路分析：基于 Trace 数据自动生成调用拓扑图，识别瓶颈 Agent 和热点路径。
工作流编排（Workflow）
当前多 Agent 协作依赖 LLM 自主决策工具调用。未来将支持显式工作流定义：DAG 编排（A → B → C 并行 → D）、条件分支、人工审核节点。 用户可在 UI 中拖拽构建 Agent 流水线，将 Agent 从"聊天玩具"升级为"生产工具"。
安全与多租户
引入 API Key / JWT 认证，支持多租户隔离：不同团队的 Agent 互不可见，消息流隔离存储。 对话内容审计日志、敏感词过滤、LLM 输出合规检查，满足企业级部署要求。

📝 总结

A2A Playground 不仅是一个技术验证项目，更是一套可落地的多智能体协作基础设施。

  • 协议层：深度调研 A2A 1.0，解决了版本兼容、SSE 流式、Header 治理等真实工程问题。
  • 发现层：基于 AgentCard 的自动发现 + 连接重试 + 状态持久化，让 Agent 注册从手工配置走向自动化。
  • 通信层：MessageBus 统一封装 A2A 消息流，Proxy 透明转发，工具定义单点维护，确保异构 Agent 行为一致。
  • 观测层：Trace 全链路追踪是本项目的独特优势，将 Agent 协作的黑盒变为白盒，为调试、审计、未来监控打下基础。
  • SDK 层：helloworld / bridge / claude_code 三类 Agent 既是演示也是模板；agents/common/registration.py 提供通用远程注册 SDK，降低新 Agent 接入门槛。
  • 接入层：静态配置、远程自注册、手动动态添加三种接入方式并存，覆盖从开发调试到生产弹性扩缩容的全场景需求。
下一步：持续完善 Agent 生命周期管理（配置数据库化、管理后台、健康检查与熔断），逐步从"实验平台"演进为"生产级 Agent 协作中枢"。
