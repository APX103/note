A2A 是什么？

A2A（Agent2Agent）协议是一个开放标准，核心目标是让不同开发者、不同框架、不同组织构建的 AI Agent 能够无缝通信与协作。它提供了一种"通用语言"，打破 Agent 生态的孤岛。

💡

一句话理解

A2A 是 Agent 之间的"通信协议"，而非构建 Agent 的工具。类似于 HTTP 对于 Web 的意义，A2A 想做的是 Agent 互联网的基础设施协议。

2

为什么需要 A2A？

A2A 解决的是 AI Agent 协作中的关键工程挑战。以下通过一个典型场景来说明：

场景示例 规划一次国际旅行

用户对 AI 助手说："帮我规划一次国际旅行"。这个复杂请求需要多个专业 Agent 协作完成：

✈️
航班预订 Agent
🏨
酒店预订 Agent
💱
货币转换 Agent
🚌
本地旅游 Agent
⚠️

如果没有统一协议，这些 Agent 因为各自独立开发部署，无法互相发现、无法通信、更无法协作。

01

Agent 被矮化为工具

传统方式把 Agent 包装成"工具"暴露给其他 Agent，但 Agent 本应自主协商，包装会严重限制其完整能力。

02

点对点集成成本极高

每对接两个 Agent 都要写定制代码，随着 Agent 数量增长，集成复杂度呈指数级上升。

03

创新速度被拖累

为每个新集成做定制开发，大量工程资源消耗在重复的基础设施工作上，而非业务创新。

04

安全缺乏统一保障

临时拼凑的通信方式缺乏一致的认证、授权和加密机制，存在严重安全隐患。

05

生态割裂，无法形成网络效应

各厂商各自为战，Agent 生态呈碎片化，无法像 Web 一样形成有机的、可扩展的生态系统。

3

核心优势

🔒

安全协作

基于 HTTPS 加密通信，且保持不透明执行——Agent 在协作过程中看不到其他 Agent 的内部逻辑和记忆。

🌐

互操作性

打破框架与厂商壁垒，让 LangGraph 构建的 Agent 能与 CrewAI 构建的 Agent 无缝协作。

🤖

Agent 自主性

Agent 保持独立身份和完整能力，不作为工具被调用，而是作为对等协作伙伴参与任务。

🧩

降低集成复杂度

标准化通信格式和接口，团队只需关注 Agent 的核心业务价值，无需重复造通信轮子。

⚡

支持长时间运行任务（LRO）

原生支持 SSE（Server-Sent Events）流式传输和异步执行，适合需要数分钟甚至数小时才能完成的任务，期间可实时推送状态更新和中间产物。

4

五大设计原则

A2A 的开发遵循一系列工程哲学，体现了对广泛采纳、企业级能力和未来兼容性的追求。

简单性 Simplicity

复用现有成熟标准——HTTP 传输、JSON-RPC 消息格式、SSE 流式推送。不重新发明核心轮子，大幅降低开发者学习和接入成本。

企业级就绪 Enterprise Ready

对齐标准 Web 安全实践，原生支持认证、授权、审计、隐私保护、链路追踪和可观测性。不是实验室玩具，而是可直接投入生产环境的协议。

异步性 Asynchronous

原生支持长时间运行任务。Agent 或用户在任务执行过程中无需保持持续连接，通过 SSE 流式和推送通知机制获取进度更新。

模态独立 Modality Independent

协议允许 Agent 使用多种内容类型通信——文本、图片、音频、视频、结构化数据等。超越纯文本限制，支持丰富灵活的交互方式。

不透明执行 Opaque Execution

Agent 协作时不暴露内部逻辑、记忆、推理过程或专有工具。交互仅依赖声明的能力（Agent Card）和交换的上下文。既保护知识产权，又增强安全性。

🎯 A2A vs MCP：最关键的区别

维度 MCP A2A
定位 连接 Agent 与工具/数据 连接Agent 与 Agent
交互对象 工具（无状态、预定义功能） Agent（有状态、自主推理/规划/协商）
交互模式 单次调用，返回结果 多轮对话、协商、任务委派
类比 Agent 的手和眼 Agent 的社交语言

核心观点：把 Agent 包装成简单工具是"降维打击"，无法发挥 Agent 的完整能力。Agent 应该作为 Agent 被调用——能自主协商、能追问澄清、能处理长时间任务。

🔧 A2A vs ADK

ADK（Agent Development Kit）

Google 开源的 Agent 开发框架，提供构建 Agent 的工具包和运行时。类似 LangGraph、CrewAI。

A2A

与构建框架 无关的通信协议。ADK 构建的 Agent、LangGraph 构建的 Agent 都可以通过 A2A 互相通信。

6

请求生命周期

一个 A2A 请求从发起到完成的完整流程分为四个阶段：

1

Agent Discovery（Agent 发现）

获取目标 Agent 的"身份证"

GET /.well-known/agent-card

Client 向 A2A Server 请求 Agent Card——这是 Agent 的元数据文档，包含：

  • •Agent 的能力描述（skills）
  • •API 端点 URL
  • •安全策略（securitySchemes）
  • •支持的交互模态
2

Authentication（认证）

基于标准 Web 安全实践的身份验证

Client 解析 Agent Card 中的 securitySchemes，根据声明的安全方案获取访问令牌：

OpenID Connect

向 authorizationUrl / tokenUrl 请求 JWT Token

其他标准方案

支持扩展，对齐企业现有 IAM 体系

3

sendMessage API

同步模式，适合短任务

POST /sendMessage Authorization: Bearer <JWT>

Client 携带 JWT 发送消息，A2A Server 处理消息并创建 Task，然后返回完整的 Task Response。适合能在短时间内完成的简单任务。

4

sendMessageStream API

流式模式，适合长任务（SSE）

POST /sendMessageStream Authorization: Bearer <JWT>

通过 Server-Sent Events (SSE) 流式推送任务进度，实时获取状态更新和中间产物：

Event TaskStatusUpdateEvent → 任务状态变更（Submitted → Working → Completed）
Event TaskArtifactUpdateEvent → 中间产物推送（部分结果、文件等）

7

关键术语表

核心

Agent Card

Agent 的"身份证"，描述 Agent 的能力、API 端点 URL、安全策略等元数据。通常通过 /.well-known/agent-card 暴露。

核心

Task

A2A 中的基本工作单元。一次用户请求对应一个 Task，包含任务 ID、状态、消息历史、产物列表等。

产物

Artifact

Agent 产出的结果/产物，如生成的报告、图片、文件、结构化数据等。通过 TaskArtifactUpdateEvent 流式推送。

传输

SSE

Server-Sent Events，服务器向客户端单向推送流数据的技术。A2A 用 SSE 实现任务状态的实时流式更新。

传输

JSON-RPC

A2A 使用的远程过程调用消息格式标准。轻量级、语言无关，适合 Agent 之间的结构化消息交换。

特性

LRO

Long-Running Operation，长时间运行操作。A2A 原生支持需要数分钟甚至数小时才能完成的任务，不要求客户端保持长连接。

安全

Opaque Execution（不透明执行）

A2A 的核心安全原则——Agent 在协作时不暴露内部逻辑、记忆、推理过程或专有工具。仅通过声明的能力（Agent Card）和交换的上下文进行交互。既保护知识产权，又增强系统安全性。

学习路径建议

1

先理解 A2A 的定位

A2A 是"Agent 的 HTTP"，解决的是通信问题，不是构建 Agent 的问题。明确它和 MCP、开发框架、LLM 的分工边界。

2

对比 MCP 加深理解

MCP = Agent 调工具，A2A = Agent 调 Agent。这是最容易混淆的点，也是面试/讨论中最常被问到的。

3

掌握四个核心概念

Agent Card、Task、Artifact、Stream。这四个概念是后续理解所有 API 设计和源码的基础。

4

关注企业级特性

认证/授权/安全/追踪/不透明执行——这是 A2A 区别于玩具级 Agent 方案的核心竞争力，也是生产落地的关键。

5

动手实践：阅读官方示例代码

在理解协议设计的基础上，阅读 A2A 官方提供的示例实现，理解 Agent Card 的 JSON 结构、JSON-RPC 消息格式、SSE 流的处理逻辑。
