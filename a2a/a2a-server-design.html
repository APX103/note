1项目定位

平台是一个透明的消息代理——不参与 Agent 内部逻辑，只负责"谁在线"和"消息怎么到"。所有 A2A 协议消息经过平台转发，但不解析消息内容的语义。

核心架构
平台即 Proxy——所有 Agent 的 A2A endpoint 都是平台上的路由（POST /agent/{name}）。Agent 不需要有自己的 URL，通过 MCP 收件箱模型拉取消息。

1.1 V1 范围 必做

Agent 注册与发现

REST API + MCP tool 注册，AgentCard 抓取，内存注册表 + 数据库持久化。支持无 URL 的 MCP Agent。

保活机制

MCP heartbeat tool + SSE 长连接 + HTTP poll 三种模式。超时自动下线，支持在线恢复。

消息代理

收件箱队列模型：消息入队 → Agent 通过 MCP 拉取 → 回复后推送给发送方。收一条回一条。

API + MCP 服务

REST API 管理 + MCP 端点暴露平台能力，含消息收发、心跳保活、Agent 查询。

1.2 明确不做

  • 内建 Agent / LLM 引擎
  • Bridge Agent
  • 消息持久化（不记录消息内容或元数据）
  • 会话管理 / 多轮对话上下文
  • 消息追踪（Trace）与可观测性
  • 前端管理界面
  • etcd 服务发现（V1 用内存 + 数据库）

4核心架构：消息流转

4.1 整体流程

发送方 Agent 平台 (go-zero) 接收方 Agent │ │ │ │ POST /agent/{target} │ │ │ ──────────────────────────► │ │ │ SSE 连接等待响应 │ 消息入队 (go-queue) │ │◄────────────────────────── │ ──────► inbox queue │ │ │ │ │ │ MCP: receive_message │ │ │ ◄─────────────────────── │ │ │ │ │ │ MCP: reply_message │ │ │ ◄─────────────────────── │ │ │ │ │ SSE 事件流推送响应 │ │ │◄────────────────────────── │ │

关键规则

  1. 收一条回一条：Agent 在回复当前消息前无法拉取下一条
  2. 首字超时 1min：Agent 开始处理的时间，超时后平台回复发送方"对方无法响应"
  3. 不设总超时：超时由平台保障，响应可以持续任意长时间
  4. A2A 错误信息：超时、Agent 离线等错误通过 JSON-RPC error 返回

4.2 双模式 Agent 支持

模式 A：MCP Agent（主要）无 URL
  • 通过 MCP tools 收发消息和保活
  • 消息通过收件箱队列中转
  • 不需要运行自己的 HTTP 服务
  • 注册时 type: "mcp"，url 可为空
  • 适合 LLM Agent、脚本 Agent 等
模式 B：HTTP Agent（次要）有 URL
  • 注册时提供 URL
  • 平台通过 HTTP POST 主动转发消息
  • Agent 通过 HTTP 响应返回结果（SSE 或 JSON）
  • SSE 长连接存续 = 在线
  • 适合已有 HTTP 服务的 Agent

消息转发对比

维度MCP AgentHTTP Agent
消息接收MCP tool receive_message 拉取平台 HTTP POST 到 Agent URL
消息回复MCP tool reply_message 提交HTTP 响应（SSE 流 / JSON）
保活方式MCP tool heartbeatSSE 长连接 / HTTP poll
是否需要 HTTP 服务否是
消息模型收件箱队列（go-queue）直接 HTTP 转发

5核心组件设计

5.1 Registry（注册表）

type AgentEvent struct {
    Type      string       // "register", "deregister", "heartbeat", "offline"
    AgentName string
    Agent     *model.Agent
}

type Registry interface {
    Register(ctx context.Context, agent *model.Agent) error
    Deregister(ctx context.Context, name string) error
    Get(ctx context.Context, name string) (*model.Agent, error)
    List(ctx context.Context) ([]*model.Agent, error)
    Heartbeat(ctx context.Context, name string) error
    Watch(ctx context.Context) (<-chan AgentEvent, error)
}

MemoryRegistry 实现

  • sync.RWMutex 保护的 map[string]*model.Agent
  • 启动时从数据库恢复已注册 Agent（状态标记为 offline）
  • Heartbeat() 更新 lastHeartbeat 时间戳
  • 后台 goroutine 每 10s 扫描，超过 TTL（30s）的标记为 offline 并广播事件
  • 接口预留后续 etcd 切换

5.2 Inbox（收件箱队列）新增

type InboxMessage struct {
    MessageID   string     // UUID
    FromAgent   string     // 发送方名称
    TargetAgent string     // 接收方名称
    Content     string     // JSON-RPC 请求体
    ReplyChan   chan *Reply // 回复通道（用于同步等待回复）
    CreatedAt   time.Time
}

type Reply struct {
    MessageID string
    Content   string     // 响应内容
    IsSSE     bool       // 是否 SSE 流式响应
    Error     string     // 错误信息
}

type Inbox interface {
    Push(ctx context.Context, agentName string, msg *InboxMessage) error
    Pop(ctx context.Context, agentName string) (*InboxMessage, error)  // 阻塞等待
    Reply(ctx context.Context, messageID string, reply *Reply) error
    Close() error
}

实现细节

  • 基于 go-queue 组件（beanstalkd 或 kafka）
  • 每个 Agent 注册后自动创建一个命名队列
  • Pop 阻塞等待直到有消息或上下文取消
  • ReplyChan 将回复同步传回等待中的发送方
  • 首字超时 1min：Pop 返回后启动计时器，1min 内未收到 reply → 超时

消息流转状态机

QUEUED → DELIVERED → REPLIED TIMEOUT (首字超时) FAILED (Agent 离线)

5.3 Store（存储层）

type Store interface {
    CreateAgent(ctx context.Context, agent *model.Agent) error
    UpdateAgent(ctx context.Context, agent *model.Agent) error
    DeleteAgent(ctx context.Context, name string) error
    GetAgent(ctx context.Context, name string) (*model.Agent, error)
    ListAgents(ctx context.Context) ([]*model.Agent, error)
    Close() error
}

双方言实现

  • SQLiteStore — 使用 modernc.org/sqlite（纯 Go，无 CGO）
  • MySQLStore — 使用 go-sql-driver/mysql
  • 通过配置文件切换，接口统一

数据库表（agents）

idint64, PK自增主键
namestring, UNIQUEAgent 唯一名称
typestringAgent 类型（mcp / http）
urlstring, nullableAgent 服务地址（MCP Agent 可为空）
statusstring在线状态（connected / offline / unreachable）
agent_card_jsontextAgentCard JSON
skills_jsontext技能列表 JSON
error_messagestring, nullable最近错误信息

设计决策
V1 不做 messages 表——不持久化消息内容或元数据。收件箱队列仅在内存 / 队列服务中暂存。

6REST API

方法路径认证说明
GET/health—平台健康检查
POST/api/agentstoken注册 Agent
GET/api/agents—列出所有 Agent
GET/api/agents/{name}—获取 Agent 详情
DELETE/api/agents/{name}token注销 Agent
POST/api/agents/{name}/heartbeat—心跳续约
POST/agent/{name}—发送 A2A 消息（入队收件箱，SSE 流式返回响应）
GET/api/events—SSE 事件流（Agent 上下线通知）
GET/.well-known/agent-card/{name}—获取 AgentCard

注册请求体

{
  "name": "my-agent",
  "type": "mcp",
  "url": "",
  "port": 0,
  "skills": [{"id": "search", "name": "Search", "description": "Web search"}],
  "secret": "optional-secret"
}
  • type: "mcp" — 无 URL，通过 MCP 收消息
  • type: "http" — 有 URL，平台 HTTP 转发

认证方式

  • 需要 token 的接口：注册、注销
  • Header: X-Admin-Token 或 Authorization: Bearer <token>
  • Token 从配置文件读取

7MCP 服务

7.1 连接流程

  1. 建立 SSE 连接：GET /mcp/sse，平台返回 endpoint 事件（含 session_id）
  2. 发送初始化：POST /mcp/messages?session_id=xxx，发送 initialize 请求
  3. 协商能力：协议版本 2024-11-05，支持 tools + resources
  4. 正常交互：调用 tools/list、tools/call、resources/list、resources/read

7.2 MCP 工具

工具参数返回说明
list_agents—Agent 列表列出所有已注册 Agent
receive_message 新增—message_id, from_agent, content从收件箱拉取一条消息，阻塞等待直到有消息
reply_message 新增message_id, response确认回复指定消息，平台将响应推送给等待中的发送方
get_agent_infoagent_nameAgent 详情 JSON获取 Agent 详情
heartbeat 新增—确认保活续约，防止被标记 offline
send_to_agentagent_name, messageAgent 响应向指定 Agent 发消息（内部调用代理逻辑）

7.3 MCP 资源

URI类型说明
a2a://agentsapplication/json所有 Agent 列表
a2a://agents/{name}application/json单个 Agent 详情

7.4 会话模式

有状态模式
  • 通过 SSE 建立长连接
  • 响应通过 SSE channel 推送
  • 适合 MCP 客户端（Claude Desktop、Cursor）
无状态模式
  • 直接 POST /mcp/messages
  • 响应作为 HTTP body 返回
  • 适合脚本测试和 curl 调用

10与原型的关系

当前仓库（a2a-platform-go）是功能完整的调研原型。新项目 a2a-server 从原型中选择性复用和重写。

复用
  • MCP 协议实现（SSE 传输、JSON-RPC 解析）
  • SSE 事件广播器
  • 数据模型定义
  • Store 双方言实现（SQLite / MySQL）
重写
  • 注册表（go-zero 风格，内存 map）
  • 消息代理（收件箱队列模型）
  • API 层（go-zero handler）
  • 中间件（go-zero middleware）
新增
  • Inbox 收件箱（go-queue 封装）
  • MCP receive_message 工具
  • MCP reply_message 工具
  • MCP heartbeat 保活工具
移除
  • 内建 Agent 引擎（engine/）
  • Bridge Agent（bridge/）
  • LLM Provider（llm/）
  • 聊天前端（web/admin/）
  • 会话管理、追踪系统、子代理
