4. 注册与心跳（Registry）

Google 官方 A2A

  • 核心 spec 不定义注册中心协议
  • Agent 通过 well-known URI 被直接发现
  • 注册是外部行为（目录、Catalog、直接配置）
  • 官方样本中有 POST /registry/register + POST /registry/heartbeat，但明确标注为 平台扩展

我们的实现

  • 自建中心化 Registry 服务
  • POST /registry/register + POST /registry/heartbeat
  • 心跳间隔 15s，清理超时 45s
  • Agent 必须先注册到平台才能被其他 Agent 发现
  • 支持 namespace 隔离 + 关键词搜索
评价：我们的 Registry 是平台层的合理扩展，与 Google 样本中的做法一致。中心化注册适合内部团队使用场景，且提供了发现、心跳、离线检测等完整生命周期管理。这不是与官方 spec 的冲突，而是平台能力的补充。

5. 消息投递机制

Google 官方 A2A

  • 默认：直接 HTTP 调用
  • Client 解析 Agent Card 后，直接 HTTP POST 到 Agent 的 url
  • Agent 必须暴露公共 HTTP 端点
  • 支持 Push Notification（Webhook）：Agent 主动 POST 到客户端注册的 URL
  • SSE 用于流式响应

我们的实现

  • 默认：Polling（MessageInbox）
  • Agent 通过 GET /inbox/poll 拉取消息
  • Agent 不需要暴露公共端点（仅 outbound 即可）
  • 可选 WebSocket 长连接、HTTP Callback
  • SDK 将 Polling 封装为同步 API（sendMessage() 内部 poll 等待回复）
影响：我们的 Polling 默认策略对 NAT/防火墙后的 Agent 非常友好，但延迟较高（平均 2.5s）。Google 的 HTTP 直连模式延迟低，但要求 Agent 有公网可达地址。两者各有适用场景。我们的 SDK 可以兼容官方模式：如果 Agent 有公网地址，平台可以直接 POST 到 Agent（即我们的 HTTP Callback 模式）。

8. 多 Agent 编排模式

Google MultiAgent Host（官方样本）

  • 基于 Google ADK + Gemini LLM
  • LLM 通过 Tool Calling 决定调用哪个 Agent
  • 一次只调 一个 Agent（顺序执行）
  • 无预定义工作流图，纯 反应式编排
  • 状态存在 ADK state 中（task_id, context_id）
  • 终端状态处理：input_required 升级给用户

我们的平台

  • 平台层支持 DAG（有向无环图） 编排
  • 通过 parentMessageId + depth 建模调用树
  • 支持 并行（同时调多个 Agent）、条件分支、串行
  • 乐观锁（SessionContext.version）处理并发上下文合并
  • 死循环检测：消息数限制 + 重复 payload 检测
  • 平台作为中心代理，而非 LLM 工具调用
评价：Google 的编排模式更简单（LLM 做决策），适合对话式场景。我们的 DAG 模式更强（支持并行 + 条件），适合复杂工作流。两者不是互斥的——我们的平台可以作为 Google ADK Host 的底层基础设施，ADK Host 通过我们的 /sendMessage API 调用远程 Agent。

9. 安全与认证

维度Google 官方 A2A我们的实现
Agent Card 安全 JWS 签名（可选），securitySchemes 声明认证方式 无签名，JWT Bearer Token 平台统一管理
传输安全 TLS 必须（gRPC over HTTP/2），mTLS 可选 HTTP（文档中未强制要求 TLS）
扩展卡片认证 GET /extendedAgentCard 需认证，可暴露更多技能 无此概念
推送通知认证 PushNotificationConfig.authentication 支持 Bearer、Basic 等 未实现推送通知
输入信任 官方 README 明确警告：外部 Agent 的输入不可信，需 sanitize 未在文档中强调
建议：生产环境应强制 HTTPS/TLS。JWT 方案适合内部平台，但如对外开放需考虑 OAuth2 / mTLS。Agent Card JWS 签名在跨组织场景有价值。

10. SDK 架构差异

Google A2A Python SDK

  • Agent 必须是 HTTP Server（FastAPI / Starlette）
  • 实现 AgentExecutor 接口：execute(context, event_queue)
  • 用 event_queue.enqueue_event() 推送状态/产物
  • DefaultRequestHandler 桥接 Executor ↔ HTTP/JSON-RPC
  • InMemoryTaskStore 存储任务状态
  • Client：A2ACardResolver + create_client() + send_message()

我们的 A2AAgentSDK（TypeScript）

  • Agent 可以没有 HTTP Server（Polling 模式下纯客户端）
  • SDK 封装注册/心跳/轮询，对外暴露 sendMessage() 同步 API
  • onMessage(handler) 注册回调处理 inbound 消息
  • SDK 自动回复：handler 返回结果 → 自动 sendReply()
  • Agent HTTP Server 仅在 Callback 模式下需要（/sendMessage）
  • SDK 内部 poll loop 隐藏异步机制
评价：Google SDK 假设 Agent 是 Server（被动接收请求），我们的 SDK 假设 Agent 是 Client（主动拉取）。两种模式互补：我们的 Polling 模式适合防火墙后的 Agent，Google 的 Server 模式适合云原生部署。我们的 SDK 已支持 Callback 模式（Agent 暴露 HTTP 端点），与 Google 模式兼容。

优先级建议工作量收益

已对齐 ✅ Task 状态已统一为 CANCELED（单 L） — 命名对齐

总体评价：我们的实现与 Google A2A 核心概念高度一致（JSON-RPC 2.0、Agent Card、Task/Message/Artifact 模型、SSE Streaming）。高优先级对齐项已完成（Agent Card 路径、PascalCase 方法名、CANCELED 拼写、INPUT_REQUIRED / REJECTED 状态）。剩余差异集中在：

  1. 平台层扩展：Registry、Polling 投递、DAG 编排、SessionContext 乐观锁——这些是官方未定义但内部平台必需的能力
  2. 缺失状态：AUTH_REQUIRED、UNKNOWN——建议按需补齐
  3. 传输绑定：gRPC / REST 绑定、Push Notification——仅在对接 Google Cloud / Vertex AI 等生态时需要

互操作路径：高优先级对齐完成后，我们的 Agent 可以被 Google A2A Python SDK 的 Client 直接调用；反之，Google 官方 Agent（如 HelloWorld）也可以通过我们的平台注册并被其他 Agent 发现。两者的 Agent Card 结构和方法命名已完全兼容。
