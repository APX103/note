TL;DR 核心要点

Google ADK 是一个多语言、代码优先的 Agent 开发框架，核心能力在于通过 Sub-Agent 层级实现本地 Agent 协作，通过 A2A（Agent-to-Agent）开放协议实现跨网络、跨框架的远程 Agent 互操作。对于构建 A2A Proxy + 群组服务，关键路径是：利用 Python SDK 的 RemoteA2aAgent 消费远程服务、to_a2a() 暴露本地服务，通过 LLM 自主路由或 Workflow Agent 实现智能编排，借助 ADK Dev UI 和 OpenTelemetry 完成全链路调试，最终在 Proxy 层扩展组播、共识聚合和动态成员管理实现群组服务。

1. ADK 核心架构与设计理念

框架定位与多语言支持

Google Agent Development Kit（ADK）是一个开源、代码优先（code-first）的框架，旨在将软件开发原则系统性地应用于 AI Agent 的创建、测试、部署和编排 [1]。

设计哲学：围绕"强大的简洁性"展开，开发者可以从简单的提示词和工具调用开始，逐步扩展到复杂的多 Agent 协作网络。

Agent 基础模型

ADK 的 Agent 体系建立在 BaseAgent 抽象基类之上，通过继承和组合形成三种核心类型。

LLM Agent：基于大语言模型的核心智能体
Workflow Agent：确定性流程控制
生命周期管理：创建、运行、销毁与状态维护

父子层级关系

运行时委托

选型对比

Agent Card 发现

抽象概念 说明 在 Proxy 设计中的作用

3. 任务编排与协作模式

顺序 Agent

SequentialAgent 实现严格的线性流程控制

关键机制：通过 output_key 实现状态传递，前一个步骤的输出自动写入会话状态

并行 Agent

ParallelAgent 同时触发所有子 Agent

应用场景：多 API 数据获取、多视角分析、冗余执行取最优

循环 Agent

LoopAgent 实现条件迭代

终止条件：通过 continue_condition 表达式或 max_iterations 控制

Workflow Agent 类型对比

Workflow 类型 类名 执行模式 数据传递机制 典型应用场景
顺序 Agent SequentialAgent 按 sub_agents 列表线性依次执行 前一个 Agent 通过 output_key 写入 session.state，后一个通过 {key} 模板引用 数据验证→处理→报告的多步骤流水线
并行 Agent ParallelAgent 同时并发执行所有子 Agent 各分支通过独立 output_key 写入状态，后续 Agent 聚合读取 多 API 数据获取后的结果综合
循环 Agent LoopAgent 重复执行直至满足终止条件 每次迭代读取并更新共享状态 代码迭代优化、质量检查直至达标

动态协作编排模式

LLM 自主路由

基于任务描述智能选择子 Agent，通过语义匹配和动态决策实现灵活路由

优势：极强的适应性，新增子 Agent 只需更新 description 即可自动参与路由

多 Agent 并行调用

同时触发多个子 Agent 并聚合结果，支持多视角并行分析

应用场景：多 Agent 投票、多源信息综合、冗余执行取最优

人机协同中断点

通过 input-required 状态实现人工审核与继续机制

价值：平衡效率与风险，在关键决策点引入人类监督

5. 面向 A2A Proxy 与群组服务的工程实践

Proxy 服务设计要点

协议转换层

A2A 标准消息与内部格式的双向映射，保留完整语义信息

服务注册中心

Agent Card 的聚合与缓存，支持拉模式、推模式、混合模式

路由决策引擎

基于技能匹配的智能转发，支持规则路由、语义路由、混合路由

负载均衡与故障转移

多实例 Agent 的调度策略，包括健康检查和故障转移

群组服务核心机制

组播模式

一对多任务分发与结果收集，支持异步并发和超时控制

共识聚合

多 Agent 结果的去重与投票，支持多种聚合策略

动态成员管理

Agent 的加入、退出、心跳检测，支持优雅退出和故障剔除

权限与隔离

群组内访问控制与命名空间，支持多租户隔离

关键实现模式对比

实现模式 架构 优势 适用场景
根 Agent 作为 Proxy 统一入口委托至后端 Agent 集群 实现简单，直接利用 ADK 内置路由能力 Agent 数量较少（10-20个）、任务类型简单
独立 Proxy 进程 Sidecar 模式解耦业务与通信 通信逻辑与业务逻辑彻底分离 生产环境，需要独立升级和水平扩展
混合云部署 本地子 Agent + 云端远程 Agent 数据安全与弹性扩展的平衡 敏感数据处理 + 大模型能力需求

可靠性保障机制

重试与退避

退避系数： 2

选型建议

选择 Python SDK 当：

  • 需要完整的 A2A 协议双向支持
  • 快速原型开发和频繁迭代
  • 需要丰富的内置工具生态
  • 构建复杂的 Agent 编排逻辑

选择 Go SDK 当：

  • 高性能网关或协议转换层
  • 云原生部署和 Serverless 架构
  • 主要作为 A2A 服务提供者
  • 需要极小的容器镜像和快速启动

混合使用建议

对于复杂系统，可以考虑Python + Go 混合架构：Python 实现业务逻辑复杂的 Agent，Go 实现高性能的 Proxy 网关和协议转换层，通过 A2A 协议进行无缝协作。

回调钩子系统

回调点 触发时机 典型用途
before_model_callback LLM 调用前 输入安全过滤、提示注入检测、上下文压缩
after_model_callback LLM 调用后 输出脱敏、格式校验、内容审核
before_tool_callback 工具执行前 参数校验、权限检查、审计日志、速率限制
after_tool_callback 工具执行后 结果缓存、错误处理、性能监控、数据脱敏
after_agent_callback Agent 完成后 资源清理、会话持久化、指标上报

最佳实践

回调钩子是实现横切关注点（cross-cutting concerns）的理想位置，建议在 Proxy 服务中统一实现认证鉴权、速率限制、请求改写、响应缓存等功能，保持业务 Agent 的纯净性。
