01

ADK 核心架构

ADK（Agent Development Kit）不是简单的 LLM 调用框架，而是一个围绕 Runner + SessionService + Agent 层级构建的 Agent 运行时。它的设计目标是把"单一大 Agent"拆成可组合、可观测、可复现的多 Agent 工作流。

graph TD A[User Message] --> B[Runner] B --> C[SessionService] B --> D[Root Agent] D --> E[LlmAgent] D --> F[Workflow Agent] D --> G[RoutedAgent] E --> H[sub_agents / tools] F --> I[SequentialAgent] F --> J[ParallelAgent] F --> K[Graph Workflow] C --> L[(Session + Events + State)]
🏃

Runner

执行入口，负责创建 Session、驱动 Agent 循环、收集 Event 流并持久化。

💾

SessionService

统一会话抽象，持久化 session、state、events。可插拔后端（Memory / SQLite / Cloud）。

🧩

Agent 层级

LlmAgent、Workflow Agent、RoutedAgent、Template Agent 可嵌套组合。

关键洞察

ADK 的核心设计哲学是：把控制流从 prompt 里抽出来，用代码显式表达。Graph Workflow 用 edges 定义路由，Workflow Agent 用组合定义顺序，RoutedAgent 用函数定义选择。这让复杂流程比纯 prompt 更可靠、更可控。

02

Collaboration 机制

ADK 的协作模式解决一个问题：当一个父 Agent 把任务委派给子 Agent 时，子 Agent 应该有多大自主权？什么时候自动返回？能不能和用户交互？

Mode 用户交互 返回父 Agent 并行执行 适用场景
chat 完整对话 手动 transfer_to_agent 不支持 需要用户深度参与的子 Agent
task 仅澄清问题 自动 complete_task 不支持 独立完成子任务后返回
single_turn 不允许 自动返回结果 支持 纯工具型子 Agent，可批量并行

实现机制：Session Branch 隔离

当父 Agent 派发多个子 Agent 时，每个 task / single_turn 子 Agent 运行在独立的 session branch 中。这意味着：

  • 子 Agent 只能看到自己的 events，看不到兄弟 Agent 的中间思考
  • 父 Agent 在所有 branch 完成后，拿到汇总结果继续编排
  • 避免上下文污染，减少 LLM 幻觉
graph LR A[Parent Agent] --> B[Branch A: single_turn] A --> C[Branch B: single_turn] A --> D[Branch C: single_turn] B --> E[Result A] C --> F[Result B] D --> G[Result C] E --> H[Parent gathers results] F --> H G --> H

注意限制

task mode 在 Graph Workflow 中被禁用（ADK Python v2.0.0）。task agent 必须是叶子节点，不能再有 sub_agents。这些限制说明 ADK 的协作模式在不同工作流类型中的兼容性还在演进。

03

Graph Workflow

Graph Workflow 是 ADK 2.0 引入的核心能力。它允许把 LLM Agent、函数、工具、人类输入组合成一个显式的执行图，用 edges 定义路由和分支。

核心抽象

N Node

执行单元，可以是 Agent、FunctionNode、Tool、Human Input Task，甚至是嵌套的 Workflow。

E Edge

连接 Node 的有向边。支持条件分支：router 返回 Event(route="...")，映射到不同目标 Node。

典型路由模式

def router(node_input: str):
    routes = [r.strip() for r in node_input.split(",")]
    return Event(route=routes)

root_agent = Workflow(
    name="routing_workflow",
    edges=[
        ("START", process_message, router),
        (router, {
            "BUG": response_1_bug,
            "CUSTOMER_SUPPORT": response_2_support,
            "LOGISTICS": response_3_logistics,
        })
    ],
)

实现机制：Event 驱动状态机

Graph Workflow 的执行本质上是一个 Event-driven state machine：

  • 每个 Node 执行后产生一个 Event
  • Event 可以携带 route、output、message 等字段
  • Runtime 根据 edges 定义和 Event 内容决定下一个 Node
  • Node 之间通过 Shared Session State 传递数据
flowchart LR Start([START]) --> Classify[LLM Classify] Classify --> Router{Router} Router -->|BUG| Bug[Handle Bug] Router -->|SUPPORT| Support[Handle Support] Router -->|LOGISTICS| Logistics[Handle Logistics] Bug --> End([END]) Support --> End Logistics --> End

已知限制

Graph Workflow 不支持 Live Streaming，部分第三方 Integration 也不兼容。这意味着如果业务需要实时流式输出，Graph Workflow 可能不是最佳选择。

04

Multi-Agent Workflow Patterns

ADK 把常见多 Agent 模式抽象成了可复用的模板。这些模式本质上是 Agent 组合 + Shared State + 控制流 的不同排列。

🎯

Coordinator & Dispatcher

一个中心协调 Agent 根据输入路由到专业子 Agent。

实现：父 Agent 的 sub_agents + LLM-driven transfer 或 AgentTool 显式调用。

➡️

Sequential Pipeline

多个 Agent 按固定顺序执行，前一个的输出写入 state，后一个读取。

实现：SequentialAgent + output_key。

⚡

Parallel Fan-out & Gather

多个 Agent 并行执行，结果汇总到 Shared State，再由一个 Agent 聚合。

实现：ParallelAgent 嵌套在 SequentialAgent 中。

🌳

Hierarchical Decomposition

树状层级：高层 Agent 拆解任务，中层 Agent 组合工具，底层 Agent 执行具体动作。

实现：多级 sub_agents + AgentTool。

📝

Generate & Review

一个 Agent 生成内容，另一个 Agent 评审。可扩展为循环直到通过。

实现：SequentialAgent + 生成器 output_key + 评审器读取。

🔄

Iterative Refinement

LoopAgent 反复执行 refine → check → stop，直到质量达标或达到最大迭代次数。

实现：LoopAgent + 自定义 stop checker 设置 escalate=True。

统一通信机制：Shared Session State

所有这些模式都依赖同一个通信机制：Shared Session State。Agent 通过 output_key 把结果写入 state，其他 Agent 通过 context.state 或 prompt 模板中的 {key} 读取。

graph TD A[Input] --> B[Agent 1] B -->|"output_key='api1_data'"| S[Shared State] A --> C[Agent 2] C -->|"output_key='api2_data'"| S S --> D["Agent 3: reads {api1_data} & {api2_data}"] D --> E[Output]
05

Human-in-the-loop

ADK 没有内置的"Human Agent"类型，但提供了两种标准模式让人类介入：轻量版用 Tool 暂停等待人工，强化版用 Policy Engine 做统一治理。

模式 A：Tool 暂停等待

Agent 调用一个特殊 Tool，该 Tool 把审批请求发到外部系统（UI/飞书/邮件），然后阻塞等待人类回复。

# 概念代码
approval_tool = FunctionTool(func=external_approval_tool)

request_approval = LlmAgent(
    name="RequestHumanApproval",
    instruction="调用 approval_tool 等待人类审批",
    tools=[approval_tool],
    output_key="human_decision"
)

模式 B：Policy Engine

通过 SecurityPlugin + BasePolicyEngine 拦截所有 Tool 调用，根据策略决定是否要求用户确认。这是 TypeScript 推荐模式。

// 概念代码
const runner = new InMemoryRunner({
  agent: rootAgent,
  plugins: [new SecurityPlugin({
    policyEngine: new CustomPolicyEngine()
  })]
});

关键洞察

Human-in-the-loop 在 ADK 里不是"一个特殊 Agent"，而是运行时拦截 + 外部回调。这种设计很灵活，但也意味着你需要自己搭建人类交互界面（Web UI、IM 机器人、工单系统）。

06

Agent Router

Agent Router 是 ADK 的实验性功能，解决运行时选择哪个 Agent 来执行的问题。它不同于 LLM-driven delegation（由 LLM 决定 transfer），而是由一个显式的 router 函数做决策。

Router 函数签名

type AgentRouter = (
  agents: Readonly>,
  context: InvocationContext,
  errorContext?: { failedKeys: ReadonlySet; lastError: unknown },
) => Promise | string | undefined;

Failover 行为

Router 的一个重要能力是失败自动切换：

  • 首次调用无 errorContext，做初始选择
  • 如果选中的 Agent 在产生任何 Event 之前失败，router 会再次调用，传入失败信息
  • 已失败的 key 不能再选
  • 如果 router 返回 undefined，则抛出最后错误
🔄

Fallback

主 Agent 失败自动切备用 Agent

🧪

A/B Testing

按配置或流量比例切换不同 Agent

📊

Complexity Routing

用轻量分类器决定简单/复杂请求走不同模型

注意

Agent Router 目前还是实验性功能，API 可能变化。它只能选择一个 Agent，不能组合多个 Agent。如果需要多 Agent 协作，要用 Workflow 或 Coordinator 模式。

07

ADK 的实现机制

ADK 能把这些高级抽象跑起来，底层靠几个关键机制。理解这些机制，才能判断自建需要付出什么代价。

1. Event-First 运行时

所有交互都是 Event：user message、agent response、tool call、tool result、state delta、transfer、escalate。Runner 按顺序处理 Event，State 通过 Event 的 actions 变更。

2. Session Branch 隔离

子 Agent 运行在独立 branch，上下文互不污染。branch 路径类似 parent.child_1，构建 LLM 上下文时按 branch 过滤。

3. Agent Transfer vs AgentTool

Transfer 是控制流切换（子 Agent 接管对话，完成后返回）。AgentTool 是把 Agent 包装成 Tool 调用（更像函数调用）。ADK 同时支持两者，让编排更灵活。

4. State Deep Merge

Agent 通过 output_key 写入 state 时，ADK 会做 deep merge，避免后执行的 Agent 覆盖前面 Agent 写入的不同字段。

sequenceDiagram participant U as User participant R as Runner participant S as SessionService participant A as Root Agent participant B as Sub Agent U->>R: message R->>S: createSession R->>A: run A->>R: Event(tool_call) R->>B: transfer B->>R: Event(response) R->>S: append_event R->>A: return control A->>R: Event(final_response) R->>U: result stream
08

接入 ADK vs 自己实现

这是最关键的问题。直接接入 ADK 能获得成熟抽象，但会引入框架依赖和语言绑定。自己实现更灵活，但需要重建 ADK 的核心机制。

接入 ADK

  • ✓ 立即获得 Graph Workflow、Collaboration、Router 等完整抽象
  • ✓ SessionService 已设计好，可插拔后端
  • ✓ Google 维护，文档和社区逐渐完善
  • ✓ 跨语言支持（Python/TS/Go/Java/Kotlin）
  • ✓ Event-First、State Deep Merge 等细节不用自己踩坑

接入 ADK 的成本

  • ✗ 强框架绑定：Agent 定义、工作流、状态管理都要按 ADK 方式写
  • ✗ Agent Swarm 现有架构（Docker Worker、A2A、自定义 Dispatcher）需要大量改造
  • ✗ Graph Workflow 不支持 Live Streaming，与现有 WebSocket 进度流冲突
  • ✗ 实验性功能（Router）API 不稳定
  • ✗ 需要理解 ADK 的 Runner/Event/State 模型，学习曲线不短

自己实现

  • ✓ 完全匹配现有 Agent Swarm 架构
  • ✓ 保留 Docker Worker、A2A、自定义 Dispatcher 投资
  • ✓ 可以只借 ADK 概念，不贪全，控制复杂度
  • ✓ 不受 ADK 语言/版本限制
  • ✓ WebSocket 流式、外部 Agent 网关等自定义能力可保留

自己实现的成本

  • ✗ 需要重建 Event-First SessionService、State Deep Merge、Branch 隔离
  • ✗ Graph Workflow 的 edges/routes/nested workflows 需要从头设计
  • ✗ Collaboration 模式的自动返回、context isolation 需要仔细实现
  • ✗ Human-in-the-loop 的暂停/恢复机制需要自己开发
  • ✗ 测试和边界 case 要自己覆盖

核心判断

如果 Agent Swarm 已经有大量自定义运行时（Docker Worker、A2A、Dispatcher、WebSocket 流），直接接入 ADK 的改造成本很高，而且很多现有能力会被 ADK 限制。更务实的做法是借鉴 ADK 的概念和接口设计，自己实现一个轻量版本，而不是把整个系统迁到 ADK 上。

09

对 Agent Swarm 的建议

结合你的定位（内网、几十 Agent、生产级 demo、手工部署、厌恶复杂度），我的建议如下：

建议：不全量接入 ADK，而是「借鉴 + 自建」

Agent Swarm 已经有自己的运行时和 Agent 接入方式。把 ADK 当作设计参考，把它的好概念摘出来，按自己的架构实现，是性价比最高的路径。

值得借鉴并自建的部分

  • SessionService 协议：已经开工，继续做下去
  • Event-First 日志：用于审计和复现现场
  • State Deep Merge：多 Agent 写入 state 时不互相覆盖
  • Graph Workflow 概念：但用更轻量的方式实现，支持现有 WebSocket 流
  • Agent Router：显式 router 函数 + failover，比纯 LLM 路由更可控
  • Multi-Agent Patterns：Sequential / Parallel / Coordinator 可以按需要逐步加

暂时不要做的部分

  • 全量迁移到 ADK Runner（破坏现有架构）
  • 照搬 ADK 的 Agent 定义方式（不兼容 Docker Worker）
  • 依赖 ADK 的 Graph Workflow 原生实现（不支持 streaming）
  • 使用实验性 Router API（不稳定）
🎯

近期目标

完成 SessionService 集成 + Event 日志 + State Deep Merge

🚀

中期目标

实现轻量 Graph Workflow 编排 + Agent Router

🏆

长期目标

Human-in-the-loop + 声明式 Agent 注册 + per-agent 可观测

结论

ADK 的设计理念和抽象层次都很优秀，尤其是 SessionService、Event-First、Graph Workflow、Agent Router 这些概念，对构建复杂 Agent 系统非常有指导意义。

但 ADK 是一个完整的框架，不是一组可拆卸的组件。对于已经有自己运行时的 Agent Swarm 来说，直接接入 ADK 相当于换引擎，成本高、限制多、收益不明确。

最合理的策略是：深度理解 ADK 的机制，借鉴其接口和思想，在 Agent Swarm 现有架构上逐步实现等价能力。这样既能获得 ADK 带来的设计优势，又能保留自己的灵活性和已有投资。
