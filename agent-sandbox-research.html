Agent 沙盒方案深度调研

多语言代码执行与环境自动构建能力评估  ·  调研日期：2026-05-28

执行摘要

本报告针对基于 Eino 构建的 AI Agent 在沙盒中自动构建环境并执行代码的需求， 对当前 agent-sandbox 工程的能力边界、AIO Sandbox 底层能力、以及市场上主流沙盒替代方案进行了全量调研与深度对比。

⚠️
当前方案：基本能跑，但不够"自动"
AIO Sandbox 具备运行 Python/TS/Go 的能力，但"自动环境构建"依赖 Agent 自己发 shell 命令，而非沙盒原生能力。
✅
推荐路径：短期扩展 AIO，长期观望 Daytona
无需换沙盒。扩展 backend 暴露 Jupyter 接口 + 定制预装镜像即可满足 80% 需求。

一句话： 你们已经在用业内最合适的沙盒之一（AIO Sandbox）。问题不是"换什么沙盒"，而是"当前 backend 只桥接了 File + Shell，没有把 AIO 的 Jupyter 代码执行能力暴露给 Agent"。 扩展 backend 增加 ExecuteCode 接口，同时在 Dockerfile 预装 pyenv/nvm/Go，Agent 就能通过工具调用来"自动"构建环境并跑代码。 如果未来需要运行完全不可信的外部代码（比如开放平台），再评估 Daytona 或 E2B。

需求拆解

用户原始需求

"agent 在输出一段代码之后会在沙盒里构建环境跑代码，目前需要跑 python ts go，然后需要能自动搞环境构建（python版本啊，依赖啊什么的）"

需求拆解为三层

层级 具体需求 说明
L1：代码执行 在隔离环境中执行代码片段 Agent 输出 print(1+1) 或 console.log("hi")，沙盒能运行并返回结果
L2：多语言支持 Python、TypeScript、Go 三种语言的运行时都必须可用，且能编译/解释执行
L3：自动环境构建 自动检测依赖并安装 Agent 输出 import pandas 后，沙盒自动检测到缺失并 pip install pandas
关键认知：L3 "自动环境构建" 在绝大多数沙盒方案中都不是沙盒本身的能力， 而是Agent 通过工具调用主动触发的。沙盒提供的是运行环境 + 包管理工具， 真正的"自动"需要 Agent 具备以下能力：
  1. 从代码中解析出依赖（如读取 requirements.txt、package.json、go.mod）
  2. 调用 pip install / npm install / go mod tidy 安装
  3. 再执行代码
少数方案（如 E2B 的 e2b-code-interpreter）内置了部分依赖自动安装，但灵活度有限。

当前方案：AIO Sandbox 能力图谱

当前工程依赖的 agent-infra/sandbox-sdk-go 对应底层是 AIO Sandbox（All-in-One Sandbox），由字节跳动 Agent Infra 团队开源维护。 这是一个单 Docker 容器内的完整 Linux 环境，内部包含：

已具备的能力

  • ✅ Shell 命令执行（/v1/shell/exec）
  • ✅ 文件读写、搜索、替换（/v1/file/*）
  • ✅ Jupyter 代码执行（/v1/jupyter/execute）
  • ✅ 浏览器自动化（VNC + CDP）
  • ✅ VSCode Server（/code-server/）
  • ✅ MCP Server（Browser / File / Shell / Markitdown）

预装环境（从镜像推断）

  • ✅ Python 3（基础镜像自带）
  • ✅ pip（Python 包管理）
  • ✅ Node.js / npm（基础镜像自带）
  • ⚠️ Go 工具链（不确定，可能需自行安装）
  • ⚠️ pyenv / nvm（未确认预装）
  • ✅ Git、bash、curl 等基础工具

SDK API 一览

API 端点 当前 backend 是否桥接 用途
/v1/shell/exec 已桥接 → ExecuteStreaming 执行任意 shell 命令（含 pip/npm/go mod）
/v1/file/read 已桥接 → Read 读取文件
/v1/file/write 已桥接 → Write 写入文件
/v1/file/list 已桥接 → LsInfo 列出目录
/v1/file/grep 已桥接 → GrepRaw 文件内容搜索
/v1/file/glob 已桥接 → GlobInfo 通配符匹配
/v1/jupyter/execute 未桥接 执行 Python/Node 代码（关键缺口）
/v1/browser/* 未桥接 浏览器自动化

当前 backend 的核心缺口

最大缺口：没有暴露代码执行接口。
当前 backend 只实现了 filesystem.Backend + StreamingShell， Agent 能写文件、能跑 shell 命令，但没有直接的代码执行工具。 这意味着 Agent 要跑代码，必须走两步：
  1. Write 工具把代码写入 test.py
  2. ExecuteStreaming 工具运行 python test.py
这虽然能工作，但体验不如一个专门的 execute_code 工具直接。

缺口清单

缺口 影响 修复成本
未桥接 /v1/jupyter/execute Agent 无法直接执行代码片段，必须先生成文件再 shell 执行 低 新增一个接口封装
Dockerfile 未预装 pyenv/nvm/Go Agent 需要先用 apt 安装工具链，增加首次启动时间 低 Dockerfile 加几行 RUN
无代码依赖自动解析 Agent 需要自行读取 requirements.txt/package.json/go.mod 并安装 中 需要 Agent Prompt 工程 + 工具增强
目录隔离而非容器隔离 可信场景够用，但不可信代码有容器逃逸风险 高 需要换沙盒方案

自动环境构建：在当前方案下的可行性

"自动环境构建"在当前 AIO Sandbox 下是可行的，但需要 Agent 侧配合。 沙盒本身只提供工具（pip/npm/go），不会主动检测代码需要什么依赖。 实现路径如下：

路径 A：Agent 自主管理（推荐短期）

1. Agent 输出代码时，同时输出依赖清单
   → 例如："这段代码需要 pandas 和 numpy"

2. Agent 调用 execute 工具，先发安装命令：
   → pip install pandas numpy
   → npm install lodash
   → go get github.com/some/pkg

3. Agent 再调用 execute 工具运行代码
   → python /home/gem/<user>/script.py

4. 如果报错 ImportError / ModuleNotFound，Agent 自行 retry 安装缺失依赖
优势：不需要改沙盒，完全利用现有 Shell 能力。
实践：OpenAI 的 Code Interpreter、Claude 的 Artifacts 都是这个模式——Agent 自己写代码、自己装依赖、自己运行。

路径 B：后端增强（推荐中期）

1. 在 backend 中新增一个 "智能执行" 工具，接口形如：
   type SmartExecuteRequest struct {
       Code string
       Language string  // python | typescript | go
       Dependencies []string  // 可选，Agent 显式指定
   }

2. backend 内部逻辑：
   a. 根据 Language 选择执行器（jupyter / node / go run）
   b. 如果 Dependencies 非空，先执行安装命令
   c. 执行代码，返回 stdout / stderr / exit_code

3. 更进一层：backend 可以自动解析代码中的 import/require，
   自动生成安装命令（但这属于"便利功能"，不是必须）

三种语言在当前方案下的执行方式

语言 当前可用方式 执行命令示例
Python Shell 执行 / Jupyter API python script.py 或 /v1/jupyter/execute
TypeScript / Node.js Shell 执行 node script.js 或 npx ts-node script.ts
Go Shell 执行（需确认工具链） go run main.go

方案一：Daytona（最值得关注）

Daytona 于 2025 年初全面 pivot 为 Agent-native 沙盒基础设施， 核心卖点是sub-90ms 的冷启动 + 无限持久化 + 多语言 SDK。

核心能力

  • 隔离：Docker 容器（可选 Kata），OCI 镜像兼容，支持 Docker-in-Docker
  • 启动速度：warm-pool 预热，sub-90ms 创建，远超 AIO Sandbox 的 2-10s
  • 持久化：无限会话，支持快照（snapshot）和恢复，Agent 可以长时间保持状态
  • 多语言 SDK：Python、TypeScript、Go、Ruby、Java 官方 SDK
  • 集成：LangChain 官方集成、MCP Server 支持、Git 原生操作、LSP 支持
  • 自托管：AGPL 开源，可以完整私有化部署，数据不出境

与当前需求的匹配度

需求 匹配度 说明
Python 代码执行 ✅ 完美 原生支持，有 sandbox.process.code_run() API
TypeScript 代码执行 ✅ 完美 原生支持 Node.js 运行时
Go 代码执行 ✅ 完美 原生支持 Go SDK
自动环境构建 ⚠️ 部分 仍需要 Agent 发命令安装依赖，但快照可以预装环境后复用
自托管 / 数据安全 ✅ 完美 AGPL 开源，完整私有化部署
Eino 集成 ⚠️ 需开发 没有现成的 Eino backend 实现，需要按 filesystem 接口封装

优势 vs 劣势

优势
  • 开源可自托管，无 vendor lock-in
  • 启动极快（<90ms），适合高频 Agent 调用
  • 快照机制：预装好环境后保存快照，后续秒级恢复
  • 支持 Computer Use（虚拟桌面），未来可做 GUI Agent
  • 多语言 SDK 成熟
劣势
  • Docker 隔离弱于 Firecracker microVM（共享内核）
  • 需要重新开发 Eino backend 桥接层
  • 生态较新，社区成熟度不如 E2B
  • 默认 15 分钟无操作 auto-pause，可能需调参

方案二：E2B（朋友觉得太复杂）

E2B 是 AI Agent 沙盒领域的"老大哥"，月活从 2024 年初的 4 万次增长到 2025 年 3 月的 1500 万次， 是市场上最成熟的方案。但朋友反馈"太复杂"，我们需要理解复杂在哪。

E2B 的"复杂"体现在哪

维度 E2B 的做法 为什么觉得复杂
隔离模型 Firecracker microVM，每个沙盒独立内核 需要理解 microVM、快照、模板等概念
自定义环境 通过 Dockerfile 构建自定义模板（e2b template） 构建模板需要 e2b CLI，流程多一步
SDK 设计 围绕 Agent 工作流设计，概念抽象度高 API 粒度细（commands.run / run_code / files 等），学习曲线陡
自托管 仅 experimental，主力是 SaaS 国内网络 / 数据合规场景下不可行
多语言 Python/JS/TS SDK，底层是 Ubuntu Go 需要自己在模板里装
核心问题：对于"有运维能力、需要自托管、团队在国内"的场景， E2B 的 SaaS 模式 + 实验性自托管 + 国内网络延迟，确实不是最优解。 但它的 Firecracker 隔离是目前最安全的，适合运行完全不可信的代码。

什么时候该选 E2B

  • ✅ 运行完全不可信的 AI 生成代码（需要 kernel 级隔离）
  • ✅ 需要 150ms 级冷启动的高频交互场景
  • ✅ 团队能接受 SaaS 模式，数据可以出境
  • ❌ 需要自托管 → 不建议（目前不成熟）
  • ❌ 主要跑 Go → 需要自定义模板，额外工作

方案三：Modal（Python-first，不适合）

Modal 是 serverless ML 平台，Sandboxes 只是其能力之一。 它使用 gVisor 隔离（syscall 拦截），不是 microVM，安全性介于 Docker 和 Firecracker 之间。

不建议 Modal 的原因：
  1. Python-first：虽然支持 JS/Go SDK，但生态和社区资源都是围绕 Python ML 的
  2. 不能自托管：必须跑在 Modal 云上，数据不出境的需求无法满足
  3. 冷启动慢：2-5s，比 Daytona 和 E2B 慢一个数量级
  4. 价格高：$250/mo 起步 + usage
  5. 没有 Eino 集成：需要从头开发
Modal 唯一的优势是 GPU 支持（A100/H100），如果你们需要 Agent 跑 ML 训练/推理，再考虑。

方案四：DifySandbox（轻量开源，但不独立）

DifySandbox 是 Dify 平台内置的代码执行组件，2024 年开源。 它的设计哲学是单容器多任务，通过 Seccomp + 系统调用白名单实现隔离， 而不是每个任务起一个容器。

特点

  • ✅ 开源，可独立部署
  • ✅ 执行速度快（无容器启动开销）
  • ⚠️ 只支持 Python 和 Node.js，Go 不支持
  • ⚠️ 依赖管理有挑战（文档自述）
  • ❌ 不是完整沙盒，没有浏览器、文件系统、Shell 等能力
  • ❌ 生态绑定 Dify，独立使用需额外开发
结论：DifySandbox 是一个代码执行引擎，不是完整的 Agent 沙盒。 如果你们只需要跑代码片段，它可以工作；但如果需要文件操作、浏览器、持久化等，它不够用。

其他方案速评

方案 一句话评价 适合场景
Blaxel perpetual standby + 25ms resume，但 pricing 不透明 高频间歇性调用，预算充足
Sprites (Fly.io) Firecracker + 持久化，但绑定 Fly.io 已在 Fly.io 上的团队
CodeSandbox Together AI 收购后转向 AI infra，但主要是 Web 开发 前端代码预览和编辑

推荐路线图

基于调研，给出分阶段实施建议。核心原则是：不换沙盒，先补能力；未来有需要再评估迁移。

Phase 0：现状（当前）
已有 AIO Sandbox 部署，backend 桥接了 File + Shell。
Agent 能写文件、能跑 shell，但无直接的代码执行工具。
Phase 1：短期（1-2 周）—— 补全代码执行能力
  1. 扩展 backend，新增 ExecuteCode 工具，内部封装 Jupyter API（Python）和 Shell 执行（Node/Go）
  2. 修改 Dockerfile，预装 pyenv + nvm + Go 工具链，减少 Agent 首次安装时间
  3. 优化 Agent Prompt，让 Agent 学会"先写代码 → 再装依赖 → 再执行"的 workflow
预期效果：Agent 可以直接调用工具执行三种语言的代码，基本满足需求。
Phase 2：中期（1-2 月）—— 增强体验
  1. 在 backend 中增加"智能执行"封装：Agent 传入代码，backend 自动检测语言、尝试执行、捕获错误、建议安装缺失依赖
  2. 引入 uv（Python 包管理）和 pnpm（Node 包管理），加速依赖安装
  3. 增加执行超时、资源限制（CPU/内存）、输出截断等安全管控
预期效果：Agent 的代码执行体验接近 OpenAI Code Interpreter。
Phase 3：长期（3-6 月）—— 评估是否需要更强隔离
  1. 如果业务需要运行完全不可信的代码（开放平台、C 端用户），评估 Daytona 或 E2B
  2. Daytona 优先：开源可自托管、多语言、快照机制、启动快
  3. E2B 备选：如果安全要求极高（kernel 级隔离），且能接受 SaaS 模式
  4. 迁移成本：需要重写 backend 桥接层，按 Eino filesystem 接口封装 Daytona/E2B SDK
预期效果：如果有安全升级需求，平滑迁移到更强隔离方案。

最终结论

结论一：当前方案能搞定，不需要换沙盒。
AIO Sandbox 已经内置了 Jupyter 代码执行和 Shell 命令执行能力， 只是当前 backend 没有桥接出来。扩展 backend 增加 ExecuteCode 接口即可。
结论二："自动环境构建"不是沙盒的能力，是 Agent 的能力。
无论是 AIO Sandbox、Daytona 还是 E2B，都没有"自动检测代码依赖并安装"的魔法。 真正的实现路径是：Agent 自己解析依赖 → 调用工具安装 → 再执行。 沙盒只需要提供工具链（pip/npm/go）和运行环境。
结论三：如果未来需要运行完全不可信的代码，Daytona 是 E2B 的最佳替代。
Daytona 开源（AGPL）、可自托管、启动快（<90ms）、多语言、有快照机制。 它的容器隔离虽然不如 E2B 的 Firecracker，但对于大多数 Agent 场景足够安全， 且运维复杂度远低于 E2B。
给朋友的建议：
E2B 确实复杂——它的 microVM + 模板构建 + SaaS 模式对于需要自托管的团队来说门槛很高。 你们的场景（内部 Agent、可控用户、需要自托管），AIO Sandbox 已经够用。 如果以后要做开放平台让外部用户跑代码，再考虑 Daytona（自托管）或 E2B（SaaS）。
