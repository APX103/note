三、核心设计

3.1 架构定位：适配器模式（Adapter）

本工程的本质是一个适配器层，将远端 Sandbox 的 REST API 能力翻译为 Eino ADK 的 filesystem 接口方言，让基于 Eino 构建的 Agent 无需关心底层 Sandbox 的通信细节。

设计模式：Adapter（适配器模式）+ Facade（门面模式）
核心抽象：Backend 结构体统一封装文件操作与流式命令执行
Eino 接口 sandbox-sdk-go API 说明
LsInfo client.File.ListPath 列出目录内容
Read client.File.ReadFile 按行范围读取文件（默认 1-2000 行）
Write client.File.WriteFile 写入/覆盖文件
Edit ReplaceInFile / Read+Write 单处替换或全局替换（replace_all）
GrepRaw client.File.GrepFiles 文件内容搜索，支持多行/上下文
GlobInfo client.File.GlobFiles 通配符批量匹配
ExecuteStreaming client.Shell.ExecCommand 流式命令执行（实际为一次性结果转流）

3.2 安全模型：纯目录隔离

系统采用纯目录级别隔离，不创建 Linux 用户、不依赖 sudo、不使用容器特权：

  • 工作目录基线：默认 /home/gem，可通过 DefaultWorkDir 配置
  • 用户子目录：/home/gem/<UserID>/，UserID 必须符合 ^[a-z_][a-z0-9_-]*[$]?$
  • 路径沙箱化：~、.、空路径、绝对路径溢出都会被重映射到用户目录内
  • 标记文件：首次初始化时写入 .sandbox 标记文件，触发服务端自动创建父目录
注意：当前实现对于溢出绝对路径使用 filepath.Base 截断，这可能丢失深层目录结构。 例如请求 /etc/passwd 会被映射到 /home/gem/<user>/passwd。

四、关键实现细节

4.1 路径解析（resolvePath）

func (b *Backend) resolvePath(path string) string {
    home := b.userHome()
    switch path {
    case "", ".", "~", "/":
        return home
    }
    if strings.HasPrefix(path, "~/") {
        return filepath.Join(home, path[2:])
    }
    cleaned := filepath.Clean(path)
    if strings.HasPrefix(cleaned, home+"/") || cleaned == home {
        return cleaned
    }
    // 外部路径 → 重映射为用户目录下的相对路径
    return filepath.Join(home, filepath.Base(cleaned))
}

路径解析策略简单直接，优先保证安全性（任何越界路径都被拉回用户目录）， 但存在语义损失：外部绝对路径仅保留 basename，深层路径信息被截断。

4.2 Edit 操作的双重策略

  • ReplaceAll=false（默认）：调用远端 ReplaceInFile API，由服务端执行精确单行/单处替换
  • ReplaceAll=true：先 ReadFile 全量读取，本地 strings.ReplaceAll，再 WriteFile 回写。替换次数为 0 时报错
风险点：ReplaceAll 策略存在竞态条件（Read-Modify-Write），并发写同一文件可能丢失更新。 另外 Edit 要求 old_string != "" 且 old_string != new_string，空旧字符串被显式拒绝。

4.3 ExecuteStreaming 的流式模拟

虽然接口名为 ExecuteStreaming，但底层 sandbox-sdk-go 的 ExecCommand 实际为同步一次性执行。当前实现通过 schema.Pipe 将同步结果包装为流式 Reader， 在 goroutine 中一次性发送完整输出后关闭管道。

这是一个伪流式实现：对调用方暴露 StreamReader 接口，但内部并无逐字节/逐行推流， 所有输出一次性到达。后续若服务端支持真流式，只需替换 goroutine 内部逻辑即可，接口无需改动。

4.4 初始化与目录探测

NewBackend 初始化时会调用 ensureHomeDir：

  1. 先尝试 ListPath 探测用户目录是否存在
  2. 若报错（目录不存在），则写入标记文件 .sandbox，利用服务端"自动创建父目录"特性完成目录初始化

该设计零权限依赖，不需要 mkdir 权限，完全依赖远端 Sandbox 服务的自动创建行为。

八、总结与评价

8.1 设计亮点

  • 职责单一：只做一件事——桥接 Eino 与 Sandbox，不掺杂业务逻辑
  • 安全优先：纯目录隔离、路径沙箱化、命令校验钩子，多层防御
  • 零权限部署：不依赖 sudo、不创建系统用户，降低运维复杂度
  • 接口解耦：暴露 StreamReader 接口，为将来真流式预留升级空间
  • 环境一致性：dev/release/prod 配置结构完全一致，降低环境差异风险

8.2 潜在改进点

  • 路径语义：当前 resolvePath 对外部绝对路径仅保留 basename，可考虑改为拒绝或更明确的安全策略
  • 竞态条件：Edit 的 ReplaceAll 分支存在 RMW 竞态，建议增加乐观锁或走服务端原子操作
  • 流式升级：当前为伪流式，长命令大输出时会占用内存缓冲，建议跟进服务端 SSE/WebSocket 真流式
  • 代码规范：.gitlab-ci.yml 中 GOPROXY 重复定义；Dockerfile 大量注释代码建议清理或说明保留原因
  • 测试覆盖：未在仓库中发现单元测试文件（.gitignore 忽略了 test.go），建议补充核心路径解析与边界条件测试

8.3 适用场景

本工程适合需要为 AI Agent（基于 Eino ADK）赋予安全沙箱化代码执行与文件操作能力的场景， 例如：自动化代码审查、AI 辅助编程、Agent 自动修复 Bug、远程实验环境等。 由于采用目录级隔离而非容器级隔离，适用于可信多租户或单租户多会话场景， 不建议用于完全不可信的多租户强隔离场景。
