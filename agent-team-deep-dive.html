1. 整体架构概览 Agent Team

Claude Code 的 Agent Team（亦称 Swarm）是一个分层混合架构：根据场景不同，同时存在三套通信协议栈：

本地多 Agent 协作

一个 Leader + 多个 Teammate 的本地多进程/同进程协作，核心依赖文件系统 Mailbox + tmux/iTerm2 Pane 管理。

远程 CCR 连接

本地客户端 ↔ 云端 CCR（Compute Platform）会话，当前主推 SSE + HTTP POST（v2），兼容 Legacy WebSocket（v1）。

本地直连服务器

Direct Connect 模式：本地启动 WebSocket 服务器，客户端通过 NDJSON 格式直接通信。

Hooks 扩展系统

用户可扩展的生命周期钩子，支持 6 种类型 × 28 种事件，深度嵌入工具调用、会话生命周期和消息流中。

2. 多 Agent 发起机制（tmux）

2.1 双轨架构：Pane-based vs In-process

项目在运行时通过 isInProcessEnabled() 决定 teammate 创建方式：

模式 后端 适用场景
Pane-based tmux / iTerm2 用户在交互式终端中使用（默认）
In-process 同一 Node.js 进程 非交互式会话、无 tmux 环境、或显式 --teammate-mode in-process

2.2 tmux 的核心角色

tmux 是 Pane-based 模式的主力后端（utils/swarm/backends/TmuxBackend.ts），职责包括：

  1. Pane 生命周期管理：createTeammatePaneInSwarmView() 创建 pane、killPane() 销毁、hidePane()/showPane() 隐藏/恢复。
  2. 布局管理：
    • 若 Leader 在 tmux 内：Leader pane 占左侧 30%，teammates 在右侧 70% 内水平/垂直分割（main-vertical 布局）。
    • 若 Leader 不在 tmux 内：创建独立的 claude-swarm session（socket 隔离，名称 claude-swarm-${PID}），teammates 在 swarm-view window 中使用 tiled 布局。
  3. 命令投递：通过 sendCommandToPane() 向 pane 发送启动命令，teammate 进程由此诞生。
  4. 视觉标识：为每个 teammate pane 设置边框颜色和标题，便于区分。

2.3 每个 Agent 的运行隔离级别

Pane 级别独立，Session/Window 级别共享

Pane-based 模式下每个 agent 是独立 pane（共享 window/session）；In-process 模式下共享同一 Node.js 进程，通过 AsyncLocalStorage 隔离上下文。

2.4 启动流程

TeamCreateTool.call() // 阶段一：创建团队 └── tools/TeamCreateTool/TeamCreateTool.ts AgentTool.call() // 阶段二：发起 agent └── tools/AgentTool/AgentTool.tsx └── spawnTeammate() // tools/shared/spawnMultiAgent.ts ├── handleSpawnSplitPane() // 默认：tmux/iTerm2 分 pane ├── handleSpawnSeparateWindow() // legacy：独立 tmux window └── handleSpawnInProcess() // in-process 模式

阶段一：Team 创建

  • 检查 leader 是否已在管理团队（appState.teamContext?.teamName），限制一 leader 一团队。
  • 创建 ~/.claude/teams/{team_name}/config.json（TeamFile），写入 leader 信息（leadAgentId 固定为 team-lead@teamName）。
  • 初始化对应 task list 目录。
  • 更新 AppState.teamContext，记录 teamName、leadAgentId、teammates 映射。

阶段二：Teammate Spawn

当调用参数同时包含 name 和 team_name 时，走 teammate spawn 分支：

  1. 检测后端：detectAndGetBackend() → 选中 TmuxBackend 或 ITermBackend。
  2. 创建 pane：createTeammatePaneInSwarmView() → 实际 split pane。
  3. 构建 CLI 启动命令（再次运行 Claude Code CLI）：
    claude --agent-id researcher@my-team --agent-name researcher --team-name my-team --parent-session-id xxx ...
  4. 投递到 pane：sendCommandToPane(paneId, spawnCommand)。
  5. 通过 mailbox 发送初始 prompt：writeToMailbox(sanitizedName, { from: TEAM_LEAD_NAME, text: prompt }, teamName)。

2.5 Leader / Teammate 层级协调

项目中存在严格的 leader/teammate 层级：

  • Leader：创建 team 的主 agent，固定名称为 TEAM_LEAD_NAME = 'team-lead'。Leader 不设置 CLAUDE_CODE_AGENT_ID，因此 isTeammate() 返回 false。
  • Teammate：通过 AgentTool spawn 的子 agent，拥有 agentId（格式 name@teamName）。isTeammate() 返回 true。

3. Agent 间通信协议

3.1 三层通信架构

场景 协议版本 读方向 写方向 核心文件
CCR 远程 v1 (Legacy) WebSocket (wss://) HTTP POST (Session Ingress) WebSocketTransport.ts + HybridTransport.ts
v2 (Current) SSE (Server-Sent Events) HTTP POST (/worker/events) SSETransport.ts + CCRClient.ts
Direct Connect - 本地 WebSocket + NDJSON（每行一个 JSON） directConnectManager.ts
Agent Team / Swarm - 文件系统 Mailbox + 进程管理后端 teammateMailbox.ts

3.2 CCR v2 初始化流程

1. POST /v1/code/sessions (OAuth) → created sessionId 2. POST /v1/code/sessions/{id}/bridge (OAuth) → worker_jwt + worker_epoch + api_base_url 3. buildCCRv2SdkUrl(api_base_url, sessionId) → sessionUrl 4. createV2ReplTransport({ sessionUrl, ingressToken: worker_jwt, epoch: worker_epoch }) └─ SSETransport(sseUrl) [读] └─ CCRClient(sessionUrl) [写 + 心跳] 5. sse.connect() → 打开 SSE 流 6. ccr.initialize(epoch) → PUT /worker {worker_status: 'idle'} 7. 启动 JWT refresh scheduler（到期前 5min proactive 刷新）

3.3 核心消息结构（Tagged Union）

所有消息都使用 type 字段做 discriminated union 路由：

// 控制协议请求
{
  type: "control_request",
  request_id: string,       // 用于 req/res 配对
  request: {
    subtype: "initialize" | "interrupt" | "can_use_tool" | "set_model" | ...,
    // subtype 特定字段...
  }
}

// 控制协议响应
{
  type: "control_response",
  response: {
    subtype: "success" | "error",
    request_id: string,
    response?: { ... },      // success 时的负载
    error?: string           // error 时的描述
  }
}

3.4 Agent Team Mailbox 消息结构

Mailbox 路径：~/.claude/teams/{team_name}/inboxes/{agent_name}.json

type TeammateMessage = {
  from: string;             // 发送者 agent 名
  text: string;             // 消息正文（通常是嵌套 JSON）
  timestamp: string;
  read: boolean;
  color?: string;
  summary?: string;
}

text 字段内部可嵌套多种结构化消息类型：

  • idle_notification — 子 agent 报告空闲/中断/失败
  • permission_request / permission_response — 权限协商
  • plan_approval_request / plan_approval_response — 计划审批
  • shutdown_request / shutdown_approved / shutdown_rejected — 优雅关闭
  • task_assignment — 任务分配
  • sandbox_permission_request / sandbox_permission_response — 沙箱网络权限

3.5 消息路由与去重

CCR 服务端路由

通过 session_id 路由到正确的 SSE/WebSocket 连接；v2 通过 worker_epoch 确保只有一个 worker 实例活跃（旧 epoch 收到 409 淘汰）。

Bridge 层入站路由

handleIngressMessage 是核心路由函数，使用 BoundedUUIDSet（环形缓冲区）防止消息回显和重复投递。

Agent Team 路由

Mailbox 点对点路由：通过文件路径 teams/{team}/inboxes/{agent}.json 直接定位目标 agent；广播通过遍历 TeamFile.members 实现。

SeqNum 续传

SSE 使用 sequence_num 和 event_id 保证有序投递，支持 from_sequence_num / Last-Event-ID 断点续传。

4. Subagent 模式下 Root → Subagent 协议

4.1 三种 Subagent 标识系统

类型 标识方式 上下文传递 返回路径
AgentTool Subagent AgentId UUID（a-...） 直接 Message[] 数组 + ToolUseContext Generator yield（sync）或 <task-notification> XML（async）
Teammate (Swarm) name@teamName 文件系统 Mailbox JSON Mailbox JSON + idle 通知
Remote Agent CCR Session ID WebSocket SDKMessage WebSocket + HTTP

4.2 AgentTool 的两种上下文传递路径

路径 A：Fork 路径（subagent_type 省略）

由 buildForkedMessages()（forkSubagent.ts）构建：

  • 保留父 agent 的完整 assistant message（所有 tool_use 块），最大化 prompt cache hits。
  • 后续跟一个 synthetic user 消息，包含：
    • placeholder tool_result 块，文本固定为 "Fork started — processing in background"
    • 每个子 agent 的指令包裹在 <fork-boilerplate> XML + FORK_DIRECTIVE_PREFIX 中
  • 防递归：isInForkChild() 扫描对话历史中的 <fork-boilerplate> 标签。

路径 B：普通路径（subagent_type 指定）

由 runAgent()（runAgent.ts）执行：

  • Subagent 获得自己的 system prompt（来自 agent definition 的 getSystemPrompt）。
  • 仅含一个简单 user 消息，内容为 prompt 参数。
  • 除非显式通过 forkContextMessages 传递，否则不继承父 agent 对话上下文。

4.3 同步 vs 异步通信

模式 触发条件 行为 结果交付
同步（阻塞） run_in_background !== true，非强制 async，非 fork，非 coordinator Root 阻塞在 runAgent() 的 AsyncGenerator 上，实时迭代消息 通过 onProgress 回调返回 AgentToolProgress；最终返回 AgentToolResult
异步（Fire-and-Forget） run_in_background === true、agent 定义 background: true、coordinator 模式、fork subagent registerAsyncAgent() 创建 LocalAgentTaskState，runAgent() 被 void detached 执行 完成后 enqueueAgentNotification() 构造 <task-notification> XML 注入父 agent 消息队列

4.4 与 Agent Team 协议的对比

核心差异

Subagent（AgentTool）与 Teammate（Swarm）虽然在调用时都可能用到 AgentTool，但通信协议完全不同：

  • Subagent：同进程内直接函数调用（runAgent()），消息通过内存中的 Message[] 和 AsyncGenerator 传递，身份用 UUID 标识。
  • Teammate：通过文件系统 Mailbox 轮询（500ms）实现跨进程/跨 pane 通信，身份用 name@teamName 标识。

只有远程 Agent才使用 WebSocket/SSE 等网络协议；本地 Subagent 和 Teammate 都不走网络。

5. Hooks 运行机制

5.1 Hooks 是什么？

Hooks 是 Claude Code 提供的用户可扩展的生命周期钩子系统，允许用户在运行过程中的特定时机，执行自定义的命令、脚本、LLM 提示词、HTTP 请求或 TypeScript 回调函数。它类似于 Git hooks，但深度嵌入 AI 会话的完整生命周期。

5.2 六种注册来源

来源 注册方式 作用域 持久化
Settings 文件 settings.json 中的 hooks 字段 用户 / 项目 / 本地 / 策略 磁盘
插件 (Plugins) 插件 manifest 声明，通过 registerHookCallbacks 全局 随插件安装
技能 (Skills) Skill frontmatter 声明，转为 session hooks 当前会话 内存
Agent frontmatter Agent 定义中的 hooks，自动转为 SubagentStop 等 Agent 子会话 内存
SDK 回调 外部 SDK 调用 registerHookCallbacks 全局 内存
Session 函数钩子 代码中调用 addFunctionHook / addSessionHook 单个会话 内存

5.3 加载流程

1. captureHooksConfigSnapshot() // 从 settings.json 读取并缓存 2. getHooksFromAllowedSources() // policy_settings 过滤 3. getHooksConfig() // snapshot + registered + session 三层合并 4. getMatchingHooks() // 按事件类型和 matcher 模式匹配，按来源去重

5.4 执行流程

核心入口：executeHooks()（REPL 内）和 executeHooksOutsideREPL()（REPL 外）。

executeHooks() ├── 安全检查：shouldSkipHookDueToTrust()（交互模式需 workspace trust） ├── 获取匹配钩子：getMatchingHooks(appState, sessionId, hookEvent, hookInput) ├── 生成进度消息：为每个 hook 生成 progress message（UI 展示） ├── 并行执行：Promise.all 并行运行所有匹配 hooks │ ├── command 钩子 → execCommandHook() → spawn shell │ ├── prompt 钩子 → execPromptHook() → 调用 LLM │ ├── agent 钩子 → execAgentHook() → 调用 agent 验证 │ ├── http 钩子 → execHttpHook() → HTTP POST │ ├── callback 钩子 → hook.callback(...) → 内存 TS 回调 │ └── function 钩子 → executeFunctionHook() → 内存 TS 回调 ├── 结果聚合：processHookJSONOutput() 解析 JSON 输出 └── 产出：yield AggregatedHookResult（message / blockingError / permissionBehavior）

5.6 Hooks 输入输出格式

输入（HookInput）— 通过 stdin 传入 command/http 钩子

{
  "session_id": "uuid",
  "transcript_path": "/path/to/session.jsonl",
  "cwd": "/current/working/dir",
  "permission_mode": "ask",
  "agent_id": "subagent-uuid-or-null",
  "agent_type": "general-purpose-or-null",
  // 事件特定字段：
  // PreToolUse → tool_name, tool_input, tool_use_id
  // Stop → stop_hook_active, last_assistant_message
  // UserPromptSubmit → prompt
}

输出（HookJSONOutput）— 通过 stdout 返回

{
  "continue": false,
  "stopReason": "可选的停止原因",
  "suppressOutput": false,
  "decision": "approve | block",
  "reason": "解释",
  "systemMessage": "给用户看的警告",
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "allow | deny | ask",
    "updatedInput": { "filePath": "modified" },
    "additionalContext": "补充上下文"
  }
}

// 异步响应
{ "async": true, "asyncTimeout": 30000 }

Exit Code 语义

  • 0：成功
  • 2：阻塞错误（blocking error），会阻止工具执行或继续对话
  • 其他非零：非阻塞错误（仅记录）

5.7 stopHooks 是什么？

stopHooks 不是单一钩子，而是指 query/stopHooks.ts 中 handleStopHooks() 所 orchestrate 的一整套后采样钩子集合，在每轮模型回复完成后触发。

执行内容按顺序包括：

  1. 保存缓存安全参数（为 REPL /btw 命令和 SDK side_question 保存上下文快照）。
  2. Job 分类（TEMPLATES feature）。
  3. Prompt Suggestion：分析对话历史，生成后续提示词建议（fire-and-forget）。
  4. Memory Extraction（EXTRACT_MEMORIES feature）：提取并保存会话记忆。
  5. Auto-Dream：触发自动背景思考。
  6. Computer Use 清理。
  7. 用户自定义 Stop Hooks：执行 settings.json 中配置的 Stop 或 SubagentStop 钩子。
  8. Teammate 钩子（若是 teammate 模式）：TaskCompleted hooks + TeammateIdle hooks。

5.8 Hooks 与核心系统的集成关系

与工具调用

PreToolUse 可阻止执行（decision: 'block'）、自动批准（decision: 'approve'）、修改输入（updatedInput）。PostToolUse 可审查输出并阻止继续对话。

与 Agent 运行

SubagentStart / SubagentStop 在 Agent 启动/完成时触发。Agent frontmatter 中的 Stop 钩子自动转为 SubagentStop。

与消息处理

UserPromptSubmit 可在用户输入提交后、发送给模型前注入 additionalContext 或阻止输入。

与消息流

执行结果通过 AggregatedHookResult 返回：progress 消息、attachment 消息（hook_success / hook_blocking_error 等）、systemMessage 注入。

6. 协议对比总结

维度 AgentTool Subagent Teammate (Swarm) Remote Agent (CCR)
进程 同一 Node.js 进程 同进程（in-process）或独立 OS 进程（tmux） 远程 CCR 容器
身份标识 AgentId UUID（a-...） name@teamName CCR Session ID
通信协议 内存直接调用（AsyncGenerator） 文件系统 Mailbox JSON（500ms 轮询） SSE + HTTP POST（v2）或 WebSocket（v1）
上下文传递 Message[] 数组 + ToolUseContext Mailbox JSON 文件 WebSocket SDKMessage
返回路径 Generator yield（sync）或 <task-notification> XML（async） Mailbox JSON + idle 通知 WebSocket + HTTP
隔离级别 createSubagentContext() 克隆状态 AsyncLocalStorage（in-process）或 OS 进程 完全远程隔离
Cache 策略 Fork 路径共享字节级相同 system prompt & tools 独立上下文 独立
一句话总结

Claude Code 的 Agent Team 是一个去中心化、分层混合、安全优先的协作系统：本地 Subagent 走内存直接调用，本地 Teammate 走文件 Mailbox + tmux 进程管理，远程 Agent 走 SSE/WebSocket 网络协议；Hooks 则像神经中枢一样，在 28 个关键生命周期节点上提供可扩展的拦截与增强能力。
