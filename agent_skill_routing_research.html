⚙️现有 Agent 调用 Skills 的核心流程

当前主流框架的 Skill 调用机制可归纳为"渐进式披露（Progressive Disclosure）"架构，核心思想是：不要让 LLM 在每次推理时都看到所有 Skills 的全量内容。

1.1 四层加载模型（Claude Code / SKILL.md 标准）

这是目前社区最成熟的 Skill 结构，被 Anthropic、Cursor、Claude Code 等广泛采用：

层级 组件 内容类型 加载策略 Token 权重
L1 Metadata Skill 名称、描述、版本、触发词 Always-On（常驻上下文） < 1%
L2 Instruction SKILL.md 正文规则 On-Demand（命中后加载） 5-10%
L3 Reference 外部文档、手册、规范 Context-Triggered 高（可变）
L4 Script Python/Shell 脚本 Execution-Only 零（不读代码）
用户Query → 匹配L1 Metadata（描述+触发词）→ 命中后注入L2 Instruction → 执行中按需拉取L3 Reference → 调用L4 Script产生副作用

实测数据显示，这种架构能将上下文 Token 消耗降低 60%-80%，同时显著提升长文本任务中的指令遵循准确率。

1.2 各主流框架的路由机制对比

框架 路由方式 Skill/Tool 暴露策略 多Agent支持
Claude Code 描述匹配 + 触发词 全量Metadata常驻，Body按需加载 Subagent 执行
Google ADK LLM-driven 动态路由 LlmAgent 的 description 字段用于路由决策；sub_agents 层级委托 层级树（Coordinator → Specialist）
LangGraph 显式状态机（Nodes + Edges） 需手动定义每个Node的Tool集 图结构，确定性路由
OpenAI Agents SDK Function Calling 原生路由 全量Tools注入System Prompt 有限
MCP Protocol 标准化工具发现 Server 注册能力列表，Client 按需调用 跨框架互通
关键洞察：Google ADK 的 LlmAgent 在层级委托时，本质上是用 LLM 做 Skill 路由——Coordinator Agent 根据用户 Query 和各 Specialist Agent 的 description 决定调用哪个子代理。这与 Claude Code 的 Metadata 匹配是同一类思路，只是 ADK 用代码显式定义了层级。

🚨万级 Skill 场景下的核心瓶颈

当 Skill 数量达到 1万+ 时，上述常规架构会遭遇三个根本性瓶颈：

瓶颈一 上下文爆炸（Context Window Inflation）

Hermes Agent 社区的真实案例显示：仅 243 个 Skills 的 <available_skills> 块就消耗约 10-15K tokens，且每个 turn 都无条件注入。按此推算，1万 Skills 的 Metadata 部分（假设每个 Skill 描述平均 100 tokens）将直接占满 100万 tokens，远超任何现有模型的上下文窗口。

瓶颈二 路由精度崩塌（Routing Accuracy Collapse）

阿里巴巴团队在包含 ~80K Skills 的基准测试中发现：当 Skill 池存在大量重叠（heavy overlap）时，仅暴露 Metadata（name + description）而隐藏 Body 内容，会导致路由准确率下降 31-44 个百分点。这意味着"只给LLM看名字"的做法在万级场景下基本失效。

瓶颈三 注意力稀释（Attention Dilution）

SkillReducer 团队对 55,315 个公开 Skill 的大规模分析发现：

  • 26.4% 的 Skill 完全缺少路由描述
  • 60%+ 的 Body 内容是非 Actionable 的（背景知识、示例、解释性文字）
  • Reference 文件单次调用可注入数万 tokens

这些噪声内容会严重稀释模型注意力，导致"指令漂移"。

🔍检索层：从"全量暴露"到"向量召回 + 重排序"

这是解决上下文爆炸的首要手段。

方案 A：Embedding-based 粗排 + Cross-encoder 精排

生产环境数据显示：

  • 基于预计算 Skill Embedding 的内存检索（100 Skills 规模）：5-15ms
  • BERT-base Cross-encoder 重排序：额外 10-30ms
  • 效果：将误路由率从 ~8% 降至 ~2%

方案 B：Full-text Retrieve-and-Rerank（SkillRouter）

阿里巴巴提出的 SkillRouter 是专门针对万级 Skill 的轻量方案：

  • 1.2B 参数 的端到端 pipeline（Embedding + Reranker）
  • 直接对 Skill 的 Full Text（Name + Description + Body） 做检索
  • 在 ~80K Skill pool 上达到 74.0% Hit@1
  • 比 16B 参数的强基线 快 5.8 倍，参数量少 13 倍

关键技术点：

  1. False-negative Filtering：在训练时过滤掉被错误标记为负例的同类 Skill
  2. Listwise Reranking Loss：让模型在排序时考虑 Skill 间的相对关系，而非独立打分

方案 C：Hierarchical Taxonomy Routing

在检索前增加一层领域分类器：

用户Query → 意图分类（Domain Classifier）→ 缩小候选集到特定领域（如"DevOps"） → 领域内Embedding检索Top-K → Cross-encoder精排 → 注入LLM上下文

这能将 1万 Skills 的候选空间先压缩到数百级别。

🗜️压缩层：Skill 内容去膨胀（Debloating）

SkillReducer 两阶段压缩框架：

Stage 1：Routing Layer 优化

  • 用 Adversarial Delta Debugging 压缩 Skill 描述
  • 将描述拆分为语义子句（Semantic Clauses），递归测试每个子句对路由决策的必要性
  • 保留 1-minimal 子集，再对每个子句做短化改写
  • 效果：48% 描述压缩率

Stage 2：Body 结构重组

  • 基于 Taxonomy 的分类器将 Body 拆分为：
    • Actionable Core Rules（必须常驻）
    • Examples & Background（转为 On-demand 模块）
  • 通过 Faithfulness Check 和 Self-correcting Feedback Loop 验证压缩后功能等价性
  • 效果：39% Body 压缩率，且功能质量反而提升 2.8%（Less-is-more 效应）
对 1万 Skills 的实操建议：
  • 建立自动化流水线：新 Skill 入库时自动运行 SkillReducer
  • 设置 Token 预算上限：单个 Skill 的 Metadata ≤ 50 tokens，Body ≤ 500 tokens
  • 对超过预算的 Skill 触发告警或拒绝入库

🏗️架构层与缓存层优化

三层路由架构（生产级最佳实践）

Layer 1: Intent Classifier（意图分类器）
轻量模型（如 DistilBERT）或规则引擎 · 将 Query 映射到 Domain / Category
延迟 < 5ms · 过滤掉 90% 无关 Skills
Layer 2: Skill Registry Router（技能注册表路由）
向量检索 Top-K=20~50 · 支持多路召回（描述向量 + 触发词倒排 + 标签过滤）
延迟 10-30ms
Layer 3: Policy Gate（策略门）
权限检查、风险评级、Safety Classifier · 硬过滤：未授权 Skill 直接拦截
延迟 < 5ms
Layer 4: LLM Executor（执行层）
仅注入最终筛选出的 3~5 个 Skill 的完整内容 · 保留原生 Function Calling 能力

懒加载（Lazy Loading）策略

  • Mode 1（保守）：System Prompt 中只保留 Skill Name 列表，Description 和 Body 在 LLM 决定调用时实时拉取
  • Mode 2（激进）：System Prompt 中完全不注入 Skills，LLM 输出 skill_view("skill_name") 请求，由外部系统返回内容

优化点 策略 效果

✅总结与行动建议

1 不要试图让 LLM 在 System Prompt 里看到 1万个 Skills

无论框架是 ADK、LangGraph 还是 OpenAI Agents SDK，万级全量注入都是不可行的。

2 Full-text 检索是万级路由的关键信号

SkillRouter 的研究证明，仅看 Metadata 在重叠严重的 Skill 池中会导致 31-44pp 的精度损失，必须让检索层看到 Body 内容。

3 建立 Skill 入库的自动化质量门禁

参考 SkillReducer 的发现，26.4% 的公开 Skill 缺少路由描述，60%+ 的 Body 是噪声。入库时必须强制：描述 ≤ 50 tokens，Body 经过 Actionable/Non-actionable 分类，触发词 ≥ 3 个。

4 采用"三级路由 + 渐进式披露"的混合架构

意图分类器负责 Domain 过滤，向量检索负责语义召回，Policy Gate 负责安全隔离，最终只向 LLM 暴露 3-5 个精选 Skill 的完整内容。

5 监控三个核心指标

Route Accuracy（路由准确率）、Fallback Rate（未命中率）、p95 Latency（端到端延迟）。健康的注册表表现为：新增 Skill 不会导致 Fallback 和安全事件不成比例增长。
