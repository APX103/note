01 项目概览与定位

一句话定位

Agent Swarm 是"一个能真正把用户一句话变成可运行工程"的多 Agent 协作运行时。它不是单纯的 Agent 框架，而是围绕计划 → 分派 → 执行 → 审查 → 交付完整闭环构建的、带有 Docker 隔离和 A2A 通信协议的生产原型。

核心能力

  • ✓ 自然语言请求自动拆解为计划与 API 契约
  • ✓ 多 Agent 并行调度（前端 + 后端可并发）
  • ✓ Worker 在隔离 Docker 容器中运行
  • ✓ 编排器主动审查并对齐前后端接口
  • ✓ 外部 Agent 注册与调用（OpenAI/A2A/MCP/CLI）
  • ✓ WebSocket 实时进度流
  • ✓ 会话持久化与多轮对话（近期新增）

技术栈

  • ● Python 3.13 + FastAPI + Uvicorn
  • ● OpenAI SDK 调用 GLM Coding Plan（glm-4.7）
  • ● Docker SDK 管理 warm pool 容器
  • ● Redis 作为 Agent 注册中心
  • ● A2A JSON-RPC over HTTP
  • ● pytest + pytest-asyncio 测试
  • ● Pydantic v2 模型校验

编排器 Orchestrator

编排器本身也是一个 LLM Agent，通过 OpenAI function-calling 循环驱动。它依次调用 plan_task、dispatch_agent(s)、read_artifacts、finalize 等工具。

核心设计是"API 契约先行"：plan_task 阶段生成的前后端接口定义会被自动注入到每个 Worker 的任务描述中，显著降低联调成本。

统一调度器 Dispatcher

Dispatcher 把 Docker Worker 和外部注册 Agent 抽象为统一的 DispatchBackend。Orchestrator 不感知目标到底是容器内 Worker 还是外部服务。

调度器内置候选解析、健康预检、重试故障转移、per-dispatch 超时、全局并发背压和 per-target 熔断器，具备较强的生产级路由可靠性。

Worker 容器池

ContainerPoolManager 在启动时预创建 N 个 warm Docker 容器，通过 checkout/return 机制复用。每个容器绑定固定端口（宿主机 9001+），挂载共享产物目录和注入式 config.json。

Worker 容器内运行一个 FastAPI A2A Server，接收任务后调用 LLM 在后台执行，并通过轮询接口返回进度。

外部 Agent 网关

/api/v1/agents/* 提供注册、心跳、注销、列表和调用能力。Agent 信息存储在 Redis，支持按 skill 发现。AdapterManager 根据协议类型自动选择 A2A/OpenAI/MCP/CLI 适配器。

这套机制让第三方 Agent 可以像内部 Worker 一样被编排器调度。

会话与产物模型

每个任务对应一个产物目录 shared_output/tenants/{tenant_id}/tasks/{task_id}/。近期新增的 SessionManager 允许同一个 session_id 跨多轮对话复用同一个 work folder 和对话历史，解决了"每次 chat 都是独立任务"的痛点。

finalize 工具在任务结束前会执行强制审查：每个已分派的 Agent 必须在自己的角色子目录下产出文件，否则拒绝完成并要求返工。这是一个简单但有效的质量门。

03 核心亮点（优点）

1

完整的端到端闭环已被真实验证

项目不只是停留在设计文档，而是已经通过 Docker E2E 验证：从用户请求到生成前后端分离的 TODO 应用，再到后端 CRUD 实跑和浏览器 Headless 渲染验证。这是同类型项目中最难得的"金标准"。

2

统一调度抽象降低了架构复杂度

Docker Worker 与外部 Agent 被抽象为同一套 DispatchBackend 接口，编排器无需关心目标类型。这种抽象让外部 Agent 接入变得自然，也为未来扩展更多后端（如 Kubernetes Pod、本地进程）预留了接口。

3

Dispatcher 具备生产级可靠性设计

健康预检、重试故障转移、超时控制、全局并发背压、per-target 熔断、结果缓存降级——这些机制叠加在一起，使调度层在面对单点故障时表现稳健。结果缓存降级更是一个贴心的"优雅失败"设计。

4

可插拔编排器与外部 Agent 机制

通过 OrchestratorResolver 支持 builtin/external 两种编排器，并在外部不可用时自动回退。这种设计保留了"换大脑"的灵活性，同时不牺牲稳定性。

5

测试覆盖率高，且坚持 TDD

280 个测试通过，测试/代码行数比接近 1:1。关键模块如 dispatcher、orchestrator、gateway、session、幂等、取消、死信、结果缓存均有专门测试。NEXT-STEPS 显示项目按"每 wave 一 commit + TDD"的节奏推进，工程纪律较强。

6

文档体系完善，踩坑经验真实

除了 README 外，还有 CLAUDE.md（重构宪法）、USAGE.md（实测踩坑指南）、onboarding.md、NEXT-STEPS.md、iteration-progress.md 以及多篇 HTML 调研文档。特别是 USAGE.md 里记录的 docker.io 不可达、模型名、容器名冲突等坑，都是实战积累，对团队 onboarding 价值极高。

7

代码风格规范，防御式编程到位

CLAUDE.md 中的"重构宪法"明确禁止引入新依赖、禁止改变 API 签名、禁止删除测试等，并强调防御式编程、错误显式传播、资源管理和类型安全。实际代码中也能看到大量显式异常处理和日志记录，符合这些原则。

04 问题与风险（缺点）

风险项 等级 说明
单进程内存状态 高 幂等索引、死信、结果缓存、熔断、per-tenant 信号量全是 in-memory，多 uvicorn worker 部署会失效。
取消不传播到 Worker 高 cancel_running 只取消编排器侧 asyncio 任务，Docker Worker 内的 LLM 调用仍在继续烧 token。
/api/v1/agents/invoke 无鉴权 高 任何人都能调用已注册的外部 Agent，存在未授权调用风险。
Worker 任意 shell 执行 中 Worker 中 run_command 使用 subprocess(shell=True)，沙箱内仍需明确信任边界或命令白名单。
配置硬编码 中 DispatcherConfig、结果缓存 TTL/size、per-tenant 并发上限等参数未进配置文件，无法在线调整。
LLM 模型名默认值错误 中 config.py 默认 glm-coding-plan 不是真实模型名，新手未改配置会直接 400（已在使用文档中反复提醒）。
Worker 镜像更新成本高 中 Worker 是 baked 镜像，修改 worker.py 后必须重新 build，开发迭代效率受限。
warm pool 容器名冲突 中 编排器重启前必须手动清理 swarm-worker-{0..4}，否则启动失败。重启体验不友好。
编排循环硬编码 中 Orchestrator 的工具集合、系统提示词、最大迭代次数、审查规则都写在类内部，难以通过配置扩展新角色或新工具。
产物审查规则过于简单 低 当前只检查"每个 Agent 是否产出了文件"，不检查文件内容、语法、接口一致性。虽然编排器会主动联调，但审查门本身较浅。
外部编排器真实验证不足 低 external provider 的回退机制已测，但"真实成功的外部编排器"未端到端跑过。
可观测性仅停留在日志 低 只有 trace_id 串联日志，缺少延迟/失败率/队列深度/每 Agent 统计指标，不利于生产排障和容量规划。

需要重点展开的几项

单进程状态是最大生产化瓶颈

当前所有关键状态（幂等索引、死信、结果缓存、熔断器、per-tenant 信号量）都保存在 Python 进程内存中。这意味着：一旦使用多个 uvicorn worker 进程做水平扩展，状态就无法共享——同一个 Idempotency-Key 可能被重复处理，熔断状态会分裂，结果缓存只在单个进程内生效。NEXT-STEPS 已经把 Redis 持久化列为 P1.2，优先级正确。

取消机制有真实成本泄漏

用户通过 WebSocket 发送 cancel 后，编排器侧的任务确实会被取消，但已经 checkout 的 Docker Worker 容器内的后台任务仍在继续执行并调用 LLM。这在按 token 计费或资源紧张的场景下会造成直接经济损失。需要补 A2A tasks/cancel 或容器停止信令。

安全设计尚未起步

外部 Agent 网关缺少鉴权，Worker 内 shell 执行缺少白名单，CORS 配置为 allow_origins=["*"]。在暴露到公网或接入不可信 Agent 前，必须补齐 API key / token、命令白名单、CORS 限制。

默认配置对新用户不友好

config.py 中的默认模型名 glm-coding-plan 和默认存储路径 /home/apx103/... 都是原作者环境特定的值。虽然 USAGE.md 反复提醒，但默认配置仍应使用更安全、更中性的占位符。

05 改进建议

P0

Redis 持久化核心状态

把幂等索引、死信、结果缓存、熔断状态、per-tenant 信号量迁移到 Redis。每项保留 in-memory fallback，确保 Redis 故障时服务仍可降级运行。这是多进程部署的前提。

P0

补齐取消传播

在 A2A 协议中加入 tasks/cancel 端点，Worker 收到后取消后台 _run 任务；或直接停止对应容器。避免 cancel 后仍继续消耗 LLM token。

P0

外部 Agent 鉴权

为 /api/v1/agents/invoke 增加 API key / JWT token 校验；注册接口也应考虑身份验证，防止恶意 Agent 注入。

P1

配置外置与默认值清理

把 DispatcherConfig、结果缓存 TTL/size、per-tenant 并发上限等参数放入 default.yaml.example。默认模型名改为 "" 或真实模型名，默认路径改为相对路径或占位符。

P1

Worker 热更新机制

考虑通过 volume 挂载 src/agents 到容器，或引入动态重载机制，避免每次修改 worker 代码都重新 build 镜像，提升开发迭代效率。

P1

指标与可观测性

新增轻量 in-memory Metrics 模块，记录 dispatch 次数、成功率、延迟、候选尝试次数、每 Agent 统计，暴露 GET /api/v1/metrics（Prometheus 格式或 JSON）。

P2

编排器配置化

把工具定义、系统提示词、最大迭代次数、审查规则从 Orchestrator 类中抽出，允许通过 YAML 配置扩展新工具、新角色、新审查策略。

P2

产物深度审查

在 finalize 审查门中增加语法检查（如 Python 编译、HTML 解析）、API 契约匹配度检查，必要时调用 LLM 做产物质量评分，减少返工次数。

06 总结

Agent Swarm 是一个已经完成了从 0 到 1 验证的多 Agent 协作系统，而不是停留在概念阶段的玩具。它的核心优势在于：完整的 LLM tool-calling 编排闭环、统一的 Docker/外部 Agent 调度抽象、扎实的前后端联调能力、以及高覆盖率的测试和详实的文档。

但作为一个想要进入生产的系统，它还需要跨过几道坎：单进程状态持久化、取消传播、安全鉴权、配置外置是接下来最值得优先投入的四个方向。NEXT-STEPS 已经把这些识别得很清楚，剩下的就是按现有 TDD 节奏逐个落地。
