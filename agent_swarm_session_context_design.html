Agent Swarm 全量会话上下文存储设计

参考 Google ADK 的 Session / Event 设计思路，为 Agent Swarm 设计一套能够完整保存人与 Agent、Agent 与 Agent 之间对话的上下文存储方案，并支持事后复现现场。

📋 设计结论

建议引入事件优先（Event-First）的会话模型：把每一次用户输入、编排器思考、工具调用、工具结果、Agent 进度、状态变更都记录为不可变的 Event。Session 由"事件日志 + 状态表 + 产物元数据"组成，通过 SessionService 抽象层持久化到内存（开发）或 Redis（生产）。

这套设计与 Google ADK 的核心思路一致：用 Event 作为统一信息单元，用 author 区分说话者，用 invocation_id 区分一次完整请求周期，用 branch 隔离子 Agent 的私有上下文。

01 背景与问题

当前 SessionManager 的能力边界

目前 src/session/manager.py 中的 SessionState 只保存了：

  • session_id、tenant_id
  • work_dir（产物目录）
  • messages（编排器 LLM 的对话历史）
  • shared_context（plan_task 生成的计划）

它解决的是"多轮对话复用 work folder 和历史"，但缺失了中间过程的完整记录：编排器什么时候调用了哪个工具、参数是什么、Worker 返回了什么、Agent 之间如何互相调用、状态何时变化。这些信息一旦丢失，事后无法复现现场。

缺失的人-Agent 对话记录

当前只保存了最终进入编排器 LLM 的 messages。用户原始请求、WebSocket 事件、取消信号、API 层面的输入输出都没有被结构化保存。

缺失的 Agent-Agent 对话记录

编排器与 Worker 之间的 dispatch_agent、check_agent_status、read_artifacts 调用，以及 Worker 内部的 LLM 思考过程，都没有被记录为可回放的事件。

无法复现现场

当某个任务结果不符合预期时，开发者只能看日志文件，无法像翻聊天记录一样按时间线查看"谁说了什么、谁调了谁、返回了什么"。

状态变更无审计

shared_context 何时被更新、由哪个工具更新、更新前后的值是什么，没有可追溯的记录。

02 Google ADK 的设计启示

Google ADK 把会话管理拆成三个层次，每层职责单一：

Agent 层
业务逻辑所在，决定如何使用上下文和记忆。
Runtime 层
Runner 负责创建/销毁 Session，跟踪执行每一步。
Service 层
SessionService 负责与外部存储（内存/DB/Vertex）交互。

其中最值得借鉴的是 Event 作为统一信息单元：用户消息、Agent 回复、工具调用、工具结果、状态变更、控制信号都被封装成 Event，按时间顺序追加到 Session.events。这让"复现现场"变成了按顺序回放事件列表。

ADK Event 核心字段

字段 含义 Agent Swarm 映射
id 事件唯一 ID 每个工具调用、每次 Agent 回复都有唯一 ID
invocation_id 一次完整请求周期的 ID 对应一次 POST /api/chat
author 事件产生者 user / orchestrator / frontend-ux-pro / backend-engineer 等
content 消息/调用/结果内容 文本、function_call、function_response、progress
actions.state_delta 状态变更增量 shared_context 更新、session state 更新
branch 层级路径 多 Agent 嵌套时的上下文隔离，如 orchestrator.frontend
timestamp 事件发生时间 用于按时间线回放

为什么这套思路适合 Agent Swarm

  • ✓ 不可变追加：Event 一旦写入不修改，天然适合审计和回放。
  • ✓ 说话者清晰：author 字段能区分 user、orchestrator、各个 Worker Agent。
  • ✓ 层级隔离：branch 让子 Agent 看不到兄弟 Agent 的内部思考，只看到需要共享的结果。
  • ✓ 状态变更可审计：所有 state 更新通过 state_delta 事件体现，可追溯。
  • ✓ 存储可插拔：SessionService 抽象允许开发用内存、生产用 Redis/DB。

03 设计目标

G1: 全量记录

记录一次 POST /api/chat 周期内所有有意义的交互：用户输入、编排器 LLM 请求/响应、每一次工具调用与结果、Worker 进度、产物读取、状态变更、错误与取消。

G2: 可复现现场

给定 session_id，能够按时间线完整回放事件流；开发者可以像看聊天记录一样查看谁说了什么、调了什么、返回了什么。

G3: 支持多进程

默认实现从 in-memory 升级到可持久化的 Redis 后端，解决当前 SessionManager 只能单进程使用的问题。

G4: 向后兼容

现有 API 签名和调用方式不变；新设计以新增模块和可选注入的方式引入，老代码逐步迁移而非一次性重写。

G5: 上下文可控

支持按 author、event_type、branch 过滤事件，用于构建不同 Agent 可见的上下文窗口。

G6: 可扩展

新增事件类型和存储后端不需要修改核心模型，通过 Pydantic 联合类型和 SessionService Protocol 扩展。

04 核心概念设计

Event 数据模型

class Event(BaseModel):
    id: str                    # 事件唯一 ID (uuid)
    invocation_id: str         # 一次 /api/chat 请求周期 ID
    session_id: str
    tenant_id: str

    timestamp: float           # 创建时间
    author: str                # user / orchestrator / frontend-ux-pro / backend-engineer / dispatcher / system
    role: Literal["user", "assistant", "tool", "system"]
    event_type: str            # 见下方事件类型枚举

    branch: str = ""           # 层级路径，如 orchestrator / orchestrator.frontend
    parent_id: Optional[str]   # 父事件 ID，用于 tool_result → tool_call 关联

    content: Optional[dict]    # 文本/调用/结果的实际内容
    actions: EventActions      # 副作用：状态增量、产物增量、transfer 等

    metadata: dict             # 扩展字段：task_id, dispatch_id, agent_type, retry 等

class EventActions(BaseModel):
    state_delta: dict = {}          # session state 变更（类似 ADK）
    artifact_delta: dict = {}       # 产物变更：{path, action: created|updated|deleted}
    transfer_to_agent: Optional[str] = None  # 控制信号：转交给某个 Agent
    escalate: bool = False          # 是否需要人工介入

Session 数据模型

class Session(BaseModel):
    id: str
    app_name: str = "agent-swarm"      # 兼容 ADK 概念
    user_id: str = "default"           # 可替换为 tenant_id
    tenant_id: str = "default"

    work_dir: str                      # 产物工作目录
    state: dict = {}                   # key-value 会话状态表
    events: list[Event] = []           # 事件日志（核心）

    created_at: float
    last_update_time: float
    version: int = 0                   # 乐观锁，用于并发更新

class SessionService(Protocol):
    async def create_session(self, session_id, tenant_id, work_dir, initial_state=None) -> Session: ...
    async def get_session(self, session_id, tenant_id) -> Optional[Session]: ...
    async def append_event(self, session_id, tenant_id, event: Event) -> Session: ...
    async def update_state(self, session_id, tenant_id, state_delta: dict) -> Session: ...
    async def list_sessions(self, tenant_id, limit, offset) -> list[Session]: ...

上下文重建策略

编排器在每一轮对话开始时，不需要把整个 events 列表都塞进 LLM。而是：

  1. 按 branch 过滤出 orchestrator 主分支的事件；
  2. 把事件转换为 OpenAI messages（user_message → user，llm_response → assistant，tool_result → tool）；
  3. 当事件过多时，对早期事件做摘要压缩，保留最近 N 轮完整记录。

这样编排器既不会丢失多轮记忆，也不会被无关的 Worker 内部细节淹没。

Agent-Agent 对话如何记录

Agent Swarm 中 Agent-Agent 的"对话"本质上是编排器对 Worker 的调用链。建议按以下方式建模：

  • ● 父调用：tool_call 事件，author=orchestrator，content 含目标 agent_type 和任务。
  • ● 子执行：agent_progress 和 agent_response 事件，author=具体的 Worker Agent ID，parent_id 指向前面的 tool_call 事件。
  • ● 分支隔离：Worker 内部产生的 llm_request/llm_response/tool_call 等事件使用 branch="orchestrator.{agent_type}"，默认不向 orchestrator 主上下文暴露。

这样，事后查看某次 dispatch 时，可以通过 parent_id 找到完整的请求-响应链条。

与现有 SessionManager 的兼容

推荐采用"新增模块、逐步替换"的策略：

  1. 保留 SessionState 作为内存中的运行时缓存，但把它升级为 Session 的本地视图。
  2. 新增 src/session/service.py 提供 SessionService 抽象及 InMemory/Redis 实现。
  3. 新增 src/session/events.py 定义 Event 和 EventActions。
  4. 新增 src/session/recorder.py 作为统一事件记录器，被 orchestrator、dispatcher、worker 调用。
  5. 原有 messages 和 shared_context 继续保留，但改为从 events/state 派生，保证过渡期间不破坏现有逻辑。

06 存储选型

阶段一：内存实现（开发/测试）

InMemorySessionService 用 dict 保存 session，适合当前单进程开发和跑测试。它保持与现有 SessionManager 相似的语义，但接口统一为 SessionService。

优点：零依赖、实现快、测试简单。缺点：进程重启丢失、无法水平扩展。

阶段二：Redis 实现（生产）

RedisSessionService 把 session 和 events 序列化后存到 Redis。可用 Hash 存 session 元数据，用 List 或 Sorted Set 存 events（按 timestamp 排序）。

优点：项目已依赖 redis、支持多进程、速度快。缺点：事件列表过大时读取性能下降，可定期归档或分片。

Redis 存储结构建议

# Session 元数据
HSET swarm:session:{tenant_id}:{session_id}
  work_dir /abs/path
  state       {json}
  created_at  1234567890
  version     5

# 事件列表（按时间追加）
LPUSH swarm:events:{tenant_id}:{session_id} {event_json}
# 或按 score = timestamp 的 Sorted Set，便于范围查询
ZADD swarm:events:{tenant_id}:{session_id} {timestamp} {event_json}

# 索引：最近更新的 session（用于管理后台列表）
ZADD swarm:sessions:{tenant_id} {last_update_time} {session_id}

由于事件是追加-only 的，Redis List 天然合适。如果事件量很大，可以按 invocation_id 分 key，或者定期把老事件迁移到对象存储/SQLite。

08 实现路线图

1

Wave 1：事件模型 + InMemorySessionService

新增 src/session/events.py、service.py、recorder.py。在 orchestrator 的关键路径（tool_call/tool_result/finalize）埋点。保持现有 API 不变。

2

Wave 2：Dispatcher / Worker 埋点

在 dispatcher 记录 dispatch_request/attempt，在 Worker 侧记录 agent_progress/agent_response。让 Agent-Agent 调用链可追踪。

3

Wave 3：RedisSessionService + 多进程安全

实现 Redis 后端，替换当前的 in-memory SessionManager。引入 version 乐观锁，解决并发更新问题。同时把原有 in-memory 的幂等索引、死信等逐步 Redis 化。

4

Wave 4：查询 / 回放 API

实现 /api/sessions/{id}/events、/api/sessions/{id}/replay、/api/sessions/{id}/timeline，让开发者可以在浏览器或 CLI 中复现现场。

5

Wave 5：上下文压缩与分支隔离

基于 events 实现编排器的上下文重建：按 branch 过滤、早期事件摘要、保留最近 N 轮。让多轮对话既能记住历史，又不会超出 LLM 上下文窗口。

09 设计要点回顾

  • ✓ 用 Event-First 模型替代当前的 messages 列表，把所有交互变成不可变事件。
  • ✓ 用 author 区分 user/orchestrator/Worker，用 invocation_id 区分一次请求周期，用 branch 隔离子 Agent 上下文。
  • ✓ 用 SessionService 抽象层隔离存储，开发用内存，生产用 Redis。
  • ✓ 在 orchestrator、dispatcher、worker 关键路径埋点，形成完整的人-Agent 和 Agent-Agent 对话记录。
  • ✓ 提供 events/replay/timeline API，支持事后复现现场。
  • ✓ 保持向后兼容，新增模块逐步替换，不一次性重写现有 API。
