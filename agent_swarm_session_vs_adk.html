Agent Swarm Session 上下文方案 vs Google ADK 对比分析

在 GLM 新增了 session disk persistence 和多轮对话能力之后，重新评估 Agent Swarm 的上下文存储设计与 Google ADK 的异同，找出可以相互借鉴的地方，以及更合理的演进方向。

📋 结论先行

GLM 最新改动把 SessionManager 从纯内存升级为磁盘持久化（_session/context.json），实现了进程重启后仍能恢复对话历史和工作目录。这是一个重要进步，已经初步具备 ADK 中 Session 的"持久化会话"概念。

但当前方案仍停留在"保存 LLM 消息列表"层面，而 ADK 的核心是"Event-First"：把用户输入、Agent 回复、工具调用、工具结果、状态变更、进度事件都记录为结构化事件。后者才能支撑"复现现场"、"审计状态变更"、"Agent-Agent 对话追溯"等高级能力。

因此最合理的演进路径是：保留当前的 disk persistence 作为运行时视图，在其旁新增 Event 日志层，逐步从"消息列表"过渡到"事件流"。

01 当前实现速览

GLM 最近改动带来的变化

从最新 commit（f600077 和 d6b6b93）可以看到，src/session/manager.py 已经变成磁盘持久化的 SessionManager：

  • SessionState 保存 messages、shared_context、work_dir
  • 新增 save() 方法，把 messages 和 shared_context 写入 _session/context.json
  • get_or_create() 支持从磁盘恢复 session
  • ChatRequest 增加 session_id 字段
  • /api/chat 在编排结束后调用 session_manager.save(sess)

测试也覆盖了 save/restore 场景，说明这是一个经过验证的基础能力。

当前持久化结构

shared_output/tenants/{tenant}/sessions/{session_id}/
├── _session/
│   └── context.json          # {messages, shared_context, created_at}
├── _plan/project_plan.md
├── frontend/
├── backend/
└── _final/

context.json 里的 messages 是 OpenAI 格式的消息列表（system/user/assistant/tool），直接被编排器复用为 LLM 上下文。这很务实，但也限制了可观测性和可复现性。

02 ADK 设计速览

ADK 的三层职责

Agent 层
业务逻辑，决定如何生成响应、使用工具。
Runner / Runtime 层
负责执行 Agent，在关键节点生成 Event 并交给 SessionService。
SessionService 层
负责持久化 Session 和 Events（内存/数据库/Vertex AI）。

ADK Session 的组成

  • id / app_name / user_id：会话标识。
  • events：按时间顺序排列的事件日志，是核心。
  • state：key-value 会话状态表，通过 Event 的 state_delta 更新。
  • last_update_time：最后更新时间。

ADK Event 的关键字段

字段 作用
id事件唯一 ID
invocation_id一次完整用户请求周期
author事件产生者：user / agent 名 / tool
content消息/调用/结果内容
actions.state_delta状态变更增量
actions.artifact_delta产物变更记录
branch层级路径，用于多 Agent 隔离
timestamp事件发生时间

03 多维度对比

维度 Agent Swarm 当前 Google ADK
持久化 磁盘文件 _session/context.json 可插拔：InMemory / Database / VertexAI
存储单元 messages 列表 + shared_context events 列表 + state
事件类型 无显式事件类型，只有 role（system/user/assistant/tool） 丰富的 event_type：tool_call、tool_result、agent_progress、state_delta 等
说话者标识 只有 role，无法区分具体哪个 Agent author 字段：user / orchestrator / frontend-ux-pro 等
请求周期 无 invocation_id，按 session 维度聚合 invocation_id 区分一次完整请求周期
Agent-Agent 追溯 只能从 messages 中推测，无 parent/branch 关联 parent_id + branch 可完整追溯调用链
状态变更审计 shared_context 更新无记录 每次 state 变更都通过 state_delta 事件记录
复现现场 只能看 messages 和日志，无法按时间线回放 按 events 顺序回放，可复现完整执行过程
上下文重建 直接复用 messages 列表 从 events 过滤/摘要生成 LLM 上下文
查询 API 无 session events 查询接口 可通过 SessionService 查询事件流
多进程安全 文件读写无并发控制，多 uvicorn worker 可能冲突 数据库后端支持并发和乐观锁

04 已对齐 ADK 的地方

1

Session 持久化概念已落地

ADK 的核心价值之一是让会话跨请求/重启存活。GLM 改动后，Agent Swarm 已经通过磁盘文件实现了这一点，这是从"一次性任务"到"可持续对话"的关键跨越。

2

work folder 与会话绑定

ADK 用 ArtifactService 管理文件产物，Agent Swarm 直接用文件系统目录绑定 session。对于"生成可运行工程"这个场景，后者更直接、更高效。

3

编排器已能恢复历史

orchestrator.execute() 已经支持从 session.messages 恢复上下文，并追加新消息。这与 ADK Runner 从 Session 加载历史的行为一致。

05 与 ADK 的差距

1

没有 Event 抽象，只有消息列表

当前保存的是 OpenAI 格式的 messages，它丢失了"这是哪一次工具调用"、"这个工具结果对应哪个调用"、"这是哪个 Agent 说的"等元数据。ADK 的 Event 模型能保留这些关键信息。

2

无法区分 Agent-Agent 对话

编排器和 Worker 的交互被折叠进 tool_result 消息里，看不到中间的 agent_progress、dispatch 尝试、失败重试等信息。ADK 的 author + branch 可以清晰区分这些参与者。

3

状态变更无审计

shared_context 在 plan_task 时被设置、在后续可能被修改，但没有记录"何时、由谁、从什么改成什么"。ADK 要求所有 state 变更都通过 Event 的 state_delta 进行。

4

无法按请求周期回放

当前所有消息混在一个列表里，无法区分"哪几条消息属于某一次 /api/chat 调用"。ADK 的 invocation_id 可以精确划分一次完整请求周期内的事件。

5

缺少 SessionService 抽象

当前 SessionManager 直接操作文件系统，没有抽象层。未来要切换到 Redis 或数据库时需要重写。ADK 的 SessionService 协议让存储后端可插拔。

06 值得借鉴 ADK 的点

引入 Event 模型

不要只存 messages，而是把"用户输入、LLM 响应、工具调用、工具结果、Agent 进度、状态变更"都存为结构化 Event。这是实现可复现现场的基础。

author + invocation_id + branch

这三个字段能解决 Agent Swarm 里"谁在说话"、"属于哪次请求"、"子 Agent 上下文隔离"的问题。

state_delta 机制

把 shared_context 的变更变成事件化的增量，而不是直接修改状态。这样可以审计每次变更的来龙去脉。

SessionService 抽象

把存储逻辑从 SessionManager 里拆出来，定义 SessionService 协议。当前磁盘实现可作为 FileSessionService，未来加 RedisSessionService。

从 Events 派生 Messages

不要直接把 events 全部喂给 LLM。而是按 branch 过滤、早期摘要、保留最近 N 轮，再生成 messages。这样既有完整记录，又控制好上下文长度。

Event 查询/回放 API

提供 /api/sessions/{id}/events 和 /api/sessions/{id}/replay，让开发者可以按时间线查看完整交互过程。

07 Agent Swarm 更合理的地方

1

文件系统持久化更简单

对于"生成代码工程"这个核心场景，产物天然就是文件。把 session context 也存到同一份文件系统里，比引入数据库更简单、更直观。ADK 的 DatabaseSessionService 对当前项目来说反而过重。

2

work_dir 与产物目录一体化

Agent Swarm 把 session work folder 直接作为 Agent 产物目录，多轮对话产生的代码自然累积。ADK 的 ArtifactService 需要额外配置存储后端，而这里直接用本地文件系统即可。

3

没有框架绑定

Agent Swarm 不依赖 LangChain/LangGraph/ADK 等框架，自定义的 orchestrator 和 dispatcher 更贴近项目需求。引入 ADK 全套概念会带来不必要的复杂度。

4

messages 直接复用更高效

当前直接把 messages 存成 OpenAI 格式，恢复时无需转换即可喂给 LLM。ADK 需要从 events 重建 messages，多了一个转换层。对于当前以 OpenAI SDK 为主的调用路径，直接存 messages 是合理优化。

08 推荐演进路径

核心原则：不替换，只叠加

当前的 disk persistence + messages 机制运行良好，不应被替换。建议在其旁边新增一个 Event Log 层，让系统同时有两份记录：

  • messages + shared_context：给 LLM 用的运行时上下文，保持当前格式。
  • events：给人和调试工具用的完整审计日志，借鉴 ADK 设计。

这样既保留了当前的高效路径，又逐步获得 ADK 式的可观测性。

1

Step 1：新增 Event 模型和事件文件

在 src/session/ 下新增 events.py，定义 Event、EventActions。新增 _session/events.jsonl，按行追加事件。

2

Step 2：在编排器关键路径埋点

记录 user_message、tool_call、tool_result、finalize、error。每个事件带 invocation_id（本次 task_id 或单独生成）和 author=orchestrator。

3

Step 3：记录 Agent-Agent 交互

在 Dispatcher 记录 dispatch_request/attempt，在 Worker 侧记录 agent_progress/agent_response。用 parent_id 把 tool_call 和 agent_response 关联起来。

4

Step 4：SessionService 抽象 + 查询 API

把文件操作封装成 FileSessionService，实现 SessionService 协议。新增 /api/sessions/{id}/events 和 /api/sessions/{id}/timeline。

5

Step 5：可选迁移到 Redis

当需要多进程部署时，实现 RedisSessionService。由于有 SessionService 抽象，切换成本很低。

09 总结

GLM 的改动让 Agent Swarm 在 Session 持久化方面已经初步对齐了 ADK 的 Session 概念：会话可跨请求、跨重启存活，工作目录和对话历史都能恢复。这是很大的进步，我之前的调研里"单进程状态"和"session 无持久化"的批评已经部分过时。

但当前方案仍未达到 ADK 的 Event-First 高度。主要差距在于：它只保存了 LLM 消息列表，没有保存结构化事件，因此无法清晰追溯 Agent-Agent 对话、无法审计状态变更、无法按请求周期回放。

最务实的下一步不是推翻现有设计，而是叠加 Event 日志层：保留 messages 给 LLM 用，同时记录 events 给人和调试工具用。这样既发挥了 Agent Swarm 文件系统持久化的简洁优势，又吸收了 ADK 可观测性的核心思想。
