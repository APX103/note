01 SQLite 持久化 Review

实现概览

新增 src/storage/sqlite_store.py，内置两张表：

  • tasks：持久化任务元数据（status、result、artifacts、work_dir、session_id 等）。
  • sessions：持久化会话上下文（messages、shared_context、work_dir）。

main.py 在启动时创建 SQLiteStore，并注入到 TaskManager 和 SessionManager。两者仍然保留内存缓存，写操作时同步落库。

亮点

  • ✓ 零依赖：只用 Python 内置 sqlite3，符合 CLAUDE.md "不引入新依赖" 原则。
  • ✓ WAL 模式：PRAGMA journal_mode=WAL 提升并发写入性能。
  • ✓ Write-through cache：内存缓存快速读，每次变更同步写库。
  • ✓ 可选注入：store=None 时自动降级为文件系统，测试更灵活。
  • ✓ 向后兼容：不破坏现有 API 签名。

可优化

  • ! 未复用连接：每次操作新建连接。当前写入频率不高，没问题；高并发时可以考虑连接池。
  • ! JSON 字段无 schema：messages/artifacts 直接 json.dumps，失去 SQL 查询能力。但当前用例不需要对 message 内容做 SQL 查询，可以接受。
  • ! DB 路径硬编码：settings.storage.shared_output_base + "/swarm.db"。建议进 config。

SessionManager 的降级策略做得好

SessionManager 同时支持 SQLite 和文件系统 JSON：_load_persistent() 优先查 store，miss 再查 _session/context.json。这种"新存储优先 + 老格式兼容"的降级思路非常稳健，避免老 session 数据在升级后丢失。

02 Metrics 实现 Review

实现概览

新增 src/observability/metrics.py，一个轻量级 in-memory Metrics 类：

  • dispatch_total：总调度次数
  • success / failure：成功/失败次数
  • avg_latency_ms：平均延迟
  • failure_rate：失败率

Dispatcher 在 dispatch() 入口统一记录耗时和结果，并通过 GET /api/v1/metrics 暴露 snapshot。

亮点

  • ✓ 线程安全：用了 threading.Lock，适合 uvicorn 多线程。
  • ✓ 聚合位置正确：在 Dispatcher 入口记录，覆盖所有后端（Docker + External Agent）。
  • ✓ API 简洁：一个 JSON endpoint，便于对接监控或本地调试。

可扩展

  • ! 重启归零：in-memory 指标，进程重启后丢失。对指标来说通常可接受。
  • ! 缺少 per-agent 统计：目前只有全局聚合，无法看出哪个 agent_type 失败率高。
  • ! 缺少候选尝试次数：重试/故障转移次数没有记录。

04 仍存在的问题

1

功能缺口 list_tasks 重启后不查 SQLite

TaskManager.list_tasks() 只返回 self._tasks.values()，没有从 SQLite 加载历史任务。这意味着：

  • 进程重启后，调用 GET /api/tasks 返回空列表。
  • 只有先通过 GET /api/tasks/{task_id} 把某个任务 get_task() 进内存后，它才会出现在 list 里。

修复建议：list_tasks(tenant_id) 应该 merge 内存缓存和 SQLite 查询结果，去重后按 created_at 排序返回。

2

其他关键状态仍 in-memory

虽然 task/session 已持久化，但以下状态仍是进程内存：

  • 幂等索引 _idempotency_index
  • per-tenant 并发信号量 _tenant_semaphores
  • 运行中编排任务映射 _running_orchestrations
  • 死信存储 DeadLetterStore
  • 结果缓存 ResultCache
  • 熔断器状态 _breakers

多 uvicorn worker 部署时，这些状态仍无法共享。但优先级低于 task/session 持久化。

3

Event-First 全量上下文仍未引入

当前 persistence 只保存了 messages 和 shared_context。工具调用细节、Agent-Agent 交互链、状态变更历史、取消事件等仍未结构化记录。如果目标是"复现现场"，还需要引入 ADK 式的 Event 模型。

4

任务/会话并发写入安全

SQLite WAL 能处理并发读，但 TaskManager 和 SessionManager 内部没有针对同一 task/session 的显式锁。极端情况下，两个请求同时更新同一个 session 可能产生竞态。可以引入 version 乐观锁或 asyncio.Lock。

05 ADK 对比更新

持久化层面对齐度提升

ADK 的 SessionService 提供可插拔持久化（内存/数据库/Vertex）。Agent Swarm 现在用 SQLiteStore 实现了类似效果，虽然还没抽象成协议，但已经具备"把 session 和 task 持久化到外部存储"的能力。

差距主要在于：ADK 持久化的是 events，Agent Swarm 持久化的是 messages + state。前者更完整，后者更高效。

当前 Agent Swarm 路线 vs ADK 路线

能力 Agent Swarm 当前 ADK
持久化 SQLite 持久化 messages/state 可插拔 SessionService，持久化 events
多轮对话 ✅ 已支持 ✅ 原生支持
Agent-Agent 追溯 ❌ 无结构化记录 ✅ 通过 author/branch/parent_id
状态变更审计 ❌ 只有最终 state ✅ state_delta 事件链
可观测性 ✅ 基础 dispatch metrics ✅ 更丰富的事件体系

06 下一步建议

P0

修复 list_tasks 的 SQLite 加载

这是当前最明显的功能缺口。重启后 GET /api/tasks 应该能列出历史任务。

P1

SessionService / TaskService 抽象

把 SQLiteStore 的职责拆成协议（Protocol），让 File/InMemory/SQLite/Redis 实现可以互换。这也方便未来迁移到 Redis。

P1

引入 Event 日志层

保留 messages 给 LLM，同时追加 Event 日志到 SQLite（或 JSONL）。优先记录 tool_call/tool_result、dispatch、agent_progress、error、cancel。

P1

扩展 Metrics

增加 per-agent 统计、候选尝试次数、重试次数。可以让 Metrics 按 (agent_type) 维度分桶。

P2

其他状态 Redis 化

幂等索引、结果缓存、熔断器、死信存储逐步迁移到 Redis。这一步在真正需要多 worker 部署前完成即可。

P2

并发写入安全

为同一 session/task 的更新加乐观锁（version 字段）或 asyncio 锁，避免竞态覆盖。

07 总结

这次更新是 Agent Swarm 从"原型"走向"可用系统"的重要一步。SQLite 持久化和 Metrics 的实现质量都很高，且严格遵循了项目的工程纪律（零新依赖、保持 API 稳定、TDD）。

但也引入了一个需要尽快修的小 bug：list_tasks() 没有从 SQLite 恢复历史任务列表。修复后，任务/会话的持久化闭环才算完整。

在 ADK 视角下，Agent Swarm 已经在"持久化"这个维度上追了上来，但在"Event-First 全量上下文"上仍有差距。考虑到项目的工程目标和复杂度，当前路线是合理的：先解决最痛的生存问题，再逐步完善可观测性。
