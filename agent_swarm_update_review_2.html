01 Cancel 传播 Review

实现链路

Cancel 传播涉及三个层次的配合：

  1. API 层：WebSocket 收到 action=cancel 后，调用 cancel_running(task_id)，取消编排器侧的 asyncio 任务。
  2. Dispatcher 层：DockerBackend.invoke() 捕获 asyncio.CancelledError，然后调用 A2AClient.cancel_task(worker_task_id)。
  3. Worker 层：Worker 的 handle_cancel_task 取消后台 _run() 任务，从而停止 LLM 调用。
用户 WS cancel
    → cancel_running(task_id)  # src/api/routes.py
    → orchestrator task cancelled
    → DockerBackend.invoke catches CancelledError
    → A2AClient.cancel_task(worker_task_id)
    → POST / (tasks/cancel) to worker
    → handle_cancel_task cancels _bg_tasks[task_id]
    → _run() raises CancelledError, llm_call stops

亮点

  • ✓ 端到端闭环：从用户 WS 一直传到 Worker 内部的后台任务，真正避免继续烧 token。
  • ✓ 遵循 A2A 语义：Worker 新增 tasks/cancel 端点，符合 A2A 协议扩展。
  • ✓ 异常处理稳健：cancel 传播失败只打日志，不影响主取消流程继续抛出 CancelledError。
  • ✓ poll_task 识别 canceled：A2APoll 的 terminal 状态包含 canceled，编排器能正确感知 Worker 已取消。

可优化 / 风险

  • ! 缺少测试：没有针对 Worker cancel 或 cancel 传播的集成测试。
  • ! External Agent 未传播：只有 DockerBackend 处理 CancelledError，ExternalAgentBackend 没有。
  • ! 同步工具不可中断：如果取消时 Worker 正在执行同步的 execute_tool，会等工具返回后才退出。
  • ! 取消后任务状态未持久化：Worker 内存中的 task 状态不会自动变成 canceled；SQLite 中的 task 状态也不会更新。

代码片段点评

src/dispatcher/backends.py 中的处理很到位： 

except asyncio.CancelledError:
    logger.info("Dispatch cancelled; propagating cancel to worker %s", target.agent_type)
    try:
        if a2a_task is not None and a2a_task.task_id:
            await client.cancel_task(a2a_task.task_id)
    except Exception:
        logger.warning("Failed to propagate cancel to worker", exc_info=True)
    raise

这里的关键是：只有非阻塞 dispatch（streaming 模式）才会创建后台 task，也才需要 cancel。阻塞模式下 a2a_task 是同步完成的，Cancel 发生时可能已经完成了。这个边界处理得合理。

02 DispatcherConfig 外置 Review

实现概览

src/config.py 的 Settings 新增 dispatcher: dict，config/default.yaml.example 新增 dispatcher: 段： 

dispatcher:
  max_retries: 2
  dispatch_timeout: 300
  max_concurrent: 8
  health_precheck: true

src/main.py 启动时从 settings.dispatcher 读取并注入 DispatcherConfig。

亮点

  • ✓ 彻底解决了 DispatcherConfig 硬编码问题。
  • ✓ default.yaml.example 注释清晰，新用户知道每个参数含义。
  • ✓ 默认值保留，不配置也能正常运行。
  • ✓ 为后续其他配置外置提供了统一模式。

可优化

  • ! Settings.dispatcher 用的是裸 dict，建议改成 DispatcherConfig dataclass，类型更安全。
  • ! 缺少配置校验：如果 YAML 里写错类型（如 max_retries: "two"），启动时会抛不太友好的异常。

05 仍然存在的缺口

1

测试缺口 Cancel 传播缺少集成测试

目前 cancel 传播只通过代码路径走通，没有自动化测试覆盖。建议加一个 mock Worker 的测试：验证当 orchestrator task 被取消时，A2AClient.cancel_task 被调用。

2

External Agent 未传播 Cancel

ExternalAgentBackend.invoke() 没有捕获 CancelledError，也没有调用外部 Agent 的 cancel。如果外部 Agent 是长耗时调用，取消后它仍会继续执行。

3

取消后 Task 状态未持久化

用户取消后，task_manager 的状态在内存中变为 CANCELLED，但没有调用 save_task 落库。进程重启后，这个任务在 SQLite 中仍可能是 RUNNING。

4

Redis 持久化仍是最大生产化瓶颈

幂等索引、per-tenant 信号量、结果缓存、熔断器、死信存储仍是 in-memory。多 worker 部署前必须完成 P1.2。

5

Event-First 上下文仍未引入

这是之前 ADK 对比里反复强调的。当前持久化只保存 messages/state，工具调用链、Agent-Agent 对话、状态变更审计仍未结构化。

06 下一步建议

P1

补 cancel propagation 集成测试

用 respx mock Worker 的 tasks/cancel 端点，验证取消编排器任务时会发起 cancel 请求。

P1

取消后持久化 task 状态

在 cancel_running 或编排器异常处理中，调用 task_manager.fail_task/update_status + save_task，让 SQLite 中的状态也变为 CANCELLED。

P2

External Agent 传播 Cancel

在 ExternalAgentBackend.invoke() 中捕获 CancelledError，并调用 adapter 的 cancel（如果协议支持）。

P2

Settings.dispatcher 类型化

把 dispatcher: dict 改成 DispatcherConfig dataclass，提升类型安全和 IDE 提示。

P2

Redis 化核心状态

按 NEXT-STEPS P1.2 推进：幂等索引 > 死信 > 结果缓存 > 熔断。这是真正多 worker 部署的前提。

P2

引入 Event 日志层

记录 tool_call/tool_result/dispatch/cancel 等事件，实现"复现现场"。
