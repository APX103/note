Cancel 闭环修复

针对 4c4161a 的修复快速 review。这次改动正好补齐了上一篇 Review 2 里标出的三个主要缺口。

一句话结论

上一篇 review 提到的"缺少 cancel 测试""External Agent 未处理 CancelledError""取消后状态未持久化"三个问题，这次全部修复。测试数从 295 增加到 297 passed。Cancel 功能现在基本闭环。

改动点评

✅ 修复

1. 取消状态持久化

在 TaskManager.update_status() 里加了 SQLite 写入。任何状态变更（created → running → completed / failed / cancelled）都会同步落库。 

# 持久化状态变更到 SQLite（覆盖 cancelled / running / 任意状态）
if self._store and task:
    self._store.save_task(...)

这样用户取消任务后，即使进程重启，任务状态仍然是 cancelled，而不是残留的 running。

✅ 修复

2. Cancel 传播测试

新增 tests/test_cancel_propagation.py，用 mock 验证：

  • DockerBackend 收到 CancelledError 后会调用 A2AClient.cancel_task()
  • 容器在 finally 中被归还
  • ExternalAgentBackend 也会捕获并重新抛出 CancelledError

测试覆盖了关键路径，但还没覆盖真实 Worker 端的 handle_cancel_task。

部分修复

3. External Agent Cancel 处理

ExternalAgentBackend.invoke() 现在捕获 CancelledError，打日志说明"sync HTTP — cannot mid-call cancel"，然后重新抛出。

这个处理是诚实且正确的：外部 Agent 大多是同步 HTTP 调用，确实无法在调用中途取消。能做到的就是让编排器侧任务正常退出，不阻塞后续清理。

小观察

SQLite 写入频率增加

update_status 每次状态变更都写 SQLite。之前只有 complete_task 写一次。现在 running 也会写。对当前流量来说完全没问题，但高并发时可以考虑批量/节流。

Worker 端 cancel 仍缺测试

test_cancel_propagation.py 验证了 Dispatcher 会发 cancel，但没有验证 Worker 收到 tasks/cancel 后是否真的取消后台任务。可以用 FastAPI TestClient 测 worker app。
