一、整体架构概览

Claude Code 的 Agent Team 系统是一个扁平化的 Leader-Worker 协作模型：

角色 职责
Leader（team-lead） 用户直接交互的主 agent，协调团队，审批权限，分配任务
Teammate（Worker） 被 spawn 出来的子 agent，执行具体任务，通过 Mailbox 与 Leader/其他 teammate 通信

核心设计原则：

  • 扁平 roster：Teammate 不能 spawn 其他 teammate（防止嵌套混乱）
  • 文件驱动通信：所有跨 agent 通信通过文件系统 Mailbox 实现
  • 统一身份：通过 CLI args（--agent-id, --team-name 等）传递身份
  • 多后端支持：tmux pane / iTerm2 pane / in-process 三种执行模式

二、核心文件与模块

文件/目录 职责
tools/TeamCreateTool/ 创建 Team 的工具入口
tools/AgentTool/AgentTool.tsx 当 team_name + name 同时存在时 spawn teammate
tools/shared/spawnMultiAgent.ts teammate 创建的统一入口，处理三种后端
tools/SendMessageTool/ 团队通信工具（单播/广播/结构化消息）

utils/swarm/permissionSync.ts 跨 agent 权限同步系统
utils/tasks.ts 团队共享任务列表

三、Team 创建流程

1. 创建 Team：TeamCreateTool

TeamCreateTool.call({ team_name, description?, agent_type? })

执行逻辑：

  1. 检查 Leader 是否已在管理其他 team（一个 Leader 只能管理一个 team）
  2. 生成唯一 team name（如果冲突则自动生成）
  3. 创建 ~/.claude/teams/{teamName}/config.json
  4. 初始化 task list 目录（~/.claude/tasks/{teamName}/）
  5. 在 AppState.teamContext 中注册 team 信息

2. Spawn Teammate：AgentTool.call()

当 Agent 工具调用传入 team_name + name 时，不走普通 subagent 路径，而是进入 teammate 创建：

// AgentTool.tsx:284
if (teamName && name) {
  const result = await spawnTeammate({ name, prompt, team_name, ... }, toolUseContext)
}

spawnTeammate 位于 tools/shared/spawnMultiAgent.ts，支持三种执行模式。

四、三种执行后端

Backend 检测优先级

如果在 tmux 中 → 使用 tmux（即使也在 iTerm2 中）
如果在 iTerm2 中且 it2 CLI 可用 → 使用 iTerm2
如果在 iTerm2 中但 it2 不可用，且 tmux 可用 → 使用 tmux（提示用户安装 it2）
如果 tmux 可用 → 使用 tmux（外部 session）
以上都不可用 → 回退到 in-process

1. Tmux Backend

  • 在 tmux 内：分割当前窗口，Leader 在左（30%），teammates 在右（70%）
  • 在 tmux 外：创建 claude-swarm session，所有 teammates 平铺显示
  • 每个 pane 有独立的颜色边框和标题
  • 通过 tmux send-keys 发送启动命令

2. iTerm2 Backend

  • 使用 iTerm2 的 it2 CLI 创建 native split pane
  • 比 tmux 更原生，支持更好的视觉效果
  • 如果 it2 未安装，会提示用户安装

3. In-Process Backend

  • 在同一个 Node.js 进程中运行
  • 使用 AsyncLocalStorage 隔离上下文
  • 共享 API client 和 MCP 连接
  • 通信同样通过文件 mailbox（与 pane-based 保持一致）
  • 通过 AbortController 终止（而非 kill pane）

五、Teammate 启动流程

// spawnMultiAgent.ts:handleSpawnSplitPane
async function handleSpawnSplitPane(input, context) {
  // 1. 解析 model（'inherit' → leader 的 model）
  const model = resolveTeammateModel(input.model, getAppState().mainLoopModel)
  
  // 2. 生成唯一名字和 agentId（name@teamName）
  const uniqueName = await generateUniqueTeammateName(name, teamName)
  const teammateId = formatAgentId(sanitizedName, teamName)
  
  // 3. 检测后端
  const detectionResult = await detectAndGetBackend()
  
  // 4. 分配颜色
  const teammateColor = assignTeammateColor(teammateId)
  
  // 5. 创建 pane
  const { paneId, isFirstTeammate } = await createTeammatePaneInSwarmView(name, color)
  
  // 6. 构建启动命令
  const spawnCommand = `cd ${cwd} && ${binaryPath} \
    --agent-id ${teammateId} \
    --agent-name ${name} \
    --team-name ${teamName} \
    --agent-color ${color} \
    --parent-session-id ${leaderSessionId} \
    ${plan_mode_required ? '--plan-mode-required' : ''} \
    ${model ? `--model ${model}` : ''}`
  
  // 7. 发送命令到 pane
  await sendCommandToPane(paneId, spawnCommand)
  
  // 8. 注册到 AppState 和 team file
  setAppState(prev => ({ teamContext: { teammates: { [teammateId]: {...} } } }))
  teamFile.members.push({ agentId: teammateId, name, ... })
  
  // 9. 通过 mailbox 发送初始 prompt
  await writeToMailbox(sanitizedName, { from: 'team-lead', text: prompt }, teamName)
}

关键点：teammate 启动时不直接带 prompt，而是通过 mailbox 发送初始指令。这样 teammate 的 inbox poller 可以统一处理所有消息来源。

六、通信系统：Mailbox

文件结构

~/.claude/teams/{teamName}/
├── config.json              # Team 元数据和成员列表
├── inboxes/
│   ├── team-lead.json       # Leader 的 inbox
│   ├── researcher.json      # Researcher 的 inbox
│   └── tester.json          # Tester 的 inbox
└── permissions/
    ├── pending/             # 待审批的权限请求
    └── resolved/            # 已解决的权限请求

Mailbox API

// 写消息
await writeToMailbox(recipientName, { from, text, timestamp, color, summary }, teamName)

// 读所有消息
const messages = await readMailbox(agentName, teamName)

// 读未读消息
const unread = await readUnreadMessages(agentName, teamName)

// 标记已读
await markMessageAsReadByIndex(agentName, teamName, messageIndex)

并发控制：使用 proper-lockfile 文件锁防止多 agent 同时写入同一个 inbox。

消息类型

类型 说明
普通文本 团队成员之间的业务通信
shutdown_request Leader 请求 teammate 退出
shutdown_response Teammate 同意/拒绝退出
permission_request Worker 请求 Leader 审批工具权限
permission_response Leader 审批结果
plan_approval_request/response 计划审批请求与响应
idle_notification Worker 完成任务进入空闲状态
sandbox_permission_request/response 沙箱网络访问权限

七、Inbox 轮询与消息处理

Teammate 的轮询循环

async function waitForNextPromptOrShutdown(identity, abortController, ...) {
  while (!abortController.signal.aborted) {
    // 1. 检查内存中的 pending messages
    // 2. 检查 mailbox 中的未读消息
    // 3. 检查 task list 中的未认领任务
    await sleep(500)
  }
}

消息优先级：

  1. Shutdown request（最高优先级）
  2. Team-lead 的消息
  3. Peer-to-peer 消息（FIFO）

八、权限同步系统

In-Process Teammate

  • 通过 leaderPermissionBridge.ts 直接使用 Leader 的 ToolUseConfirm dialog
  • 在 Leader UI 中显示带 worker badge 的权限请求

Pane-Based Teammate（tmux/iTerm2）

  1. Worker 将权限请求写入 permissions/pending/{requestId}.json
  2. Worker 发送 permission_request 到 Leader 的 mailbox
  3. Leader 的 inbox poller 检测到请求
  4. Leader UI 显示权限请求（带 worker name 和 color badge）
  5. 用户审批后，Leader 写入 permissions/resolved/{requestId}.json
  6. Worker 轮询 permissions/resolved/ 获取结果

Mailbox-Based Permission（新一代）

// Worker 发送权限请求
await sendPermissionRequestViaMailbox(request)

// Leader 审批后发送响应
await sendPermissionResponseViaMailbox(workerName, resolution, requestId)

九、任务系统

Team 共享一个任务列表，存储在 ~/.claude/tasks/{teamName}/：

type Task = {
  id: string
  subject: string
  description: string
  owner?: string       // 负责执行的 agent name
  status: 'pending' | 'in_progress' | 'completed'
  blocks: string[]     // 阻塞哪些任务
  blockedBy: string[]  // 被哪些任务阻塞
}

Teammate 可以通过工具协作：

  • TaskCreate：创建任务
  • TaskList：查看所有任务
  • TaskGet：获取任务详情
  • TaskUpdate：更新任务状态/负责人

任务认领逻辑：teammate 优先检查 inbox，然后检查 task list 中是否有未认领的可用任务（状态为 pending、无负责人、所有依赖已解决）。

十一、生命周期管理

正常流程

  1. Teammate 完成任务后发送 idle_notification 给 Leader
  2. Leader 可以发送 shutdown_request 请求退出
  3. Teammate 通过 shutdown_response 同意或拒绝
  4. 同意退出的 in-process teammate：abort 其 controller
  5. 同意退出的 pane-based teammate：调用 gracefulShutdown(0)

异常清理

async function cleanupSessionTeams() {
  // 1. 杀死所有 pane-backed teammate 的 pane
  await Promise.allSettled(teams.map(name => killOrphanedTeammatePanes(name)))
  // 2. 删除 team 目录和 task 目录
  await Promise.allSettled(teams.map(name => cleanupTeamDirectories(name)))
}

在 SIGINT/SIGTERM 时自动调用，防止 orphan process。

十二、关键设计亮点

1. 统一抽象：TeammateExecutor

type TeammateExecutor = {
  readonly type: BackendType
  isAvailable(): Promise
  spawn(config: TeammateSpawnConfig): Promise
  sendMessage(agentId: string, message: TeammateMessage): Promise
  terminate(agentId: string, reason?: string): Promise
  kill(agentId: string): Promise
  isActive(agentId: string): Promise
}

三种后端（tmux / iTerm2 / in-process）都实现这个接口，上层代码无需关心底层差异。

2. 身份解析的多层优先级

function getAgentName() {
  // 1. AsyncLocalStorage（in-process）
  const inProcessCtx = getTeammateContext()
  if (inProcessCtx) return inProcessCtx.agentName
  
  // 2. dynamicTeamContext（tmux 通过 CLI args）
  return dynamicTeamContext?.agentName
}

3. Pane 管理细节

  • 锁机制：acquirePaneCreationLock() 防止并行 spawn 时 pane 竞争
  • Shell 初始化延迟：PANE_SHELL_INIT_DELAY_MS = 200ms，等待 shell rc 加载
  • 布局自动平衡：新增 teammate 时自动 select-layout tiled 或 main-vertical
  • 隐藏/显示 pane：通过 break-pane 和 join-pane 实现

4. In-Process 的内存优化

export const TEAMMATE_MESSAGES_UI_CAP = 50

in-process teammate 的 task.messages（用于 zoomed transcript dialog）只保留最近 50 条，防止 whale session 中大量 agent 同时运行时内存爆炸。

总结：Claude Code 的 Agent Team 系统通过文件邮箱 + 多后端执行 + 统一抽象实现了真正的多 Agent 协作，而不是简单的 subagent 串行调用。三种后端（tmux / iTerm2 / in-process）共享同一套 Mailbox 通信协议和权限同步机制，使得系统既能在终端中可视化展示多 Agent 协作，也能在资源受限时回退到进程内执行。
