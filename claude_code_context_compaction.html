Claude Code 上下文压缩实现深度解析

源码分析 📅 2026-05-15 🔖 TypeScript / Anthropic

Claude Code 的上下文压缩系统分为 5 个层级，从轻到重依次为：Cached Microcompact → Time-based Microcompact → Session Memory Compact → Legacy Compact → Reactive Compact。本文基于源码分析其判定条件与实现原理。

01 触发判定条件

1.1 自动压缩（AutoCompact）阈值

自动压缩的阈值计算采用了"双重缓冲"策略，确保在触及模型硬上限前预留足够空间。

// services/compact/autoCompact.ts

// 有效窗口 = 模型上下文窗口 - 预留摘要输出空间（20K tokens）
function getEffectiveContextWindowSize(model: string): number {
  const reservedTokensForSummary = Math.min(
    getMaxOutputTokensForModel(model),
    MAX_OUTPUT_TOKENS_FOR_SUMMARY,  // 20,000
  )
  return getContextWindowForModel(model) - reservedTokensForSummary
}

// 自动压缩阈值 = 有效窗口 - 13K tokens 缓冲
function getAutoCompactThreshold(model: string): number {
  return getEffectiveContextWindowSize(model) - AUTOCOMPACT_BUFFER_TOKENS  // 13,000
}

对于 200K 窗口的模型，实际阈值约为 180K - 13K = 167K tokens。可通过 CLAUDE_CODE_AUTO_COMPACT_WINDOW 环境变量覆盖窗口大小，或通过 CLAUDE_AUTOCOMPACT_PCT_OVERRIDE 设置百分比阈值。

排除场景（防止死锁/竞态）：
• querySource === 'session_memory' 或 'compact' — forked agent 避免递归死锁
• querySource === 'marble_origami' — ctx-agent 避免破坏主线程状态
• DISABLE_AUTO_COMPACT / DISABLE_COMPACT 环境变量为真
• 用户在设置中关闭了 auto-compact
• Context Collapse 模式开启时（避免与 collapse 竞争）
熔断机制：连续失败 3 次后停止尝试（MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3）。此前曾有 1,279 个会话出现 50+ 连续失败，每天浪费约 25 万次 API 调用。

1.2 微压缩（Microcompact）

类型触发条件范围
Time-based距离上条 assistant 消息超过 60 分钟仅主线程
Cached工具结果数量超过 GrowthBook 配置阈值仅主线程 + 支持 cache_edits 的模型

1.3 手动/响应式压缩

  • 手动：用户输入 /compact 命令时触发，支持自定义压缩指令
  • 响应式（Reactive）：当 API 返回 prompt_too_long（413）错误时触发，从消息尾部按 API round 分组逐层剥离
02 核心实现原理

2.1 传统压缩（Legacy Compact）

核心文件：services/compact/compact.ts

1
预处理 — 移除图片/文档块替换为标记，过滤 skill_discovery/skill_listing 附件
2
摘要生成 — forked agent 优先复用 prompt cache，失败回退 streaming
3
附件重建 — 恢复最近文件、skills、plan mode、MCP 指令等关键上下文
4
消息重组 — boundary marker → summary → kept messages → attachments → hooks
5
PTL 容错 — 压缩请求自身触发 PTL 时，从头部丢弃旧 API round 组

摘要 Prompt 模板要求模型生成包含 9 个部分的结构化摘要：

  1. Primary Request and Intent — 用户显式请求与意图
  2. Key Technical Concepts — 关键技术概念与框架
  3. Files and Code Sections — 文件与代码段（含完整代码片段）
  4. Errors and fixes — 遇到的错误及修复方式
  5. Problem Solving — 问题解决过程
  6. All user messages — 所有非工具结果的用户消息
  7. Pending Tasks — 待处理任务
  8. Current Work — 当前正在进行的工作
  9. Optional Next Step — 下一步建议

值得注意的是，Prompt 前插入了极其强硬的 NO_TOOLS_PREAMBLE，明确禁止模型在压缩过程中调用任何工具——否则会被拒绝且浪费唯一的 turn。

2.3 Session Memory 压缩（实验性）

核心文件：services/compact/sessionMemoryCompact.ts

利用已持久化的 session memory 文件作为摘要，无需额外 API 调用。保留策略采用三约束条件：

  • minTokens: 10_000 — 至少保留 10K tokens
  • minTextBlockMessages: 5 — 至少 5 条含文本的消息
  • maxTokens: 40_000 — 硬上限

从 lastSummarizedMessageId 开始向后扩展，同时确保不拆散 tool_use/tool_result 对。

2.4 微压缩（Microcompact）

Time-based 微压缩：直接修改本地 message.content，将旧工具结果清空为 "[Old tool result content cleared]"，保留最近 5 个可压缩工具结果。这是最轻量的压缩方式。
Cached Microcompact：使用 Anthropic 的 cache_edits API 删除服务器端缓存中的工具结果，不修改本地消息内容 — 仅添加 cache_reference 和 cache_edits 块。仅支持特定模型和主线程查询。

2.5 PTL（Prompt Too Long）容错

如果压缩请求本身触发 PTL，系统从最旧的 API round 组开始丢弃（每次最多丢弃 20%），最多重试 3 次。这是"丢车保帅"策略——丢失旧上下文总比完全卡住好。

const MAX_PTL_RETRIES = 3

// 从头部丢弃最旧的 API round 组

const dropCount = Math.max(1, Math.floor(groups.length * 0.2))
const truncated = groups.slice(dropCount).flat()

04 关键设计决策
设计说明
Cache Sharing 优先压缩优先使用 forked agent 复用主对话的 prompt cache，关闭后 cache miss 率高达 98%
Session Memory 优先实验性路径优先尝试已提取的 session memory，避免额外 API 摘要调用
PTL 渐进降级压缩请求自身也可能 PTL，通过从头部丢弃旧消息来渐进降级
熔断机制连续 3 次失败后熔断，避免无意义地 hammer API
Post-Compact 附件重建压缩后重新注入最近文件、skills、plan mode 等关键上下文，避免信息丢失
子线程安全微压缩和自动压缩严格限制在主线程，避免子 agent 污染全局状态
