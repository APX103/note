一、整体架构概览

Claude Code 的 Subagent 系统核心围绕 AgentTool 展开，它是一个标准的 Tool（类似 BashTool、FileReadTool），定义在 tools/AgentTool/AgentTool.tsx 中。Subagent 的调用本质上就是主 Agent 在工具调用阶段选择调用 Agent 这个工具。

文件 职责
tools/AgentTool/AgentTool.tsx Subagent 调用的主入口，call() 方法处理所有 Agent 工具调用
tools/AgentTool/runAgent.ts 运行单个 Agent 的查询循环，封装 query()
tools/AgentTool/forkSubagent.ts Fork 子 Agent 的实验特性，处理上下文继承
utils/forkedAgent.ts Forked Agent 的通用工具函数，createSubagentContext
tools/AgentTool/prompt.ts Agent 工具的系统提示（描述何时使用 Agent）
tools/AgentTool/loadAgentsDir.ts Agent 定义加载（内置 + 自定义 + 插件）
tools/AgentTool/agentToolUtils.ts 工具过滤、结果最终化、异步生命周期管理

二、调用详细流程

1. 工具调用入口：AgentTool.call()

当 LLM 决定调用 Agent 工具时，进入 AgentTool.call() 方法。接收参数包括：

{
  description: string,      // 3-5 字任务描述
  prompt: string,           // 给 subagent 的完整任务指令
  subagent_type?: string,   // Agent 类型（Explore/Plan/general-purpose 等）
  model?: 'sonnet'|'opus'|'haiku',
  run_in_background?: boolean,
  name?: string,            // 用于 teammate 命名
  team_name?: string,       // 多 Agent 团队名称
  isolation?: 'worktree'|'remote',
  cwd?: string
}

2. 路由判定：两条互斥路径

路径 A：Multi-Agent / Teammate 路径

如果 team_name && name 同时存在，则调用 spawnTeammate() 创建 teammate（支持 in-process 或 tmux 模式）。这是多 Agent 协作的入口。

路径 B：Subagent / Fork 路径

常规 subagent 调用，核心逻辑：

const effectiveType = subagent_type ?? (isForkSubagentEnabled() ? undefined : GENERAL_PURPOSE_AGENT.agentType);
const isForkPath = effectiveType === undefined;

关键判定：

  • subagent_type 明确指定：走普通 subagent 路径，查找对应的 AgentDefinition
  • subagent_type 省略 + Fork 实验开启：走 Fork 路径（隐式 fork）
  • subagent_type 省略 + Fork 实验关闭：默认使用 general-purpose agent

3. Agent 定义解析与过滤

从 toolUseContext.options.agentDefinitions.activeAgents 中查找匹配的 AgentDefinition。支持内置 Agent、自定义 Agent（用户通过 .claude/agents/*.md 定义）、插件 Agent。

4. 系统提示与消息构建

普通路径（isForkPath = false）

  • 调用 selectedAgent.getSystemPrompt() 获取该 agent 的专属系统提示
  • 使用 enhanceSystemPromptWithEnvDetails() 增强环境信息
  • promptMessages = [createUserMessage({ content: prompt })]

Fork 路径（isForkPath = true）

  • 子 agent 继承父 agent 的系统提示（为保证 prompt cache 命中，使用父 agent 的 renderedSystemPrompt）
  • 调用 buildForkedMessages(prompt, assistantMessage) 构建消息：
function buildForkedMessages(directive, assistantMessage) {
  // 1. 保留父 assistant message 的完整内容（所有 tool_use、thinking、text）
  // 2. 为每个 tool_use 构建占位符 tool_result
  // 3. 附加一个用户消息，包含所有占位符结果 + fork 指令
}

目的：所有 fork 子 agent 产生字节级相同的 API 请求前缀，从而共享父 agent 的 prompt cache。

5. 工具池组装

const workerTools = assembleToolPool(workerPermissionContext, appState.mcp.tools);

assembleToolPool 将内置工具与 MCP 工具合并、去重、排序。

6. 执行：runAgent()

调用 runAgent()，关键步骤：

  1. 创建隔离上下文：createSubagentContext(parentContext, overrides)
    • 克隆 readFileState（文件读取状态缓存）
    • 新的 abortController（子 agent 可独立取消）
    • 默认隔离 setAppState、setToolJSX 等 UI 回调
  2. 处理消息上下文：Fork 路径传入父 agent 的完整对话历史
  3. 初始化 Agent MCP 服务器：agent 可在 frontmatter 中定义专属 MCP 服务器
  4. 注册 Hooks：执行 SubagentStart hooks，收集额外上下文
  5. 进入查询循环：调用 query() 进入主循环
  6. 清理：finally 块中释放 MCP 连接、清除 session hooks、释放文件缓存、杀死后台 shell 任务

三、调用时机判定

调用时机完全由 LLM 根据系统提示中的指导自行判定。系统提示通过 tools/AgentTool/prompt.ts 中的 getPrompt() 注入。

1. 总体原则

Launch a new agent to handle complex, multi-step tasks autonomously.

Agent 工具的定位：启动专门化的子进程（subprocess）自主处理复杂任务。

2. Fork 路径的时机判定

当 FORK_SUBAGENT 特性开启时，系统提示中增加专门的 "When to fork" 章节：

Fork yourself (omit subagent_type) when the intermediate tool output isn't worth keeping in your context.

The criterion is qualitative — "will I need this output again" — not task size.

场景 判定
Research（研究） Fork 开放式问题。如果研究可拆分为独立问题，在一个消息中并行启动多个 fork
Implementation（实现） 优先 fork 需要多次编辑的实现工作。先做研究再实现
读取特定文件 ❌ 不要 fork，直接用 Read 工具
搜索特定类定义 ❌ 不要 fork，直接用 Glob/Grep
在 2-3 个文件中搜索 ❌ 不要 fork，直接用 Read

核心判定标准：中间工具输出是否值得保留在主 agent 的上下文中。Fork 的本质是将"噪音"隔离到子 agent，保护主 agent 的 context window。

3. 普通 Subagent 的时机判定

每个内置 Agent 的 whenToUse 字段被注入系统提示，指导 LLM 在何时选择哪种 Agent 类型。

四、Fork 路径 vs 普通路径

维度 Fork 路径 普通路径（subagent_type）
触发条件 省略 subagent_type + Fork 实验开启 显式指定 subagent_type
系统提示 继承父 agent 的系统提示（cache-identical） 使用 agent 自身的系统提示
上下文 继承父 agent 的完整对话历史 从零开始，只有 prompt
Prompt Cache 与父 agent 共享 cache 独立 cache
工具池 useExactTools: true，继承父工具池 按 agent 定义重新过滤
消息构建 buildForkedMessages()：占位符 tool_result + 指令 createUserMessage({ content: prompt })
递归限制 Fork 子 agent 内禁止再次 fork 受普通 agent 层级限制

五、同步 vs 异步执行

同步执行（默认）

  • 阻塞主 agent 的回合
  • runAgent() 的 async iterator 被逐条消费
  • 实时显示进度（onProgress 回调）
  • 支持在前台运行一段时间后 后台化（backgroundPromise race）

异步执行

当满足以下条件时强制异步：

const shouldRunAsync = (
  run_in_background === true ||
  selectedAgent.background === true ||
  isCoordinator ||
  forceAsync ||           // isForkSubagentEnabled() → true
  assistantForceAsync ||  // Kairos 模式
  proactiveModule?.isProactiveActive()
) && !isBackgroundTasksDisabled;

异步路径：

  1. registerAsyncAgent() 注册后台任务到 AppState
  2. runAsyncAgentLifecycle() 驱动整个生命周期
  3. 立即返回 async_launched 状态给父 agent
  4. 完成后通过 enqueueAgentNotification() 发送通知

六、工具过滤与权限控制

Subagent 的工具列表会被 filterToolsForAgent() 过滤：

过滤规则 说明
ALL_AGENT_DISALLOWED_TOOLS 所有 agent 禁止使用的工具（如 TaskUpdate、TaskOutput 等）
CUSTOM_AGENT_DISALLOWED_TOOLS 非内置 agent 额外禁止的工具
ASYNC_AGENT_ALLOWED_TOOLS 异步 agent 只允许使用的白名单工具（更严格）
Agent 自身 disallowedTools 如 Explore/Plan 禁用写操作工具

七、内置 Agent 类型

Agent 可用工具 特点
general-purpose ['*']（全部） 通用，可读写
Explore 禁用 Edit、Write、NotebookEdit、Agent 只读模式，专用于搜索
Plan 禁用 Edit、Write、NotebookEdit、Agent 只读模式，专用于架构规划
FORK_AGENT ['*'] + useExactTools: true 继承父工具池，复用 prompt cache

八、关键设计亮点

1. Prompt Cache 共享（Fork 路径）

Fork 子 agent 通过以下手段确保与父 agent 的 API 请求前缀字节级相同：

  • 使用父 agent 的 renderedSystemPrompt
  • 使用父 agent 的 options.tools（useExactTools: true）
  • 继承 thinkingConfig 和 isNonInteractiveSession
  • buildForkedMessages() 中所有占位符 tool_result 文本完全相同

2. 递归 Fork 防护

if (isInForkChild(toolUseContext.messages)) {
  throw new Error('Fork is not available inside a forked worker...');
}

通过检测消息历史中的 <FORK_BOILERPLATE_TAG> 防止 fork 嵌套。

3. 上下文隔离

createSubagentContext() 默认完全隔离可变状态：

  • readFileState：克隆（非共享）
  • abortController：新建（父 agent 取消不影响子 agent）
  • setAppState：默认 no-op
  • toolDecisions、nestedMemoryAttachmentTriggers：全新实例

4. 后台化机制

同步 subagent 在运行超过 PROGRESS_THRESHOLD_MS（2秒）后，UI 显示后台化提示。用户可主动后台化，agent 会通过 Promise.race([agentIterator.next(), backgroundPromise]) 检测，将迭代器优雅关闭后在后台重新启动。
