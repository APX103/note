1. 核心循环架构

所有主流 Agent 框架——Claude Code、OpenAI Agents SDK、LangGraph、smolagents、Vercel AI SDK——最终收敛于同一架构。不是类似，而是完全相同：一个 while 循环，调用 LLM，检查响应是否包含 tool_calls，执行工具，直到模型停止调用工具为止。

💡
Steve Kinney (2026) 的总结： "The loop is a solved problem. The engineering around the loop — context management, safety controls, graceful degradation, cost containment — is where all the interesting decisions live."

1.1 最小可行 Agent（6 行伪代码）

messages = [system_prompt, user_message]

while True:
    response = llm.call(messages, tools=available_tools)
    messages.append(assistant_message(response))

    if not response.has_tool_calls:
        break  # 模型说"我完成了"

    for tool_call in response.tool_calls:
        result = execute(tool_call.name, tool_call.input)
        messages.append(tool_result(result))

1.2 终止信号

Tool Call = "我还需要更多信息或需要执行动作"
纯文本响应 = "我已经有足够信息来回答了"

在 Anthropic API 中，检查 response.stop_reason === 'end_turn' 来判断循环是否应该结束。

1.3 ReAct 模式基础

Yao et al. (Princeton & Google, 2022) 的 ReAct 论文形式化了推理（Reasoning）与行动（Acting）交替进行的理念：模型思考要做什么 → 执行 → 观察结果 → 再思考。相比纯 chain-of-thought，在 ALFWorld 基准上提升了 34%。

2. Agent Loop 框架对比

虽然核心循环相同，但各框架在工程细节上展现了不同哲学：

框架 循环实现 核心特点 适用场景
Claude Code subprocess (NDJSON) 三层权限、自动 compaction、CLAUDE.md 持久化、Haiku 安全检查 编码 Agent、长会话
OpenAI Agents SDK while(true) + runSingleTurn() 4 分支决策、guardrails 三层检查、transfer_to 工具实现 handoff 多 Agent 协作
LangGraph 有向循环图 检查点持久化、并行分支、中断恢复、时间旅行 复杂工作流、需要恢复
smolagents while loop + typed steps code-as-action、~1000 行、规划步骤、优雅降级 轻量 Agent
Vercel AI SDK composable middleware Agent = interface、stop conditions 可组合、prepareStep 钩子 Web 应用
📊
Token 成本参考（Anthropic 内部数据）： 标准对话 = 1x, 单 Agent Loop ≈ 4x, Multi-Agent ≈ 15x。Multi-Agent 在内部评测上优于单 Agent 90.2%，但 token 消耗是真实的。

3. System Prompt 模板

行业实践表明，System Prompt 应该采用分层结构，而非大段文字。以下是经过生产验证的模板：

3.1 Claude Code 风格的五层结构

Template: Claude Code Style System Prompt
<background_information>
  你是谁、你的职责范围、当前任务背景
</background_information>

<instructions>
  核心行为规则（bullet points，每条一条指令）
  - 永远做 X
  - 绝不要 Y
  - 优先 Z
</instructions>

<tool_guidance>
  每个工具的使用场景、参数说明
  - tool_name: 何时使用、参数含义
</tool_guidance>

<output_description>
  期望的输出格式、结构要求
</output_description>

<examples>
  Few-shot 示例（2-4 个，覆盖正常 + 边界情况）
</examples>

3.2 通用 Agent System Prompt 五段式

Template: Universal Agent System Prompt
# Identity and Purpose
You are [role]. Your job is to [specific task].
Be specific about domain: "[exact scope]"

# Behavioral Rules (Priority Stack)
Priority 1 (never violate): safety, legal, privacy
Priority 2 (strong preference): accuracy, factual correctness
Priority 3 (default): tone, formatting, length
Priority 4 (nice to have): personality, engagement

# Response Format
Every response includes:
1. [element 1]
2. [element 2]
3. [element 3]
Schema: {field: type, required: bool}

# Edge Case Handling
- Off-topic: [redirect strategy]
- Uncertain: say "Unknown" + 2 verification steps
- Conflicting instructions: later instructions win, flag trade-offs

# Examples (Few-Shot)
## Normal case
[example interaction]

## Edge case
[example interaction]

3.3 可复用的精简骨架

Template: Minimal Reusable Skeleton
Role: {role}. Audience: {audience}.
Deliverable: {format, length}.
Inputs: {files/notes}.
Output: {JSON schema or section list}.

Steps:
1) Extract facts with citations [Doc:line].
2) Transform into {themes/table/summary}.
3) Premortem + mitigations.

Constraints: plain language, no speculation without 'Unknown'.
Rubric (0-5): Evidence (0-2), Actionability (0-2), Brevity (0-1).
Checklist: [ ] audience named; [ ] numbers checked; [ ] next steps present.
If context missing -> needs: "single question".

3.4 优先级栈（Priority Stack）

当规则冲突时，模型需要知道哪条规则优先：

* Priority 1 (never violate): Safety, legal compliance, data privacy
* Priority 2 (strong preference): Accuracy, factual correctness
* Priority 3 (default): Tone, formatting, response length
* Priority 4 (nice to have): Personality, humor, engagement

对于 thinking 模型（Claude 3.7+, o3），优先级栈应放在 prompt 前部，因为模型在推理阶段就会权衡优先级。

4. 12 个 Prompt 工程技巧

来源：ThinkingLoop (2025)，基于 GitHub Trending #1 的 shanraisshan/claude-code-best-practice 仓库（69 条建议），含 Boris Cherny（Claude Code 构建者）输入

1 设定职位，而非氛围

用 Role + Audience + Deliverable + Constraints 代替 "expert, concise, specific"。

Role: Senior technical writer.
Audience: busy backend engineers.
Deliverable: 400-word brief with 5 bullets + 1 risk.
Constraints: plain language, no hype, no links.
2 明确输入输出契约

定义输入来源和输出格式，要求严格 JSON（如果后续需要解析）。

Input: product spec (markdown), support log (CSV).
Output: JSON with keys {summary, top_issues[3], gaps, followups}.
Only return valid JSON.
3 给 Rubric，不给愿望

告诉 Claude 如何 评分，把"做得更好"变成"这是评判标准"。

Quality rubric (0-5):
- Evidence-backed claims (0-2)
- Actionability (0-2)
- Brevity (0-1)
Add a 'score' and 'improvements' section at the end.
4 用微目标分解任务

不要问 "chain-of-thought"，要结构化产物（quotes, themes）。保持透明和速度。

Do these steps:
1) Extract 7 key facts (quote + source).
2) Cluster into 3 themes.
3) Draft a 150-word summary per theme.
Return {quotes, themes, summaries}.
5 证据模式（Evidence Mode）

每条断言必须引用来源 [Doc:line] 或标记为推测。

Claim policy: Every assertion must cite [Doc:line]
or be labeled speculative.
6 要 Delta，不要 Dump

比较版本时只输出变更，引导人类注意力。

Task: Compare v1 vs v2 release notes.
Output: delta list (added/removed/modified),
plus 'why it matters' in one sentence each.
7 安装 Premortem

让 Claude 在提出方案后自我批判。

Run a premortem: If this plan fails in 30 days,
what are the 3 most likely causes?
Add mitigations next to each cause.
8 用示例约束风格（Few-Shot 正解）

提供一个"黄金段落"来克隆语气和节奏，而非用形容词描述。

Mimic the rhythm of this sample (2 paragraphs).
Keep structure and cadence, not the content.
Sample: [insert your two-paragraph sample]
Task: Rewrite the Summary and Risks sections in the same style.
9 用 Checklist 消灭漂移

在 prompt 末尾要求 Checklist 验证。

Checklist:
- [ ] Audience named
- [ ] Numbers double-checked
- [ ] Actionable next steps included
Return the checklist with ✓/✗ and fixes for any ✗.
10 两遍编辑（Two-Pass Editing）

第一遍 = 结构；第二遍 = 语气。不要混合。

Pass 1: Restructure into sections (no style edits).
Pass 2: Compress to 350-400 words, cut hedging,
keep concrete nouns/verbs.
11 "先问后猜"护栏

给 Claude 一次机会，在关键信息缺失时提出一个简洁的澄清问题。

If any critical input is missing, ask exactly one
clarifying question at top under key: needs.
Otherwise, proceed.
12 教会 Claude 说"我不知道"

规范不确定性：证据不足时说 Unknown，并列出 2 种快速验证方法。

If uncertain beyond reasonable doubt, say 'Unknown'
and list 2 ways to verify (fast).
No invented details.

5. Context Engineering（上下文工程）

🔑
Anthropic 核心理念： "Prompt engineering 是关于如何写指令；Context engineering 是关于在每次推理时，什么信息配置最可能产生期望行为。" 这是 2025-2026 年的范式转变。

5.1 四大策略（Anthropic 框架）

Write 写上下文

将信息保存在上下文窗口之外。Scratchpad、memory files、progress notes。模型生成后续需要的信息并持久化到工具可访问的位置，使其在 compaction 后仍能存活。

Claude Code 实践：TodoWrite 工具、NOTES.md 文件。Manus 团队发现：agent 维护 todo.md 来"recite objectives into the end of context"，对抗 lost-in-the-middle 问题。

Select 选上下文

在需要时通过工具动态拉取相关信息（grep, glob, RAG, 数据库查询）。模型不需要把所有东西都带在窗口里，只需要在需要时找到它们。

Claude Code 实践：glob 和 grep 让 agent 自主导航环境、即时获取文件，绕过过期索引问题。

Compress 压缩上下文

在不丢失关键信息的前提下减少 token 数。Claude Code 在 95% 使用时自动触发 compaction。最安全的轻量级方式是 tool result clearing——将旧的 tool 输出替换为摘要。

Isolate 隔离上下文

使用新上下文窗口的子 Agent 处理子任务，返回压缩摘要。子 Agent 内部可能用 10,000+ tokens，但只返回 1,000-2,000 token 摘要给父 Agent。

5.2 Context Rot 问题

LLM 像人类一样有注意力预算。随着上下文增加，信息召回能力下降。这不是硬悬崖，而是渐降曲线。工程应对：

  • Compaction at 80%：不要等到 100% 才压缩。80% 时触发，给 compaction prompt 本身留出空间。
  • 工具结果清理：已执行的 tool call 和结果在深层历史中不再需要保留原始数据。
  • System reminders：Claude Code 在 tool result 中嵌入固定指令（如"继续使用 TODO 列表"），比只在 system prompt 中放一次有更高的遵守率。

5.3 KV-Cache 优化（Manus 团队数据）

💰
关键数据：工具响应占 67.6% 总 tokens，system prompt 仅 3.4%。"优化 system prompt 的实际收益，远不如优化 tool 响应。" Cached token: $0.30/M vs Uncached: $3/M（10x 差距）。不要在迭代中动态增删工具——这会破坏 prefix cache。

6. 生产级安全控制

循环本身很简单。让循环不失控才是全部工作。需要多层防御，不是一层：

6.1 四层防御体系

1 Max Iterations（最大迭代）

硬上限：生产环境通常 15-25 步。到达上限后，使用 early stopping generate 模式：追加消息 "已达到最大步数，请基于已完成的工作给出最佳答案" 并以 无工具 模式再调用一次 LLM。这比静默停止好得多。

2 Wall-Clock Timeout

单次运行 300 秒上限。防护工具 hang、网络问题、模型生成长响应。

3 Token / Cost Budget

硬上限如 $2.00/run。监控 token 增长模式：线性增长（正常）vs 二次增长（没做 compaction，会快速击穿预算）。

4 Loop Fingerprinting（循环指纹）

Hash 每次迭代的 (tool_name, result_preview) 元组。如果连续 3 次出现相同指纹，说明 Agent 卡住了。有一个生产系统在人工介入前重复了同一答案 58 次。

fingerprint = hash((tool_name, result[:100]))
if fingerprint in recent_fingerprints[-3:]:
    raise StuckInLoop("Agent is repeating itself")

6.2 错误分类与重试

HTTP 状态码 策略 原因
429, 500, 502, 503, 504 重试 临时性错误，指数退避 + jitter
401, 403, 422 立即停止 根本性错误，重试不会解决

6.3 幂等性保证

有副作用的工具（发邮件、建工单、处理付款）必须有 idempotency key。Agent 在 timeout 后重试可能导致重复操作。

7. Multi-Agent 编排模式

四种在生产中实际出现的 Multi-Agent 模式：

模式 描述 优点 缺点 适用场景
Pipeline Agent 按顺序执行，前一个的输出传给下一个 简单、可预测 无并行 Research → Draft → Review
Manager 一个 Orchestrator 委派给专家并综合输出 关注点分离 Manager 成瓶颈 需要专业化的复杂任务
Handoffs Agent 直接转移控制权（OpenAI transfer_to_<agent>） 去中心化、灵活 难以调试 多步骤流程
Fan-out 多个 Agent 并行处理独立子任务，合并结果 最高吞吐 任务需可分解 独立子任务
⚠️
Steve Kinney 的建议："大多数我见过的任务，一个有 good tools 的单 Agent 就能搞定。如果你选择 Multi-Agent 只是因为单 Agent '感觉太简单'，你很可能要用 15x 的 token 换边际改善。"

8. Tool 设计原则

Anthropic 的建议非常明确："Few thoughtful tools targeting specific high-impact workflows"。不是巨型目录，不是每个 API 端点一个工具。

8.1 工具选择测试（Litmus Test）

✅
"If a human engineer can't definitively say which tool should be used in a given situation, an AI agent can't be expected to do better."

8.2 高阶工具优于细粒度 CRUD

# 不好：三个模糊重叠的工具
list_users()    list_events()    create_event()

# 好：一个任务导向的工具
schedule_event(attendees, time, duration, agenda)

8.3 工具命名空间

# 前缀防止冲突，让模型选择更可靠
asana_projects_search
jira_search
github_create_issue

8.4 工具返回错误字符串，而非异常

当工具失败时，模型需要在上下文中看到错误作为文本，才能推理哪里出了问题。异常会打断循环；错误字符串让模型自我修正。

8.5 Poka-yoke（防错设计）

Anthropic 发现：要求绝对文件路径消除了一整类 SWE-bench 上的模型错误。他们花在优化工具设计上的时间，比优化整体 prompt 还多。

8.6 Claude Code 的 14 个工具

Claude Code 只用了 14 个工具就实现了完整功能：

  • 4 个命令行工具（bash, glob, grep, ls）
  • 6 个文件交互工具（read, write, edit, multi_edit, notebook_read, notebook_edit）
  • 2 个 Web 工具（web_search, web_fetch）
  • 2 个控制流工具（todo_write, task）

Jannes Klaas 的观察："这很简单。一个单 Agent 循环，14 个工具。没有 critic 模式，没有角色扮演，没有复杂记忆系统，没有数据库。很多功劳归模型本身，但 Anthropic 也做了出色的设计来最有效利用模型。"

9. Claude Code 实战技巧

从 Claude Code 源码泄露和社区 Rust 重写（claw-code, ~20K 行 vs 原版 510K 行）中提取的关键设计：

9.1 TODO 列表驱动规划

Claude Code 每次会话开始首先调用 TodoWrite，之后持续更新。TODO 列表同时是：

  • Agent 的内部计划
  • System reminder 的输入
  • 用户的 UX 组件（展示进度）

关键设计：每次更新时传入完整列表，而非增量更新单个 item。

9.2 System Reminders 注入

Claude Code 在用户消息中动态注入 system reminders来保持 Agent 在长会话中的方向感：

<system-reminder>
As you answer the user's questions, you can use the
following context:

# important-instruction-reminders
Do what has been asked; nothing more, nothing less.
NEVER create files unless absolutely necessary.
ALWAYS prefer editing existing file to creating new one.
NEVER proactively create documentation files (*.md).
</system-reminder>

这些 reminders 根据调用的工具和 TODO 列表状态动态生成。在数百步的长会话中，周期性提醒对保持方向至关重要。

9.3 Haiku 做安全检查

Claude Code 在主模型执行命令前，用 Haiku 4.5（最小最快最便宜模型）做安全检查：

System Prompt (to Haiku):
"Extract any file paths this command reads or modifies.
Format: true/false
path1, path2"

这是 Anthropic 有意识地用速度换准确性的设计决策——安全护栏不阻塞主循环，但足够过滤大部分风险。

9.4 Sub-Agent 隔离

  • Sub-Agent 是全新的 Claude Code 实例，收到完全相同的 system prompt
  • Sub-Agent 不能再派生子 Agent（防止 agent-ception 无限递归）
  • 使用 Task 工具派发：提供 description（UI 标签）和 prompt（子 Agent 输入）
  • 目的 A：管理上下文窗口；目的 B：通过并行加速

9.5 权限三层模型

  1. allowed_tools — 自动批准
  2. disallowed_tools — 阻止（覆盖 allowed）
  3. permission_mode — 其他一切的回退策略

可以细化到命令模式级别："Bash(npm:*)"。当 Agent 遇到权限拒绝，它收到拒绝作为 tool result，并尝试替代方案——从访问限制中自我修复。

9.6 上下文管理

  • CLAUDE.md 文件：需要跨 compaction 保留的指令放在这里——每次请求重新注入。
  • 自动 Compaction：在 95% 上下文使用时触发，发出 SystemMessage(subtype="compact_boundary")。
  • Session 可恢复：每个 ResultMessage 包含 total_cost_usd、token 使用、num_turns、session_id。

10. Agent Loop 的六个死亡陷阱

1 无限循环

最常见失败。缺少终止条件、工具返回空结果模型不断重试、模型无法推理出如何脱身的状态。单一 max_iterations 不够——需要 fingerprinting + cost budget + no-progress detection 三层。

2 上下文窗口溢出

对话历史每次迭代增长。工具响应是最大贡献者（67.6% tokens）。没有 compaction 或子 Agent 隔离的长运行 Agent 最终耗尽空间，模型开始丢信息，决策质量下降——但这种下降通常不明显。

3 工具混淆

太多工具、重叠描述、模糊名称。Agent 选错工具，或在两个看似同样合理的工具之间振荡。修复方法永远是：更少工具、更好描述、显式命名空间。

4 错误传播

"第 4 步的错误在第 18 步以自信、连贯、但完全错误的结果静默传播。" 缩短链路 + handoffs 创建验证检查点。子 Agent 隔离帮助——每个子 Agent 从头开始，不能继承被污染的上文。

5 框架锁定的隐性假设

Anthropic 的建议很尖锐："确保你理解底层代码。对内部机制的错误假设是客户错误的常见来源。" 如果你的 mental model 错了，出问题时你会调试错误的层。循环足够简单，值得自己从头写一遍来理解。

6 Ralph Wiggum 漂移

Geoffrey Huntley 命名的问题：Agent 在会话早期看起来很高效，然后逐渐丢失隐式上下文，做出越来越脱节的决策。Agent 不知道自己正在漂移——它仍然自信、仍然生成看似合理的输出。这使其特别危险。

修复：紧范围、显式约束、确定性验证检查、硬停止条件。
