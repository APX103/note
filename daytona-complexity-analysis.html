安装中遇到的问题

端口冲突 — API 默认用 3000 端口，Runner 用 3003，与宿主机已有服务冲突，需要手动改映射
启动顺序竞态 — Runner 在 API 就绪前启动，导致连接失败。Runner 不会自动重试（需手动 restart）
快照初始化失败 — SnapshotManager 持续报 "No available runners"，需要手动操作数据库修复（设置 state=active，插入 snapshot_runner 记录，插入 region_quota）
API Key 无法通过 UI 创建 — 需要先完成 OIDC 登录流程；我直接操作数据库插入了 API Key
SSH Gateway 持续重启 — 不影响核心功能（代码执行），但说明组件间耦合紧密
SDK api_url 配置不直观 — 需要加 /api 后缀，文档未明确说明自托管场景
DNS 配置要求 — Proxy 需要通配符子域名解析（*.proxy.localhost），生产部署需配置 dnsmasq

可维护性分析

优势

  • 单个 docker-compose.yaml 一键启停
  • 所有组件都是标准 Docker 镜像，升级方便
  • 无需编译内核或 Firecracker（比 E2B 简单）
  • Python/TS/Go/Ruby/Java 多语言 SDK
  • 沙盒隔离度高（独立内核和网络栈）
  • 支持 pip install / 网络访问 / 文件操作
  • 开源项目活跃（GitHub star 多、迭代快）

风险

  • 组件多 — 14 个容器，任一故障需排查
  • Runner 需要 privileged 模式 — 安全风险
  • 自托管文档不完善 — 很多细节只能靠源码/数据库推断
  • 快照机制脆弱 — 初始化依赖多个表配合
  • API 镜像过大 — 6GB+，拉取和升级慢
  • 无内置监控告警 — 需要自建
  • 版本迭代快 — 数据库 schema 可能频繁变更
  • 生产部署额外依赖 — 通配符 TLS、Caddy、DNS

结论与建议

Daytona 自托管可用，但维护成本不低。

适合场景：需要多租户隔离、完整 SDK 体验、不想处理底层容器编排的团队。

不适合场景：只需要简单跑代码片段的场景 — 直接用 Docker SDK + 容器池会简单 10 倍。

如果团队只有 1-2 个人维护基础设施，建议：

  • 短期用 Daytona Cloud（免运维），等项目稳定了再考虑自托管
  • 或者直接用 Docker SDK + 预构建镜像实现轻量沙盒（放弃 Daytona 的 SDK 生态）
  • 如果决定自托管，务必写好自动化脚本处理快照初始化、健康检查、自动重启等问题
