🗂️ 记忆系统概览

Hermes Agent 采用三层分级记忆架构，灵感来源于计算机体系结构中的缓存分层策略（L1/L2/L3）。每一层在容量、延迟、持久性和访问方式上都有明确的分工，以在有限的 Token 预算下最大化上下文质量。

层级 存储介质 容量 / 延迟 核心定位
Layer 1
内置策划记忆		 ~/.hermes/memories/
MEMORY.md + USER.md		 ~2,200 + 1,375 chars
瞬时（系统提示中）		 高频关键事实，始终在线
Layer 2
会话搜索		 SQLite state.db
FTS5 全文索引		 无上限（全历史）
~20ms 查询		 按需召回，对话考古

Honcho / Mem0 / Hindsight 等		 提供者决定
每轮预取 + 同步		 深度长期记忆，智能抽取
设计哲学：系统提示（System Prompt）一旦构建，在会话期间保持不变，仅由上下文压缩触发重建。这样做的核心目的是保护上游 Prefix Cache——大多数推理后端会对相同前缀进行 KV Cache 复用，重建系统提示意味着丢弃这一优化。因此，Layer 1 采用"冻结快照"模式，写入磁盘但不实时反映到系统提示中。

冻结快照模式（Frozen Snapshot）

这是 Layer 1 最核心的设计决策。在 tools/memory_tool.py 中，MemoryStore 类维护了一个 _system_prompt_snapshot 字典：

class MemoryStore:
    def __init__(self, memory_char_limit=2200, user_char_limit=1375):
        self.memory_entries: List[str] = []
        self.user_entries: List[str] = []
        # Frozen snapshot for system prompt — set once at load_from_disk()
        self._system_prompt_snapshot: Dict[str, str] = {
            "memory": "", "user": ""
        }

    def load_from_disk(self):
        # ... 读取文件 ...
        self._system_prompt_snapshot = {
            "memory": self._render_block("memory", sanitized_memory),
            "user": self._render_block("user", sanitized_user),
        }

关键行为：

  • 会话启动时：从磁盘加载，捕获快照，注入系统提示
  • 工具调用时（add/replace/remove）：立即写入磁盘，但不更新快照
  • 上下文压缩后：重建系统提示时，才会重新调用 load_from_disk() 刷新快照

为什么用字符而非 Token 计量？

字符数（char count）是模型无关的计量方式。不同模型的 Tokenizer 对同一文本的切分结果差异很大（如 GPT-4 与 Claude 的 Token 比例约为 1:0.75）。使用字符作为上限可以保证在任何后端模型下都能保持大致相同的上下文占用，避免换模型后溢出或浪费。

🔍 Layer 2: 会话搜索（Session Search）

会话搜索基于 SQLite FTS5 全文搜索引擎，存储在 ~/.hermes/state.db 中。它的设计哲学与 Layer 1 完全不同：不做主动注入，只在被问到时响应。这避免了在每一轮对话中支付 Token 成本。

与 Layer 1 的核心差异

记忆（Memory）是 Agent 认为"必须在每一轮都记得"的关键事实——它始终占据系统提示的宝贵空间。

会话搜索（Session Search）是"当我们上周讨论过 X 时发生了什么？"这类追溯性问题——它不占用常驻 Token，只在 Agent 主动调用 session_search 工具时按需查询。

🌐 Layer 3: 外部记忆提供者（External Memory Providers）

外部记忆是 Hermes 的可插拔扩展层。系统通过 agent/memory_manager.py 统一编排，但后端可以由用户自由选择。一个关键约束是：同时只能激活一个外部记忆提供者，防止工具 Schema 膨胀和 backend 冲突。

MemoryProvider ABC 接口

所有外部提供者必须实现 agent/memory_provider.py 中定义的抽象基类：

class MemoryProvider(ABC):
    def initialize(self) -> None: ...          # 连接后端、预热
    def system_prompt_block(self) -> str: ...   # 静态提供者信息
    def prefetch(self, query: str) -> str: ...  # 每轮 API 调用前召回
    def sync_turn(self, user, assistant) -> None: ...  # 每轮结束后持久化
    def on_session_end(self, messages) -> None: ...  # 会话结束时最终刷新

Layer 2: 会话搜索

纯被动查询层——没有自动触发。只有在 Agent 主动调用 session_search 工具时才会执行 FTS5 查询。Agent 通常在以下场景调用它：

  • 用户问"我们上次讨论 X 时说了什么？"
  • Agent 需要验证某个事实是否在之前会话中已确认
  • 用户引用了之前会话中的某个概念，Agent 需要找回上下文

预取缓存策略（Prefetch Cache）

在 agent/conversation_loop.py 中，外部记忆的预取结果会被缓存，避免在单次用户请求涉及多次工具调用时重复查询：

# 在工具循环开始前预取一次，后续迭代复用
_ext_prefetch_cache = ""
if agent._memory_manager:
    try:
        _query = original_user_message if isinstance(original_user_message, str) else ""
        _ext_prefetch_cache = agent._memory_manager.prefetch_all(_query) or ""
    except Exception:
        pass

记忆提醒（Memory Nudge）

每 nudge_interval 个用户轮次（默认 10 轮），后台线程会触发一次记忆审查，提示模型回顾当前会话并建议记忆更新：

if (agent._memory_nudge_interval > 0
        and "memory" in agent.valid_tool_names
        and agent._memory_store):
    agent._turns_since_memory += 1
    if agent._turns_since_memory >= agent._memory_nudge_interval:
        _should_review_memory = True
        agent._turns_since_memory = 0

注意 在 Gateway 模式下，每个入站消息会创建新的 AIAgent 实例，因此 _turns_since_memory 会从 0 开始。系统会从 conversation_history 中重建有效的轮次计数。

Layer 3 注入：用户消息级的动态上下文

外部记忆预取的内容不会进入系统提示，而是在每次 API 调用时附加到当前用户消息中（agent/conversation_loop.py:823-839）：

if idx == current_turn_user_idx and msg.get("role") == "user":
    _injections = []
    if _ext_prefetch_cache:
        _fenced = build_memory_context_block(_ext_prefetch_cache)
        if _fenced:
            _injections.append(_fenced)
    if _injections:
        _base = api_msg.get("content", "")
        if isinstance(_base, str):
            api_msg["content"] = _base + "\n\n" + "\n\n".join(_injections)

围栏格式（agent/memory_manager.py:227-241）：

<memory-context>
[System note: The following is recalled memory context, NOT new user input.
Treat as authoritative reference data — this is the agent's persistent memory
and should inform all responses.]

{清洗后的外部记忆内容}
</memory-context>
为什么 Layer 3 不放在系统提示中？
外部记忆的内容是动态变化的——每轮预取结果可能不同。如果放入系统提示，会导致 Prefix Cache 频繁失效。此外，StreamingContextScrubber 会确保这些围栏标签在流式输出时不会泄露到 UI，保持用户体验整洁。

🛠️ Skill 机制全景

Hermes 的 Skill 系统是模块化的知识/工作流注入机制，灵感来自 Anthropic 的 Claude Skills 系统，采用渐进式披露架构（Progressive Disclosure）。Skill 不是简单的提示词片段，而是带有元数据、版本控制和生命周期管理的结构化知识包。

双轨制：skills/ vs optional-skills/

目录 用途 激活方式
skills/ 内置技能，随仓库发布 启动时自动同步到 ~/.hermes/skills/，默认加载
optional-skills/ 重型/小众技能，随仓库发布但不默认激活 需显式安装：hermes skills install official/<category>/<skill>

Skill 目录结构

skills/<category>/<skill-name>/
├── SKILL.md              # 主指令文件（必需）
├── references/           # 参考文档
├── templates/            # 输出模板
├── scripts/              # 可执行脚本
└── assets/               # 补充资源

渐进式披露架构

Skill 的内容不是一次性全部塞入上下文的，而是按需分层加载：

层级 内容 何时加载
元数据层 名称（≤64 chars）、描述（≤1024 chars）、标签、分类 系统提示构建时——用于生成可用技能清单
指令层 SKILL.md 完整内容（≤100K chars） Agent 调用 skill_view 时按需加载
链接层 references/、templates/、scripts/ 中的关联文件 Agent 主动读取或模板变量替换时

agent/prompt_builder.py 中的 build_skills_system_prompt() 负责将已安装的技能渲染为系统提示中的 XML 片段：

📊 Skill 评估标准与更新机制

什么时候应该创建一个 Skill？

tools/skill_manager_tool.py 中的 skill_manage 工具 Schema 明确给出了创建条件：

创建时机：
  • 复杂任务成功完成（涉及 5+ 次工具调用）
  • 过程中克服了错误或异常
  • 用户纠正过的方法最终被证明有效
  • 发现了非平凡的工作流或最佳实践
  • 用户明确要求"记住这个流程"
更新时机：
  • 指令过时或错误
  • 发现 OS 特定的失败模式
  • 缺少步骤或在实际使用中发现陷阱
跳过时机：简单的一次性任务（one-off），不值得持久化。

AGENTS.md 硬性标准（Hardline Standards）

任何新的或现代化的 Skill 必须满足以下标准，否则 PR 会被拒绝：

# 标准 说明
1 description ≤ 60 字符 一句话，句号结尾。禁止营销词汇（"powerful"、"comprehensive"、"seamless"）。
2 工具引用规范 必须引用原生 Hermes 工具（`terminal`、`read_file`、`patch`），禁止直接引用 shell 工具。
3 platforms: 平台门控 使用 POSIX 特有原语（fcntl、termios、osascript、apt 等）必须声明支持平台。
4 作者署名 外部贡献者的真实姓名 + GitHub 句柄优先；"Hermes Agent" 列为次要协作者。
5 现代章节顺序 # 标题 → ## When to Use → ## Prerequisites → ## How to Run → ## Quick Reference → ## Procedure → ## Pitfalls → ## Verification
6 文件组织 脚本放 scripts/，参考文档放 references/，模板放 templates/。

📝 总结

记忆系统三层架构设计精要

Layer 1（内置策划记忆）是 Agent 的"短期工作记忆"——容量受限但始终在线，采用冻结快照保护 Prefix Cache，用字符而非 Token 计量以保证模型无关性。

Layer 2（会话搜索）是 Agent 的"长期档案室"——无容量限制、按需查询、零常驻 Token 成本，FTS5 全文索引保证毫秒级响应。

Layer 3（外部记忆）是 Agent 的"外部大脑"——可插拔的智能后端，通过 prefetch/sync 钩子实现每轮动态召回和持久化，围栏格式确保动态内容不污染系统提示缓存。

Skill 系统核心设计

  • 渐进式披露：元数据常驻系统提示，完整指令按需加载，避免上下文膨胀。
  • 双轨制分发：内置技能默认加载，可选技能显式安装，兼顾开箱即用和按需扩展。
  • 自动命令化：每个技能自动成为 /skill-name Slash 命令，无需额外配置。
  • 多源生态：支持官方、GitHub、skills.sh、ClawHub、Well-known、直链六种安装来源。
  • 安全隔离：Hub 安装必须经过隔离区 + 安全扫描，防止恶意技能注入。
  • 用户修改保护：内置技能同步时检测用户修改并跳过覆盖，尊重用户定制。
