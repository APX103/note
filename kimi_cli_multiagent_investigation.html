Kimi CLI Multiagent 模式调研报告

从源代码和 Git 历史出发，全面调研 Kimi CLI 是否具备 Multiagent / Team / Swarm 协作模式

结论：当前不支持 历史上曾支持（已移除）

核心结论

Kimi CLI 当前不具备 Multiagent / Team / Swarm 协作模式。它只有一个"主从式 Subagent 委托"架构： Root Agent 可以创建独立的 Subagent（coder / explore / plan），但这些 Subagent 之间不能通信、不能协作、不能嵌套。 历史上（v0.32 ~ v1.24）曾有过 CreateSubagent 和 Task 工具构成的 multiagent 模块，但在 v1.25.0 被彻底移除。

当前支持的 Subagent 能力

支持

  • • 并发启动多个 subagent（LLM 并行工具调用）
  • • Foreground / Background 两种执行模式
  • • Subagent 持久化 + Resume（agent_id）
  • • 独立的 Context、Wire、DenwaRenji
  • • 审批请求透传到 Root UI
  • • 3 种内置类型：coder / explore / plan

不支持

  • • Subagent 间直接通信
  • • Subagent 嵌套创建
  • • 结果聚合 / Reduce 原语
  • • Coordinator / Leader 角色
  • • 协作式任务分解与合并
  • • 动态创建 subagent 类型

历史演变：从 Multiagent 到 Subagent

通过 Git 历史分析，Kimi CLI 确实曾经有过一个 multiagent 模块，但在 v1.25.0 被彻底重构移除。

时间线

v0.32 (~2025-11 早期)

引入 subagents 字段到 agent spec，添加 Task 工具

v0.57 (1815c090, 2025-11-20)

Task 工具移至 kimi_cli.tools.multiagent 模块

v0.58 (0fc0d210, 2025-11-21)

添加 CreateSubagent 工具，支持动态创建 subagent

v1.25.0 (ff09e97c, 2026-03-23) 重大重构

彻底移除 multiagent 模块，引入现在的 Agent 工具和 subagents/ 包。 删除文件：multiagent/__init__.py, create.md, create.py, task.md, task.py

v1.30.0 (2026-04)

修复并行 subagent 审批挂起问题

当前 (v1.43.0+)

稳定的"委托-执行-返回"模型，未向团队协作方向演进

旧版 Multiagent 模块代码（来自 Git 历史）

旧 Task 工具

class Task(CallableTool2[Params]):
    name: str = "Task"
    # 通过 runtime.labor_market.subagents
    # 查找已注册的 subagent
    # 调用 _run_subagent(agent, prompt)
    # 支持 summary continuation

通过名称查找并运行单个 subagent，仍是单次调用无持久化

旧 CreateSubagent 工具

class CreateSubagent(CallableTool2[Params]):
    name: str = "CreateSubagent"
    # 动态创建 subagent
    # 注册到 labor_market.dynamic_subagents
    # 持久化到 session.state

允许 LLM 在运行时动态创建新的 subagent 角色，v1.25.0 后移除

为什么移除 Multiagent 模块？

旧版的 Task 和 CreateSubagent 虽然是 multiagent 的雏形，但缺乏持久化、隔离性和恢复能力。 v1.25.0 的重构用 Agent 工具 + SubagentStore 替代了它们， 强调持久化实例（agent_id + resume）、隔离上下文（独立 Context）和统一审批（ApprovalRuntime）。 但重构方向是更强的单agent委托能力，而非多agent协作。

LLM 级别的"伪并发"

虽然代码层面没有并发协作原语，但 Kimi CLI 的 system prompt 明确鼓励 LLM 在一次响应中输出多个 Agent 工具调用， 从而实现多个 subagent 的并行启动。

System Prompt 中的并发建议

# src/kimi_cli/agents/default/system.md:52
- For broader codebase exploration and deep research, 
  use the Agent tool with subagent_type="explore".
  You can launch multiple explore agents concurrently 
  to investigate independent questions in parallel.

# src/kimi_cli/agents/default/explore.yaml:29
Prefer launching multiple explore agents concurrently 
when investigating independent questions.

# src/kimi_cli/tools/agent/description.md:30
- You want to investigate multiple independent questions 
  — launch multiple explore agents concurrently

本质

这种"并发"是提示词级别的建议，不是架构级别的支持。多个 Agent 工具调用会作为独立的工具调用来处理， 由 LLM 框架（kosong）决定是串行还是并行执行。代码中没有专门的并发调度器来管理这些 subagent 的生命周期、结果聚合或冲突解决。

如果要实现真正的 Multiagent，需要加什么？

1

Team Bus / Shared Channel

当前 Wire 是点对点的（soul -> UI）。需要 publish-subscribe 机制，新增 TeamMessage Wire 事件类型， 让 teammates 订阅共享频道。

2

TeamCoordinator

扩展 LaborMarket（静态类型注册表）为动态团队管理器，维护 team_members， 负责任务分解、结果聚合、冲突仲裁。

3

TeamStore

当前 SubagentStore 是每个 agent 独立的。需要共享状态层：共享上下文片段（全局 todo、共享知识库）、 文件锁或变更日志。

4

放宽嵌套限制

当前 role != "root" 时直接拒绝创建 subagent。需要引入 TeamLead 角色或层级权限系统， 允许 sub-team 管理。

5

并发原语

添加 asyncio.gather 级别的并发调度，支持"等待多个 subagent 完成后汇总"的原语， 类似 Claude Code 的 run_all()。

6

新的交互协议

Wire 协议扩展：TeamEvent（团队成员间消息）、TeamTaskAssign（任务分配）、 TeamVote / TeamConsensus（集体决策）。

6

总结

当前不是 Multiagent

  • • 没有 agent 间直接通信
  • • 没有协调器或领导者角色
  • • 没有结果聚合原语
  • • 禁止 subagent 嵌套
  • • 没有共享状态层

当前擅长什么

  • • 将独立子任务委托给专门化代理
  • • 并发启动多个只读探索 agent
  • • 前后台执行、持久化、恢复
  • • 隔离上下文，不污染 root
  • • 统一审批，用户体验一致

Kimi CLI 当前是一个设计良好的"主从委托"系统，适合把独立子任务交给专门化的代理执行。 如果要演进为真正的 Multiagent / Agent Team，最大的架构缺口是：peer-to-peer 通信机制、 共享状态层、动态团队编排器。这些在当前的 LaborMarket、SubagentStore 和 Wire 协议中都没有设计痕迹。
