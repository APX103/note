KimiSoul 核心架构与主循环

KimiSoul 是 Kimi CLI 的"灵魂"，位于 src/kimi_cli/soul/kimisoul.py。 每个 Agent（Root 或 Subagent）都拥有独立的 KimiSoul 实例，负责驱动 LLM 调用的主循环。

核心调用链

flowchart TB A[run user_input] --> B[_turn user_message] B --> C[_checkpoint 0] B --> D[append_message user] B --> E[_agent_loop] E --> F[Step 1] E --> G[Step 2] E --> H[Step N] F --> I[_step] G --> I H --> I I --> J[_grow_context] J --> K[append_message assistant] J --> L[append_message tool_results] I --> M{BackToTheFuture?} M -->|Yes| N[revert_to checkpoint] M -->|No| O{stop?} O -->|No| E O -->|Yes| P[TurnOutcome]

四个核心方法

run()

Turn 的入口。创建 ApprovalSource，触发 hook，发送 TurnBegin wire 事件，解析 slash command，最终调用 _turn()。

_turn()

单轮初始化。生成唯一的 turn_id，创建 checkpoint 0，将用户消息追加到 context，然后进入 _agent_loop()。

_agent_loop()

Step 循环控制器。每 step 检查 max_steps_per_turn，触发 compaction，创建 checkpoint，调用 _step()，处理异常和 steers。

_step()

单步执行核心。投递通知、收集动态注入、归一化 history、调用 LLM（kosong.step）、执行 tools、增长 context。

关键设计

_step() 中的 LLM 调用时，system prompt 作为参数传入（kosong.step(system_prompt=...)）， 并不存储在 context._history 中。这使得 system prompt 不会占用 context 的 token 额度， 且在 compaction 后可以被独立重新写入。

上下文更新机制

Context（src/kimi_cli/soul/context.py）是对话历史的容器，同时负责持久化到 context.jsonl。 所有消息都通过 Context.append_message() 进入 history。

消息进入 Context 的路径

flowchart LR subgraph User["用户输入"] U1[run user_input] U2[_turn] U3[append_message user] end subgraph LLM["LLM 响应"] L1[_step] L2[kosong.step] L3[_grow_context] L4[append_message assistant] end subgraph Tool["工具结果"] T1[_step] T2[execute tools] T3[_grow_context] T4[append_message tool_results] end U3 --> C[Context] L4 --> C T4 --> C C --> F[context.jsonl]

各类消息的处理方式

消息类型来源进入 Context 的方式备注
user 用户输入 / Steer _turn() → append_message() 触发新一轮 turn 的原始输入
assistant LLM 生成 _grow_context() → append_message(result.message) 包含文本和 tool_calls
tool Tool 执行结果 _grow_context() → append_message(tool_messages) 由 tool_result_to_message() 转换
_system_prompt Agent spec Context.write_system_prompt() 单独存储，不在 history 列表中
_checkpoint KimiSoul Context.checkpoint() 用于 D-Mail 回退的标记点
_usage LLM usage 响应 Context.update_token_count() 记录 input tokens，清空 pending

Context 的 dual-track token 追踪

Context 内部维护两个 token 计数器：
• _token_count：上次 LLM 调用返回的准确 input tokens
• _pending_token_estimate：自上次 usage 后新增消息的启发式估算（字符数 ÷ 4）
token_count_with_pending = _token_count + _pending_token_estimate 用于触发 compaction。

Compaction：为什么不会超长？

这是回答"每个 agent 都用 KimiSoul，不会超长么"这个问题的关键。 Kimi CLI 没有使用 Sliding Window 或 LRU，而是采用了独特的 LLM 驱动的 Compaction（压缩） 机制。

一句话总结

当上下文接近 token 上限时，系统会保留最后 2 条对话，把前面的所有历史丢给另一个 LLM 调用进行压缩总结， 然后将总结 + 保留的 2 条消息作为新的上下文。这是一种"以 LLM 管理 LLM"的策略。

触发条件

flowchart LR A[token_count_with_pending] --> B{>= max_size * 0.85?} A --> C{+ 50000 >= max_size?} B -->|Yes| D[触发 Compaction] C -->|Yes| D D --> E[SimpleCompaction.prepare] E --> F[保留尾部 2 条消息] E --> G[压缩前面所有历史] G --> H[调用 compaction LLM] H --> I[生成 summary] I --> J[清空原 context] J --> K[重新写入 system_prompt] J --> L[创建新 checkpoint] J --> M[追加 summary + 保留消息]

配置阈值（具体数字）

配置项默认值作用
compaction_trigger_ratio 0.85 当 token 占比达到 85% 时触发压缩
reserved_context_size 50,000 为 LLM 回复预留的 token 空间
max_steps_per_turn 1,000 单轮最大 step 数（防止死循环）

以常见的 128K 上下文模型为例： 128000 * 0.85 = 108,800 tokens 触发 ratio 条件； 128000 - 50000 = 78,000 tokens 触发 reserved 条件。 满足任一即压缩。

没有 Sliding Window / LRU

与许多其他 Agent 框架不同，Kimi CLI 不使用基于消息年龄或数量的滑动窗口淘汰策略。 它的策略只有两种：Compaction（压缩总结）和 Checkpoint Revert（D-Mail 回退）。 这意味着旧消息要么被总结保留，要么被完整回退恢复，不会无声消失。

Token 占用与上下文管理

每个 Agent 的 KimiSoul 独立吗？会超长吗？

Root Agent

  • • 独立的 KimiSoul 实例
  • • Context 绑定到 session/context.jsonl
  • • 独立的 DenwaRenji
  • • 自己的 compaction 周期

Subagent A / B / C

  • • 各自独立的 KimiSoul 实例
  • • Context 绑定到 subagents/{id}/context.jsonl
  • • 各自的 DenwaRenji（全新实例）
  • • 各自的 compaction 周期

结论：不会互相影响

每个 agent 的 context 是完全物理隔离的（不同的 jsonl 文件）。 Root agent 的 compaction 不会清理 subagent 的历史，反之亦然。 但 subagent 运行结束后，它的 context.jsonl 会永久保留在磁盘上（直到 session 被删除），不会自动清理。

Token 统计方式

# compaction.py:44
# 启发式估算（pending 阶段使用）
def estimate_text_tokens(messages: Sequence[Message]) -> int:
    total_chars = 0
    for msg in messages:
        for part in msg.content:
            if isinstance(part, TextPart):
                total_chars += len(part.text)
    return total_chars // 4   # ~4 chars per token for English

DenwaRenji 与 D-Mail 机制

名字来源

Denwa Renji（電話レンジ）是动漫《Steins;Gate》（命运石之门）中"电话微波炉（暂定）"的日文读音。 在作品中，D-Mail 是通过这台设备向过去发送的邮件，能够改变时间线。代码中的 D-Mail 机制正是对这一概念的致敬—— 向"过去的自己"（即之前的 checkpoint）发送信息。

核心组件

DenwaRenji

极简的邮箱中转站。只有两个字段：_pending_dmail（最多一条待处理邮件）和 _n_checkpoints（已知 checkpoint 总数）。

DMail

数据模型：message: str（要发送的内容）和 checkpoint_id: int（目标 checkpoint）。

BackToTheFuture

受控异常。携带 checkpoint_id 和要注入的 messages，被 _agent_loop() 捕获后执行回退。

checkpoint_id=2, message="..." Checkpoint 系统

Checkpoint 是对话历史中的可回退标记点。每次 _agent_loop 的 step 开始时会创建一个新的 checkpoint： 

# context.py:123-133
async def checkpoint(self, add_user_message: bool):
    checkpoint_id = self._next_checkpoint_id  # 从 0 递增
    self._next_checkpoint_id += 1
    # 在 jsonl 中写入标记
    await f.write(json.dumps({"role": "_checkpoint", "id": checkpoint_id}) + "\n")
    # 如果启用了 SendDMail 工具，插入可见的 CHECKPOINT N 消息
    if add_user_message:
        await self.append_message(
            Message(role="user", content=[system(f"CHECKPOINT {checkpoint_id}")])
        )

硬隔离

Runtime.copy_for_subagent() 中为 subagent 创建全新的 DenwaRenji() 实例。 SendDMail 工具注入的是当前 Runtime 的 DenwaRenji， 而 KimiSoul 只检查自己 Runtime 的 DenwaRenji。 这是一个设计上的硬隔离，D-Mail 严格限制在单个 agent 内部。

Wire 协议与 Agent 间通信

Foreground 消息分类处理

在 ForegroundSubagentRunner._make_ui_loop_fn() 中，subagent 的 wire 消息被分为三类：
1. ApprovalRequest / ApprovalResponse / ToolCallRequest / QuestionRequest → 直接透传到父 Wire
2. HookRequest → 忽略（子 agent 的 hook 不应影响父 UI）
3. 其他所有 Event → 包装为 SubagentEvent 后发送给父 agent

RootWireHub：Session 级广播中心

RootWireHub 是一个 session 级别的广播队列，所有 agent（root + subagent）共享同一个实例。 它主要用于 out-of-turn 消息 的广播，最典型的场景是 Approval 请求/响应的跨 agent 投射。

merge=True 的含义

merge=True（UI 渲染用）

订阅 _merged_queue。连续的 TextPart 会被合并为一条消息， ToolCall + 多个 ToolCallPart 合并为完整调用。 减少 UI 刷新次数，适合渲染。

merge=False（记录/回放用）

订阅 _raw_queue。接收每一条原始消息，100% 保真。 适合写入 wire.jsonl 进行持久化和回放。

总结

KimiSoul

  • • run → _turn → _agent_loop → _step 四级调用链
  • • 每个 agent 独立的 KimiSoul + Context 实例
  • • system prompt 作为参数传入，不占 context
  • • Compaction 在 85% 或 +50K 阈值时触发
  • • 保留 2 条消息，其余压缩为 summary
  • • 无 Sliding Window

DenwaRenji

  • • 名字致敬《Steins;Gate》的"电话微波炉"
  • • D-Mail = 向过去的 checkpoint 发送消息
  • • 通过 BackToTheFuture 异常实现时间旅行
  • • revert_to(checkpoint_id) 回退 context
  • • 每个 agent 独立的 DenwaRenji 实例
  • • 严格禁止跨 agent 通信

Wire 协议

  • • SPMC 发布-订阅通道（Soul → UI）
  • • 双队列：raw（保真）+ merged（合并）
  • • SubagentEvent 包装子 agent 事件
  • • Approval 请求直接透传，不包装
  • • RootWireHub 广播 session 级消息
  • • wire.jsonl 记录完整事件流
