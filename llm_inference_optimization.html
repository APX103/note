⚡ PD分离（Prefill-Decode Separation）

1. 两个阶段的本质差异

维度 Prefill阶段 Decode阶段
计算模式 计算密集型（Compute-bound） 内存带宽密集型（Memory-bound）
输入特征 一次性处理长Prompt（几百~几万token） 逐token自回归生成
并行度 高（可充分并行计算注意力） 极低（每次只处理1个token）
瓶颈 FLOPs利用率、矩阵乘法吞吐 HBM带宽、KV Cache读写延迟
批处理 适合大batch（吞吐量高） 适合小batch（延迟敏感）
GPU利用率 高（Tensor Core饱和） 极低（通常<10%）

2. 分离架构数据流

┌─────────────┐ ┌─────────────────┐ ┌─────────────┐ │ 用户请求 │─────▶│ Prefill Cluster │─────▶│ Decode Cluster│ │ (Prompt) │ │ (计算密集型) │ │ (带宽密集型) │ └─────────────┘ └─────────────────┘ └─────────────┘ │ │ ▼ ▼ 输出: 首个token + 持续生成后续token KV Cache (通过RDMA/ 返回给用户 NVLink传输)

3. 关键挑战：KV Cache传输

Prefill完成后需将完整KV Cache传输到Decode节点。以32K上下文、32层、32头、128维为例：

# KV Cache大小计算 total_kv = 2 × seq_len × num_layers × num_kv_heads × head_dim × bytes_per_param = 2 × 32768 × 32 × 8 × 128 × 2 # FP16 = 2 bytes ≈ 16 GB # 单请求单batch

优化手段：

  • RDMA：绕过CPU，GPU直接网卡传输，延迟<10μs
  • KV Cache压缩/量化：传输INT8/FP8甚至INT4格式
  • Pipeline overlap：传输与计算重叠
  • Chunked Prefill：长Prompt分chunk流水线传输

💾 KV Cache管理技术演进

1. 存储结构

# 单层Transformer的KV Cache形状 K_cache shape: [batch_size, num_kv_heads, seq_len, head_dim] V_cache shape: [batch_size, num_kv_heads, seq_len, head_dim] # 示例: Llama-3-70B, batch=1, seq=8192, 80层, GQA(8 KV头) 每层KV Cache = 2 × 1 × 8 × 8192 × 128 × 2B = 32MB 总KV Cache = 80 × 32MB = 2.56GB (单请求) # 若batch=32, seq=128K: 约1.3TB

结论：KV Cache通常占总显存的70-90%，是推理优化的绝对核心。

2. 技术演进五代

第一代：朴素实现（Naive）

为每个请求预分配最大长度，短请求浪费90%+显存，无法处理变长。

第二代：PagedAttention（vLLM, 2023）

核心创新：将KV Cache划分为固定大小的Block（页），通过Block Table非连续映射。

┌─────────────────────────────────────────┐ │ 逻辑KV Cache (连续) │ │ [t1][t2][t3][t4][t5][t6][t7][t8]... │ └─────────────────────────────────────────┘ ↓ 映射到物理Block ┌─────────────────────────────────────────┐ │ 物理GPU内存 (非连续) │ │ Block0: [t1][t2][t3] Block1: [t4][t5] │ │ Block2: [t6][t7] Block3: [t8]... │ │ (通过Block Table映射，支持共享和回收) │ └─────────────────────────────────────────┘

收益：内存浪费从>80%降至<4%，支持KV Cache共享。

第三代：Prefix Caching / RadixAttention（SGLang, 2024）

通过Radix Tree匹配最长公共前缀，命中部分直接复用已缓存KV。

root / "你是" "The" / \ \ "一个" "AI" "weather" / / "助手" "is" / [共享KV Cache节点]

收益：系统Prompt场景Prefill时间减少90%+。

第四代：KV Cache压缩与驱逐

技术 原理 效果
H2O 保留注意力贡献最大的token的KV 保留20% KV，性能保持95%
SnapKV 动态选择关键KV 压缩50%，性能保持98%
StreamingLLM 保留初始token+最近token 无限长流式生成
量化压缩 INT8/INT4/FP8存储 显存减半

第五代：分离式KV Cache

将KV Cache从GPU HBM部分分离到CPU内存/SSD，通过预测性预取隐藏加载延迟。Moonshot Kimi的200万上下文即采用此方案。

🧠 注意力架构演进：MQA / GQA / MLA / Mamba

核心问题对比

技术 层级 核心手段 KV Cache变化 是否改变模型 成熟度
MQA 模型架构 所有Q头共享1组KV ÷ n_heads ✅ 成熟
GQA 模型架构 分组共享KV ÷ (n_heads/n_kv) ✅ 主流
MLA 模型架构 低秩隐向量压缩 ÷ 8~20 ✅ 先进
Mamba 模型架构 状态空间替代注意力 O(1)常数级 ✅ 替代Transformer 实验
PagedAttention 系统运行时 分页虚拟内存管理 管理效率提升 ❌ 标准

1. MQA（Multi-Query Attention）

核心：所有查询头共享同一组Key和Value头。

# 标准MHA: 32个Q头, 32个K头, 32个V头 # MQA: 32个Q头, 1个K头, 1个V头 KV Cache大小 = 2 × batch × 1 × seq_len × head_dim # 原为32 = 原始大小的 1/32

代价 注意力表达能力减弱，模型质量轻微下降。代表 PaLM, GPT-3.5

2. GQA（Grouped-Query Attention）

核心：折中方案，分组共享KV，既省内存又保质量。

# GQA: 32个Q头分8组，每组4个Q共享1个KV Query Heads: [Q₁ Q₂ Q₃ Q₄] [Q₅ Q₆ Q₇ Q₈] ... (共32个，分8组) ↓ ↓ Key/Value: [K₁/V₁] [K₂/V₂] ... (共8个共享组) KV Cache = 原始大小的 1/4 (32→8)

当前主流 Llama-2/3, Qwen, Mistral

3. MLA（Multi-head Latent Attention）

核心：不直接缓存K/V，而是缓存低维隐向量，使用时通过投影恢复。

# 标准: 缓存 [K, V] → 高维 (如2048维) # MLA: 缓存 [c] → 低维隐向量 (如512维) 使用时: K = W_K · c, V = W_V · c # DeepSeek-V2实测: KV Cache减少为传统MHA的 5-13% # 推理成本降至行业平均的 1/5 ~ 1/10

DeepSeek杀手锏 数学上最优低秩近似，比GQA更精细。

4. Mamba

核心：用状态空间模型（SSM）替代注意力机制，从根本上不需要KV Cache。

Transformer: h_t = Attention(Q_t, [K₁...K_t], [V₁...V_t]) ↑ 需要缓存所有历史K,V → O(t)内存 Mamba: h_t = SSM(x_t, h_{t-1}) ↑ 只维护固定大小状态向量 → O(1)内存

局限 生态极不成熟，尚未证明能scale到GPT-4级别。状态 2024年研究方向。

🏆 竞争格局与三条技术路线

路线A：PD分离（资源专精）

代表：DistServe, Splitwise, Moonshot, DeepSeek, vLLM-PD

核心：Prefill和Decode物理分离，各自优化

优势 TTFT降低30-50%，Throughput提升2-5x
劣势 需要高速网络（400Gbps+），调度复杂

路线B：统一调度优化（部署简单）

代表：Orca, FastServe, Sarathi-Serve

核心：单集群内通过迭代级调度动态调整batch

优势 无需高速互联，部署简单
劣势 无法根本解决资源冲突，长上下文下效果差

路线C：模型架构革新（一劳永逸）

代表：MQA/GQA/MLA, Mamba, RWKV, RetNet

核心：改变注意力机制，减少或消除KV Cache

优势 不依赖系统优化，理论上无限长上下文
劣势 Mamba等尚未验证能scale到超大模型

📌 核心结论

  1. PD分离解决的是计算阶段的调度效率问题，KV Cache管理解决的是状态存储的规模与效率问题
  2. 两者结合使现代LLM服务能够同时实现：低TTFT + 高Throughput + 长上下文 + 低成本
  3. 当前 winner 是 PD分离 + GQA/MLA + PagedAttention 的组合拳
  4. vLLM是开源标准，但企业级SOTA是闭源的（Moonshot/DeepSeek/Google）
  5. 未来方向：CXL内存扩展、KV Cache量化压缩、PD分离+Serverless化（PrFaaS）
