1. KV Cache 显存管理

1.1 显存占用公式

KV Cache (bytes) = 2 × num_layers × num_heads × head_dim × seq_len × batch_size × sizeof(dtype)

以 Llama-3-70B 为例：80 层、64 头、128 dim、FP16、batch=32、seq=4096 → 约 34 GB。权重本身约 140 GB（FP16），KV Cache 在长序列下会反超权重。

1.2 PagedAttention 的 Block Table 机制

vLLM 的核心创新不是 Attention 算法，而是显存分配器。它把 KV Cache 从连续内存改为分页管理：

  • Block：固定大小（通常为 16 tokens），存储对应 token 的 K/V 张量。
  • Block Table：每个序列维护一个逻辑到物理 block 的映射表，类似 OS 页表。
  • 物理块不连续：通过映射表保持逻辑连续性，显存利用率从 ~60% 提升到 ~90%。
// vLLM 核心数据结构简化示意
struct BlockTable {
    std::vector<int> physical_blocks;  // 逻辑 block i -> 物理 block id
    int num_tokens;                    // 当前序列实际长度
};

// 分配策略：按需分配，而非预分配 max_seq_len

1.3 Copy-on-Write & Prefix Caching

vLLM v0.3+ 引入的Prefix Caching（又称 RadixAttention）：

  • 系统维护一个 Radix Tree，缓存已计算过的 KV Cache block。
  • 新请求若前缀匹配缓存 block，直接引用，引用计数 +1。
  • 当某个序列需要修改共享 block（如继续生成 diverge），触发 Copy-on-Write，复制该 block 到新的物理块，原 block 引用计数 -1。

这对多轮对话、RAG（大量重复 system prompt）场景提升巨大，TTFT 可降低 50% 以上。

2. Prefill-Decode 分离架构

2.1 为什么必须分离

维度PrefillDecode
计算模式矩阵-矩阵乘法 (GEMM)矩阵-向量乘法 (GEMV)
瓶颈计算密集 (Compute-bound)访存密集 (Memory-bound)
GPU 利用率高（Tensor Core 饱和）低（SM 空闲等待 HBM）
关键指标TTFT (Time To First Token)TPOT (Time Per Output Token)
Batch 策略合并长 prompt，增大 batch保持大 batch，摊销权重读取

混部时，一个长 prompt 的 Prefill 会抢占 Decode 的 SM 和带宽，导致 TPOT 抖动（P99 延迟飙升）。分离后，两类节点可按各自瓶颈独立扩缩容。

2.2 KV Cache 传输：零拷贝与 RDMA

Prefill 节点完成计算后，需将 KV Cache 传输到 Decode 节点。传输量：

Transfer = 2 × layers × heads × head_dim × seq_len × sizeof(dtype)

以 70B 模型、4K 序列、FP16 为例：约 4.2 GB。若走 TCP/IP，延迟不可接受。工业级方案：

  • GPUDirect RDMA：网卡直接读写 GPU HBM，绕过 CPU 和系统内存，延迟 ~5-10 μs。
  • NVLink / NVSwitch：同节点内 GPU 间 P2P，带宽 900 GB/s，用于 TP 组内传输。
  • 零拷贝序列化：避免 PyTorch tensor → numpy → protobuf 的拷贝，直接发送 GPU 内存指针。
网络管理员视角：InfiniBand 的 Adaptive Routing 对 LLM 推理不利。Prefill→Decode 的 KV 传输是大消息流，应配置 Fixed Path Routing 避免乱序，并开启 DCQCN 拥塞控制。RoCEv2 需关注 PFC/ECN 配置，防止 RDMA 拥塞导致的 NIC 性能断崖。

2.3 Chunked Prefill：不纯粹的分离

纯 PD 分离的问题：Prefill 节点在高负载下 TTFT 积压，Decode 节点空闲。Chunked Prefill 将长 prompt 切成多个 chunk，在 Decode 节点上穿插执行：

  • 每个 chunk 当作一个 "mini-prefill"，复用 Decode 节点的空闲算力。
  • chunk 间通过 KV Cache 增量传递，避免重复计算。
  • 调度器需维护 num_new_tokens 预算，确保 Decode 的 TPOT SLO 不被破坏。

2.4 异构 PD 分离

Prefill 节点可用 H100（高算力），Decode 节点用 A100 或 L40S（高显存、低成本）。甚至 Decode 用 CPU（llama.cpp）处理小模型。挑战：

  • KV Cache 格式对齐（不同架构的 tensor layout 可能不同）。
  • 精度一致性（Prefill FP16，Decode INT8 需校准）。
  • 负载均衡：Prefill 是突发流量，Decode 是稳态流量，需独立 autoscaler。

3. Attention 算子优化

3.1 FlashAttention 演进

版本核心改进适用场景
FlashAttention-1Softmax Tiling + Online 重计算，减少 HBM 读写训练 & Prefill
FlashAttention-2减少 non-matmul FLOPs，更好的 Warp 划分，无需 split KV训练 & Prefill
FlashAttention-3异步 WGMMA + FP8 支持，Hopper 架构专用H100/B200 训练
FlashDecoding在 KV 序列维度并行化，解决 decode 阶段 GPU 利用率低Decode 长上下文
FlashDecoding++异步 Softmax（统一 max 值）、Flat GEMM 双缓冲Decode 通用

3.2 FlashDecoding 的数学细节

Decode 阶段 Q 长度为 1，FlashAttention 仅在 (batch, query_block) 维度并行化，若 batch < SM 数（A100 有 108 SM），大量 SM 空闲。

FlashDecoding 引入第三并行维度：KV 序列分块。

// FlashDecoding 三步法
// 1. 将 KV 分成 chunks（如每 chunk 1024 tokens）
// 2. 每个 chunk 独立计算 attention，输出 partial softmax 结果 + log-sum-exp (lse)
// 3. 全局归约：根据 lse 加权合并各 chunk 输出

// Kernel 1: 并行处理所有 chunk（占满 SM）
// Kernel 2: 轻量级归约（每个 token 一个 warp）

关键：Softmax 的归约性质允许分块计算后合并。设某 chunk 的局部 max 为 m_i，sum 为 l_i，则全局归约：

m_global = max(m_i),    l_global = Σ l_i · exp(m_i - m_global)

3.3 FlashDecoding++：异步 Softmax

FlashDecoding 的瓶颈：分块间的 softmax 同步更新（需等待全局 max）导致 ~20% 开销。FlashDecoding++ 的核心洞察：

  • Softmax 的缩放因子可以是任意常数 φ，不一定要用 max(x)。
  • 对 LLM 的注意力分数统计：99.99% 的值落在固定区间（如 Llama2-7B 为 [-16.8, 6.5]）。
  • 选取一个保守的 φ（如 -20），各 chunk 可完全独立计算，无需同步。
  • 溢出保护：若实际值超出 φ 范围，fallback 到标准 FlashAttention 重计算（发生频率 < 0.5%）。

效果：Decode 阶段比 FlashDecoding 快 up to 2.02×，Prefill 阶段比 FlashAttention-2 快 1.19×。

3.4 Flat GEMM 优化

Decode 阶段的线性投影（Q/K/V/O/FFN）输入是扁平矩阵（M=1 或 M=batch_size，很小）。cuBLAS 对此优化不佳。FlashDecoding++ 采用：

  • Double Buffering：利用 Shared Memory 双缓冲隐藏数据加载延迟。
  • Warp-level 特化：当 M=1 时，一个 warp 处理一个输出列，避免 block-level 同步。
  • 对 A100 平均提升 7%，对 RTX 3090 提升 up to 52%。

3.5 算子融合趋势：MegaKernel

2025-2026 年的方向：将整个 Transformer Layer 融合为单个 CUDA Kernel，消除 kernel launch 和中间结果 HBM 读写：

  • FlashInfer：vLLM/SGLang 的底层 kernel 库，提供 fused attention + sampling。
  • Stanford MegaKernel：单 kernel 完成整个 layer，通过 DAG 离线搜索最优融合路径。
  • Ada-MK（百度）：工业级 MegaKernel 部署，证明在广告系统可行。
注意：MegaKernel 的代价是编译时间极长（小时级），且对动态 shape（变长序列）支持有限。目前更适合固定 batch size 的推荐/广告场景，对话场景仍需 Progressive Lowering。

4. 量化与压缩

4.1 权重量化方案对比

方法位宽核心思想精度损失适用
RTNW8/W4逐层 min/max 量化高快速实验
GPTQW4/W3逐层 OBS 优化，补偿误差中静态部署
AWQW4激活感知：保护 salient weight 通道低通用推荐
SmoothQuantW8A8将量化难度从激活迁移到权重低INT8 推理
FP8 (E4M3/E5M2)W8A8利用 H100/B200 原生 FP8 Tensor Core极低Hopper+

4.2 KV Cache 量化

KV Cache 比权重更敏感，因为量化误差会随自回归累积。常用策略：

  • Per-channel / Per-token：对 K 按 head 维度、对 V 按 token 维度分别找缩放因子。
  • Dynamic INT8：每轮 decode 在线计算 max/min，实时量化。开销极小（< 1%）。
  • FP8 KV Cache：H100 上直接存储 FP8，无需反量化，带宽减半，精度损失 < 1%（perplexity）。

4.3 量化与 PD 分离的交互

Prefill 节点可用 FP16 保证首字精度，传输到 Decode 节点后转为 INT8/FP8。需注意：

  • 缩放因子（scale）需随 KV Cache 一并传输，增加少量 metadata。
  • 若 Decode 节点用 FP8，Prefill 节点的 FP16→FP8 转换应在 GPU 内完成，避免 CPU 中转。

5. Batching & 调度

5.1 Continuous Batching (In-flight Batching)

传统 Static Batching：等一个 batch 内所有序列完成才释放资源，短序列被长序列阻塞。Continuous Batching 允许：

  • 每轮 forward 后，已完成序列退出，新序列加入。
  • 通过 batch_token_budget 控制每轮处理的 token 总数（而非序列数）。
  • vLLM 的 Scheduler 维护 running、swapped、waiting 三个队列。
// vLLM Scheduler 核心逻辑（极度简化）
class Scheduler {
    std::deque<SequenceGroup*> waiting;   // 新请求
    std::deque<SequenceGroup*> running;   // 当前执行中
    std::deque<SequenceGroup*> swapped;   // 已换出到 CPU

    void schedule_step() {
        // 1. 尝试将 waiting 中请求加入 running（检查显存）
        // 2. 若显存不足，将 running 中低优先级序列 swap_out
        // 3. 若 swapped 中有高优先级序列，swap_in（需带宽预算）
        // 4. 返回本次 step 的 batch 配置
    }
};

5.2 Splitwise：PD 分离的调度抽象

Splitwise（OSDI'24）将请求生命周期分为三段：

  • P-phase：在 Prefill 节点执行，输出 KV Cache。
  • D-phase：在 Decode 节点执行，自回归生成。
  • Transfer-phase：KV Cache 通过网络迁移。

调度器需同时管理三类资源：Prefill 算力、Decode 算力、网络带宽。采用 Water-filling 算法：优先将请求路由到负载最轻的 Prefill 节点，并预分配 Decode 槽位。

5.3 抢占策略

策略机制代价适用
Swapping将 KV Cache 换出到 CPU DRAM带宽占用，恢复时需 swap_in长对话、低优先级后台任务
Recomputation丢弃 KV Cache，恢复时重新 Prefill计算浪费，但无带宽压力短序列、网络拥塞时
Abort直接终止请求用户体验差严重过载、超出 SLO

6. MoE 推理（你需要补的）

6.1 核心差异：稀疏激活

MoE（Mixture of Experts）在推理时的关键特征：每个 token 只激活 Top-K 个专家（通常 K=1~8），而非全部参数。例如 DeepSeek-V3 有 256 个专家，每 token 激活 8 个，实际参数量 671B，激活量仅 37B。

6.2 推理瓶颈：All-to-All 通信

MoE 的 FFN 前向传播需经过 Router 分发 token 到不同专家：

  • Dispatch：将 token 发送到持有目标专家的 GPU（All-to-All）。
  • Combine：专家计算后将结果回传（All-to-All）。

All-to-All 的通信量与 batch size × hidden_dim × num_experts_per_token 成正比。在 EP（Expert Parallelism）组内，这成为主要瓶颈。优化手段：

  • EP + TP 混合：EP 跨节点分布专家，TP 在节点内张量并行。如 64 卡：8 路 EP × 8 路 TP。
  • All-to-All 与计算重叠：在专家计算时预取下一批 token 的 dispatch。
  • 负载均衡约束：训练时用 aux loss 强制均衡，推理时若某专家过载，需动态 overflow 到备用专家。

6.3 DeepSeekMoE 的推理特点

  • 共享专家：所有 token 必过 1 个共享专家 + K 个 routed 专家，共享专家的 KV Cache 可全局缓存。
  • 细粒度专家：专家数量多（256+），单个专家参数量小，适合 EP 细粒度分布。
  • MLA：DeepSeek-V2/V3 使用 Multi-head Latent Attention，KV Cache 压缩比 MHA 降低 5-10×，缓解 MoE 的显存压力。
MLA 简要原理：将 K/V 通过低秩矩阵压缩为 latent vector（维度远小于 num_heads × head_dim），推理时只缓存 latent vector，计算时在线展开。这是 DeepSeek 能在单节点跑 128K 上下文的关键。

7.2 K8s 部署要点

基于你的 K8s 背景，直接列生产级 checklist：

  • GPU 调度：使用 NVIDIA device plugin + GPU Operator，开启 MIG（A100/H100）或 MPS（共享上下文）提高利用率。
  • 拓扑感知：通过 TopologyManager + NUMANode 绑定，确保 Pod 分配的 GPU 在同一 NVSwitch 域内，降低 NCCL 通信延迟。
  • 弹性伸缩：HPA 基于 GPU 利用率（非 CPU）触发。自定义 metrics：queue_length（等待请求数）、ttft_p99、tpot_p99。
  • Pod 隔离：推理 Pod 需 Guaranteed QoS，避免被驱逐。Prefill 和 Decode 用不同 PriorityClass。
  • 网络策略：PD 分离节点间开放 RDMA 端口（默认 4791 for RoCEv2），限制跨 namespace 访问。

7.3 模型加载与热更新

  • 分层加载：权重从对象存储（S3/OSS）→ CPU DRAM → GPU HBM。大模型（70B+）加载需 10-30 秒，用 parallel loading（多流 cudaMemcpy）加速。
  • 模型缓存：节点本地 SSD 缓存热门模型，避免重复拉取。
  • 热更新：滚动更新时，新 Pod 先加载权重，再切流量，避免冷启动 TTFT 飙升。K8s 可用 preStop + readinessProbe 控制。

8. 性能分析

8.1 Roofline Model 在推理中的应用

快速定位瓶颈类型：

  • Compute-bound：Prefill 阶段、大 batch decode、MoE 的 All-to-All 前计算。
  • Memory-bound：小 batch decode、长上下文 KV Cache 读取、权重读取。
  • Communication-bound：TP/EP 的 AllReduce/All-to-All、PD 分离的 KV 传输。
算术强度 = FLOPs / Bytes  （单位：FLOP/Byte）
A100 HBM 带宽 2039 GB/s，FP16 Tensor Core 312 TFLOPS → 屋顶线拐点 ≈ 153 FLOP/Byte

Decode 阶段 GEMV 的算术强度通常 < 10，远低于拐点，故绝对内存 bound。

8.3 一个典型 Decode 瓶颈分析案例

  1. 用 Nsight Systems 发现 GPU 利用率仅 30%，但 HBM 带宽 1800 GB/s（接近上限）。
  2. 确认内存 bound：FlashDecoding 的 KV chunk 读取占 85% 时间。
  3. 尝试增大 batch size：显存不足（KV Cache 占满）。
  4. 启用 KV Cache INT8 量化：显存减半，batch size 翻倍，TPOT 降低 40%。
  5. 但精度下降，改用 FP8（H100）：速度比 INT8 快 15%，精度无损。
最后提醒：推理优化没有银弹。长上下文场景优先优化 KV Cache 管理和 Attention 算子；高并发场景优先优化 Batching 和调度；MoE 场景优先优化 All-to-All 通信。先 profile，再优化，避免盲目上量化或 PD 分离。
