SkillRouter × SkillReducer

大规模 Agent 技能系统的「精准路由」与「结构化减肥」架构解析

核心命题

当 Agent 技能生态从几十个增长到上万个时，两个致命问题浮现： 找不准（海量同质技能中路由失效）和装不下（技能膨胀导致上下文淹没）。 两篇论文分别给出了答案，合起来是一套完整的技能基础设施方案。

SkillRouter — 路由层
🎯
全文本信号革命

现有框架（Claude Code / OpenClaw）路由时只暴露名称+描述，隐藏完整实现（body）。实验证明：去掉 body 导致路由准确率暴跌 31–44pp。

核心洞察：Body 里的实现细节（参数、API、边界条件）才是区分「看起来相似、功能完全不同」的技能的关键信号。
SkillReducer — 优化层
✂️
结构化去膨胀

对 55,315 个公开技能的实证：26.4% 技能完全缺失描述；60%+ body 内容是非可执行的背景/示例；参考文件可能一次性注入数万 token。

核心洞察：Less-is-more — 移除干扰信息后，Agent 功能质量反而提升 2.8%。

融合架构全景

SkillRouter 负责 Runtime 精准路由，SkillReducer 负责 Build-time 结构化减肥，两者在「全文本索引」处交汇。

Bi-Encoder 检索 — 召回阶段

SR-Emb-0.6B 基于 全文本（name + desc + body）生成 Embedding，通过 FAISS ANN 从 80K 池中召回 Top-20。关键：训练时使用 10 个 Hard Negatives（语义邻居 4 + BM25 匹配 3 + 同分类干扰 2 + 随机 1），迫使模型学会区分高度相似的技能。

2

假阴性过滤 — 数据清洗

技能池中大量「同功能不同名」或「同名不同实现」的技能。直接当负样本会污染对比学习。采用三层过滤：名称去重 → 三元组 Jaccard > 0.6 → Embedding 相似度 > 0.92。过滤后贡献 +4.0pp Hit@1。

3

Cross-Encoder 重排序 — 精排阶段

SR-Rank-0.6B 对 20 个候选做完整文本交叉编码。关键：必须使用 Listwise Cross-Entropy。消融实验证明 Pointwise BCE 会崩溃到 43.3% Hit@1，而 Listwise 达到 74.0% — 因为 Top-20 候选全是 topical plausible，需要相对比较而非独立打分。

🟣 SkillReducer 详解

两阶段优化框架，将单体技能重构为「常驻 Core + 按需 Reference」的分层架构。

📉
发现 1：描述层失效

55,315 个公开技能中，26.4% 完全没有描述，44.1% 缺失或短于 20 tokens。这些描述仍被加载到路由池，但永远无法被匹配 — 每次调用都是纯 Token 浪费。

📊
发现 2：Body 内容冗余

对 15,107 个段落级 item 分类后发现：仅 38.5% 是可执行 Core Rules，40.7% 是背景解释，12.9% 是示例。超过 60% 的内容注入上下文后并不参与任务执行。

📁
发现 3：参考文件爆炸

14.8% 的技能包含外部参考文件。100 个 SkillHub 技能的参考文件总计 1.67M tokens，单次调用可能注入数万 token，而当前任务往往只需要其中一小部分。

Stage 1：路由层优化 — Delta Debugging

1a

对抗性模拟 Oracle

构建候选池：目标技能 + 4 个 TF-IDF 相似干扰项 + 1 个 LLM 生成的「影子技能」（同域不同功能）。Oracle 要求目标技能对所有测试 query 都被正确选中。

1b

语义单元级 ddmin

LLM 将描述分割为「语义单元」（如 "JWT authentication"），而非句子或单词。ddmin 递归二分查找，O(n log n) 次 Oracle 调用后得到 1-minimal 子集 — 每个剩余单元都是路由必需的。

1c

真实环境验证 + 选择性恢复

将压缩描述部署到 Claude Code CLI，通过 stream event 解析检测触发率。若失败，从删除单元中贪婪恢复最能提升触发率的单元（最多 3 步），确保 100% 路由等价。

Stage 2：Body 重组 — Progressive Disclosure

# 优化前：单体架构（Monolithic） Cost(s) = |description| + |body| + Σ|references| # 全部常驻上下文 # 优化后：分层架构（Tiered） Cost'(s) = |s'.d| + |b*| + Σr∈R*used|r| # 仅 Core 常驻，Refs 按需 read_file
2a

分类学驱动分类

LLM 将 body 段落分类为 5 类：Core Rule（可执行指令）、Background（原理解释）、Example（代码示例）、Template（ boilerplate）、Redundant（重复内容）。仅 Core Rule 保留在常驻模块。

2b

类型特定压缩 & 去重

Core Rules 合并为精简 bullet points；Examples 按概念去重仅保留最佳代表（减少 60-70%）；Background 摘要为保留关键事实的短段落；Templates 按概念去重。

2c

跨文件去重 & 注解

检测 reference 与 body 的内容重叠并删除冗余。每个 reference 附加 when 触发条件（如 "you need to write YAML"）和 topics 关键词，让 Agent 无需读取即可判断是否加载。

2d

双门控质量保障 + 自纠正反馈

Gate 1（Faithfulness）：LLM 检查所有操作概念是否保留在 Core ∪ References 中。
Gate 2（Task Eval）：5 个真实任务测试，Agent 使用 read_file 按需加载。若失败，分析缺失 criteria 并将所需内容 Promote 回 Core（最多 2 轮迭代）。

端到端模块设计

将论文方案落地为可部署的 4 层架构：存储层、路由服务、上下文装配层、Agent 执行层。

📦 1. 技能存储与索引层（Skill Store） ▼

不要只存 metadata。 按 SkillRouter 结论，索引必须包含完整 body；但按 SkillReducer 结论，body 应被预结构化。

# 建议的入库格式 skill_id: "jwt-auth-v2" routing: name: "JWT Authentication" description: "压缩后的1-minimal描述" # Stage 1 产物 embedding_text: "name + desc + core_rules摘要" # 用于 Bi-Encoder body: core: "可执行核心规则（≤40%原body token）" # Stage 2 产物 references: - id: "examples" when: "需要实现具体代码时" topics: ["code-example", "implementation"] content: "..." - id: "background" when: "需要理解安全原理时" topics: ["rationale", "security-context"] content: "..."

索引策略：离线用 Bi-Encoder（0.6B 级别）预计算 Embedding，存 FAISS/Milvus；建索引前跑假阴性过滤流水线（名称 + 三元组 Jaccard + Embedding 相似度）。

🎯 2. 路由服务（Skill Routing Service） ▼

A2A 平台里的核心服务，建议独立 GPU 部署：

class SkillRouter: def route(self, user_query: str, top_k: int = 5) -> List[Skill]: # Stage 1: 检索（全文本 Bi-Encoder） query_emb = self.encoder.encode(user_query) candidates = self.index.search(query_emb, k=20) # Stage 2: 重排序（列表式 Cross-Encoder） ranked = self.reranker.rank( query=user_query, candidates=[self.store.get_full_text(c) for c in candidates], loss="listwise_cross_entropy" # 不是 pointwise BCE！ ) return ranked[:top_k]

训练数据：用 GPT-4o-mini 基于 skill body 生成 synthetic query（避免泄露 skill 名称），确保 query 反映功能需求而非词汇匹配。每个 query 配 10 个 hard negatives。

🧩 3. 上下文装配层（Context Manager） ▼

Agent 执行前的关键层，直接决定 Token 成本和注意力质量：

class SkillContextManager: def assemble(self, skills: List[Skill], query: str) -> Context: ctx = { "system_prompt": base_prompt, "user_query": query, "injected_skills": [] } for skill in skills: ctx["injected_skills"].append({ "name": skill.routing.name, "description": skill.routing.description, "core_rules": skill.body.core, # 仅注入核心规则 "available_refs": [ {"id": r.id, "when": r.when, "topics": r.topics} for r in skill.body.references ] }) ctx["tools"].append(read_file_tool) # 供 Agent 按需加载 return ctx

为什么这样做？ SkillRouter 证明全文本是路由信号，但路由完成后 Agent 不需要看到全文本 — 它只需要 Core Rules + 按需 Reference。SkillReducer 证明 core-only 加载可将单次调用从 12,019 token 降到 540 token，且功能质量提升 2.8%。

关键决策矩阵

决策点 论文依据 建议方案
索引是否存完整 body？ SkillRouter：body 是 critical signal，去掉后暴跌 31–44pp 必须存。但 SkillReducer 优化后的 body（core+refs 摘要）比原始 monolithic body 更适合索引
描述缺失怎么办？ SkillReducer：26.4% 技能缺失描述，导致路由失效 用 body 自动生成描述，再经 Delta Debugging 压缩到 1-minimal
Reranker 用 pointwise 还是 listwise？ SkillRouter 消融：pointwise BCE 崩溃到 43.3%，listwise CE 达 74.0% 必须用 listwise CE。Top-20 候选全是 plausible，需相对比较
Agent 看到完整 body 还是 core？ SkillReducer：less-is-more，core-only 功能质量提升 2.8% Agent 只加载 core，references 通过 read_file 按需加载
假阴性过滤阈值？ SkillRouter：trigram Jaccard > 0.6 + emb > 0.92 直接复用三层过滤，避免同功能不同名技能污染训练

总结

这两篇论文为你的 deep agent + skill guru 系统提供了 「找得准」+「给得精」 的闭环：

  • 🔵 SkillRouter 告诉你：路由层必须用全文本做信号，且要检索 + 重排序两阶段，配合假阴性过滤和列表式损失才能在大规模同质技能池中不迷路。
  • 🟣 SkillReducer 告诉你：入库前必须做结构化减肥，将单体技能拆成 Core（常驻）+ References（按需），用 Delta Debugging 压缩描述，用任务反馈循环保证功能不丢。
  • 🟢 落实到你的架构，就是：Skill Guru 负责 build-time 优化（SkillReducer），A2A Proxy 负责 runtime 路由（SkillRouter），Agent Context Manager 负责按需装配（Progressive Disclosure）。这三层合起来，才能支撑你「1 万个 skills」的愿景。
