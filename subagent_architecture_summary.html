Subagent 启动逻辑

当 LLM 调用 Agent 工具时，系统会经过权限检查 → 实例准备 → 运行时克隆 → Agent 构建 → Soul 启动的完整流程。 只有 role="root" 的 agent 才能创建 subagent，这是一个硬限制。

启动流程图

flowchart TB A[用户/LLM 调用 Agent 工具] --> B{权限检查} B -->|role != root| C[拒绝: Subagents cannot launch other subagents] B -->|role == root| D{run_in_background?} D -->|No| E[ForegroundSubagentRunner.run] D -->|Yes| F[BackgroundAgentRunner.run] E --> G[_prepare_instance] G --> H{resume?} H -->|Yes| I[从 SubagentStore 恢复实例] H -->|No| J[生成 agent_id, 创建新实例] J --> K[SubagentBuilder.build_builtin_instance] K --> L[resolve_effective_model] L --> M[clone_llm_with_model_alias] M --> N[Runtime.copy_for_subagent] N --> O[load_agent YAML + system_prompt + tools] O --> P[prepare_soul] P --> Q[Context.restore 恢复历史] P --> R[写入 prompt 快照] P --> S[创建 KimiSoul] S --> T[run_with_summary_continuation] T --> U[soul.run 进入主循环] U --> V[提取 final_response 作为 summary] V --> W[返回 ToolOk] F --> X[创建 Background Task] X --> Y[prepare_soul 同上] Y --> Z[异步运行 soul] Z --> AA[完成后 Notification 通知 Root]

● Foreground 模式

  • • 阻塞式执行，父 agent 等待结果
  • • Wire 消息实时透传到父 UI（ApprovalRequest 直接透传）
  • • 其他事件包装为 SubagentEvent 发送
  • • 结果以字符串 summary 返回给父 agent
  • • 支持自动续写（输出太短会自动追加 prompt 重跑）

● Background 模式

  • • 非阻塞，立即返回 task_id 和 agent_id
  • • Wire 消息仅写入本地文件，不实时转发
  • • 完成后通过 NotificationManager 通知 root
  • • 支持通过 TaskOutput / TaskStop 管理
  • • Approval 状态独立追踪（awaiting_approval）

Runtime 共享与隔离

Runtime.copy_for_subagent() 是 subagent 启动的核心。它决定哪些组件共享、哪些隔离。 共享的组件通过同一个 Python 对象引用实现，隔离的组件则是全新实例。

Subagent 间通信机制

⚠️ 核心结论

Subagent 之间无法直接通信。当前架构是严格的层级委托模型：Root Agent 作为唯一中心， subagent 只能与父 agent 单向交互（prompt 进，summary 出）。

Foreground Subagent 消息流向

sequenceDiagram participant Root as Root Agent participant Wire as Subagent Wire participant Parent as Parent Wire participant UI as Root UI Root->>Wire: soul.run(prompt) Wire->>Wire: LLM 调用 + 工具执行 loop UI Loop (Foreground) Wire->>Parent: receive() alt ApprovalRequest / ToolCallRequest / QuestionRequest Parent->>UI: 直接透传 UI->>Parent: 用户响应 Parent->>Wire: 传回 subagent else HookRequest Wire->>Wire: 忽略 else 其他事件 (TextPart, ToolResult...) Parent->>Root: 包装为 SubagentEvent Root->>Root: 放入 context 作为 tool result end end Wire->>Root: 返回 final_response (summary)

Background Subagent 消息流向

sequenceDiagram participant Root as Root Agent participant BRunner as BackgroundAgentRunner participant Wire as Subagent Wire participant Notify as NotificationManager Root->>BRunner: create_agent_task() Root->>Root: 立即返回 ToolOk (task_id) BRunner->>Wire: soul.run(prompt) Wire->>Wire: 异步执行... loop UI Loop (Background) Wire->>Wire: 仅写入 output 文件 Note over Wire: 不实时转发到父 wire end Wire->>BRunner: 执行完成 BRunner->>Notify: publish_terminal_notifications() Notify->>Root: NotificationEvent (下次 LLM 调用前检查) Root->>Root: 将通知追加到 context

间接通信方式

📁 文件系统

通过 WriteFile / ReadFile 读写共享目录（如 session 目录） 实现跨 agent 的数据交换

🔄 父 Agent 中转

Root agent 读取一个 subagent 的返回结果， 再作为 prompt 传给另一个 subagent

📂 additional_dirs

共享同一个 Python list 引用，通过 /add-dir 添加的目录 对所有 agent 立即可见

Agent Team vs Subagent

Agent Team（智能体团队） 是一个更高级的多智能体协作概念：多个 agent 以对等（peer-to-peer）或集群方式协作， 共享状态、动态分配任务、相互通信。当前 Kimi CLI 的 subagent 模型还远未达到这个层次。

如果要构建 Agent Team

基于当前代码架构，如果要演进到 Agent Team，需要新增以下核心组件：

📡

Team Bus / Shared Channel

当前 Wire 是点对点的（soul ↔ UI）。Team 模型需要 publish-subscribe 机制， 新增 TeamMessage Wire 事件类型，让 teammates 订阅共享频道。

🎯

TeamCoordinator

扩展当前的 LaborMarket（静态类型注册表）为动态 team 管理器，维护 team_members， 负责任务分解、结果聚合、冲突仲裁。

🗃️

TeamStore

当前 SubagentStore 是每个 agent 独立的。Team 模型需要共享状态层： 共享上下文片段（全局 todo、共享知识库）、文件锁或变更日志。

👥

动态角色招募

恢复旧版 CreateSubagent 能力并扩展为 JoinTeam / RecruitAgent， 允许运行时根据任务需求动态组建团队。

总结

Kimi CLI 当前的 subagent 架构是一个设计良好的"主从委托"系统，适合将独立子任务交给专门化的代理执行， 但它不是一个多智能体团队（Agent Team）系统：

✅ 当前支持

  • • 并发启动多个 subagent
  • • 前后台执行、持久化、恢复
  • • 审批透传和 Wire 事件代理
  • • 独立的上下文隔离

❌ 当前不支持

  • • Subagent 间直接通信
  • • 动态角色创建
  • • 共享状态或协调原语
  • • 嵌套委托

如果要演进为 Agent Team，最大的架构缺口是：peer-to-peer 通信机制、共享状态层、 动态团队编排器。这些在当前的 LaborMarket、SubagentStore 和 Wire 协议中都没有设计痕迹。
