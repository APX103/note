01为什么必须 Multi-Agent？

单 Agent 架构在简单任务（查日历、算数、搜索）上表现良好，但一旦涉及多工具、多步推理、跨领域验证，性能会迅速劣化。核心原因不是模型智商不足，而是上下文窗口的物理限制。

1.1 单 Agent 的硬边界

⚠️

上下文过载

单 Agent 无法同时保持整个代码库、知识库在上下文中，长程任务早期目标被遗忘。

🔧

工具路由混乱

工具越多，路由决策越混乱，错误调用率随工具数量指数上升。

🎭

角色冲突

一个 Agent 同时承担规划、执行、验证，prompt 中角色冲突导致输出质量下降。

📉

长程退化

LLM 表现随上下文膨胀而劣化，准确率随任务步骤增加呈下降趋势。

💡 核心洞察

团队模式（Team Mode）的本质，是给每个 Agent 狭窄、聚焦的上下文，让它们在自己的专业域内达到最佳表现。一个同时理解前端、后端、数据库和测试的 Agent，其输出质量显著低于只专注一个领域的 Agent。

1.2 多 Agent 的四大核心优势

维度 单 Agent 多 Agent 团队
问题分解 线性处理，容易在复杂依赖中迷失 分布式子问题求解，每个 Agent 处理特定域
专业化 通用提示，难以深度优化 每个 Agent 有专属角色、工具集、系统提示
可扩展性 功能增加导致 prompt 膨胀 通过新增 Agent 扩展能力，模块化水平扩展

视角多样性 单一推理路径 多个专家从不同角度分析，减少盲区

02Agent 原语设计

当前主流框架（AutoGen、CrewAI、LangGraph）的共同问题是：每个新任务都需要手工设计 Agent 角色和交互协议。MetaGPT 模拟软件工程团队，给每个 Agent 分配产品经理、架构师、工程师等角色；CAMEL 设计特定的对话流程。这些方案有效，但高度任务相关，难以复用。

2.1 什么是 Agent 原语

🧬 核心类比

就像神经网络用残差块和注意力头构建复杂模型一样，多 Agent 系统也应该由少量可复用的"原语"构建。原语是对 recurring computation patterns 的封装，对外暴露标准 LLM Agent 接口，支持即插即用。

2.2 三大核心原语（Agent Primitives）

通过对现有 MAS 设计的实证分析，学术界发现绝大多数架构可以分解为以下三种 recurring patterns：

🔍

Review Primitive（评审原语）

结构：Solver Agent + Critic Agent，通过潜在反馈通道连接

机制：Solver 生成 KV Cache，Critic 直接消费 latent representation 生成修正反馈，修改后的 latent state 回传 Solver 迭代 refine

停止：由中间 latent state 推导自适应停止

本质：将 Self-Refine、Multi-Agent Debate 抽象为可复用 latent operator

🗳️

Voting & Selection（投票选择）

结构：N 个并行 Solver + 1 个 Selector 模块

机制：每个 Solver 独立生成 KV Cache 候选，Selector 在潜在空间而非文本空间进行投票和选择

优势：避免文本级多数决的信息损失，直接在语义空间融合多视角推理

本质：将 Ensemble / Majority Vote 抽象为 latent aggregator

📋

Planning & Execution（规划执行）

结构：Planner Agent + Executor Agent

机制：Planner 生成 latent plan（KV Cache 形式的结构化步骤），Executor 消费该计划进行任务特定推理

优势：计划与执行解耦，计划可复用、可验证、可回滚

本质：将 ReAct / Plan-and-Solve 抽象为可复用分解算子

🔑 关键设计约束

这三个原语的核心约束是：内部协调完全通过 KV Cache 交换实现，不依赖文本级消息传递。这确保原语行为独立于自然语言协议，同时对外暴露标准 LLM Agent 接口，可以在任何多 Agent 框架中即插即用。

2.3 工程实践中的其他关键原语

  • fire-and-forget：派发任务不等待结果
  • wait-for-result：同步等待子 Agent 完成
  • parallel handlers：并行派发多个子任务

这是 OpenAI Agents SDK 中 handoff 机制的底层抽象，也是 Cognition Core 设计中的核心委托模式。

OpenClaw 中的三原语设计：

  • agentId：独立工作区 + 认证 + 会话存储
  • accountId：渠道账号实例
  • bindings：确定性消息路由规则

人格、边界、团队规则沉淀在工作区文件（AGENTS.md、SOUL.md、USER.md）而非内联 prompt 中。

Cursor 和 Anthropic 的 Agent Swarm 都不约而同选择 Git 而非自定义消息总线：

  • 每个 Worker 在独立 Git worktree / Docker 中工作
  • 通过 git commit / git merge 实现结构化集成
  • 用 lock file（current_tasks/xxx.txt）在 Git 层面实现任务认领的原子性
  • 最终一致性：接受临时分歧，让系统自然收敛

核心洞察：Agent 不适合作为强一致性锁的持有者（它们会崩溃、会幻觉、会忘记释放锁），因此"接受临时分歧，后期协调"是更务实的模型。

2.4 潜在空间通信：从"说人话"到"直接传思维"

传统多 Agent 系统用自然语言通信，存在根本性瓶颈：信息在文本编解码中逐级降级，噪声累积导致准确率衰减。

指标文本通信KV Cache 通信
Token 使用基准（3–4x 开销）减少 3–4 倍
推理延迟高（文本生成 + 解析）显著降低
信息保真逐级降级多阶段保持稳定
整体开销单 Agent 的 3–4 倍仅 1.3–1.6 倍

这意味着，未来团队模式 Agent 的"内部沟通"可能不需要自然语言，而是直接在模型的 latent space 中传递 KV Cache。这对架构设计有深远影响：Agent 间的通信协议需要从"文本协议"升级为"潜在空间协议"。

03多 Agent 编排模式

3.1 五大核心编排架构

模式 结构 适用场景 代表框架
Planner-Executor 规划者分解 → 执行者逐步执行 需要明确步骤分解的任务 ReAct 泛化、SmolAgents
Hierarchical 监督者路由到专家池，监督者不执行 复杂工作流、企业级治理 CrewAI、OpenAI Swarm
Maker-Checker 生成者产出 → 验证者评分/重写 高 stakes 领域（法律、医疗） 评审原语、Evaluator-Optimizer
Pipeline 固定序列：A → B → C，每个转换前一个输出 确定性流程（内容发布、数据处理） LangGraph 线性链
Network/Swarm 对等节点共享草稿本，自由讨论至共识 探索性任务（研究、头脑风暴） AutoGen GroupChat、CAMEL

3.2 拓扑结构：三种组织形态

🔗 Mesh（对等网络）

无中心节点，Agent 广播消息，分布式决策。适合容错要求高、需要分布式共识的场景。风险：O(n²) 通信复杂度，容易出现循环和冗余 chatter。

👑 Hierarchical（女王-工蜂）

中心协调者（Queen/Lead）分解任务，派发给专业工作者。工作者可点对点通信（如前端 Agent 直接问后端 Agent API 契约），无需全部经过中心。适合明确分工的复杂项目。

🔄 Adaptive（自适应）

根据任务复杂度动态切换拓扑：简单任务用 Pipeline，复杂任务展开为 Hierarchical，探索任务切换为 Mesh。目前主要在实验性框架中实现，但代表了未来方向。

3.3 执行模式：声明式 vs 自主式

Workflow Mode（声明式）

YAML 声明 epistemic sequence：retrieve → classify → investigate → deliberate → govern。执行引擎按固定序列调用原语，每个步骤命名原语、指定参数、定义转移条件。适合合规要求高、流程可预见的场景。

Agentic Mode（自主式）

编排器（Orchestrator）根据目标 + 累积证据自主选择轨迹。YAML 只声明可用原语词汇、硬约束（must_include、max_steps、must_end_with）和 goal。执行引擎在底层强制硬约束，但不干预编排器的路径选择。适合问题结构未知、需要探索的复杂任务。

两种模式共享同一套原语、治理模型和审计账本。

05通信与协调机制

5.1 通信协议层

结构化消息（Structured Messaging）

纯文本通信是 Agent 系统的主要 bug 来源：解析歧义、格式不一致、幻觉注入。工业界共识：使用 Pydantic 验证的强类型消息，每个消息定义明确的 schema。消息可见性基于 FSM 状态过滤，确保 Agent 只接收与其当前角色和阶段相关的消息。

标准化协议：

  • A2A（Agent2Agent）：Google 推动的开放协议，定义 Agent 能力发现、任务委托、上下文传递
  • ACP（Agent Communication Protocol）：IBM 的提案
  • ANP / AWCP：学术界的互操作协议探索

这些协议的目标是构建"Agent 互联网"（Internet of Agents），让不同框架、不同厂商的 Agent 能够相互发现和调用。

5.2 状态同步层

🌿 Git 作为协调协议

Git 提供最终一致性 out of the box。分支机制天然支持隔离工作区，合并冲突的解决流程是并发编辑的标准化解法。Lock file 利用 Git 的原子性实现任务认领。

📌 共享任务列表

状态：pending → in-progress → completed。依赖跟踪：当依赖任务完成时，下游任务自动 unblock。类比人类团队的 Jira/Trello。

📝 黑板系统（Blackboard）

共享知识工作空间，Agent 异步读写。适合需要累积中间发现、逐步收敛的协作任务。与消息传递的区别：黑板是"拉取"模型，消息传递是"推送"模型。

5.3 隔离与执行层

隔离工作区：CAID 使用 git worktree 为每个 Agent 创建独立工作区；Anthropic 使用 Docker 容器隔离。目的：防止并发编辑互相干扰，支持独立验证后再集成。

测试驱动的集成：CAID 的核心创新：Agent 完成任务后必须通过可执行测试验证，才能合并到主分支。这相当于给 Agent 团队设置了 CI/CD gate，避免"看起来对但跑不通"的代码合并。

06设计原则与反模式

6.1 何时用多 Agent，何时用单 Agent

决策规则

单 Agent：任务简单、步骤少、工具少、无需跨领域验证。
多 Agent：任务需要专业化角色、多步推理、强验证、跨工具/工作流协调。
混合架构（最推荐）：中央编排层根据任务复杂度路由到单 Agent 或 Agent 团队。

6.2 关键设计决策清单

#决策项推荐选择理由
1拓扑选择层级式适合明确分工，网状适合探索性任务不要为简单任务硬上 Mesh
2通信粒度结构化消息 > 自由文本自由文本适合人机交互，不适合 Agent 间交互
3状态管理显式状态图 > 隐式对话上下文后者在规模扩大后不可调试
4一致性模型最终一致性 > 强一致性Agent 是不可靠的锁持有者
5人机协作设计阶段定义 handoff 点明确"什么数据、什么格式、什么条件下转人"

6.3 反模式（Anti-Patterns）

反模式表现后果
过度设计简单任务硬拆 5 个 Agent延迟和成本倍增，无质量收益
对话混乱Agent 自由聊天无明确目标循环论证、冗余 chatter、token 爆炸
状态黑洞状态隐式存储在对话历史中无法恢复、无法调试、无法审计
强一致性幻觉让 Agent 持有分布式锁锁泄漏、死锁、不一致
角色膨胀一个 Agent 被赋予 3+ 不相关角色上下文冲突，输出质量劣化

07前沿趋势

🧠 潜在空间协作

KV Cache 通信只是开始，未来可能出现跨模型家族的 latent interoperability，让不同架构的 Agent 在共享连续空间中协作，无需文本翻译。

🤖 自动系统构建

Agent Primitives 中的 Organizer + Knowledge Pool 展示了自动选择原语组合的可能性。未来可能实现"输入任务描述，自动输出最优 Agent 团队架构"。

📡 标准化协议战争

A2A、ACP、ANP、AWCP 等协议正在竞争成为"Agent 互联网"的 TCP/IP。对架构师来说，协议层抽象比框架选择更重要。

🔍 可观测性成为瓶颈

调研显示，可观测性是生产 adoption 的 #1 障碍。团队模式 Agent 需要 span 语义、分布式链路追踪、Agent 级指标监控。

🛡️ Agent 互联网安全

随着 Agent 跨平台互操作，注册、发现、资源审查、治理成为新的安全边界。必须与能力协同设计的架构原则。

∑总结

团队模式的 Agent 原语与多 Agent 编排，本质上是在解决"如何让多个有限上下文的专业 Agent，像人类团队一样协作完成超出现有 LLM 上下文窗口的复杂任务"。

必要性
单 Agent 在长程、跨域、高验证任务上存在物理瓶颈，多 Agent 优势已获实证验证。
原语
Review、Voting/Selection、Planning/Execution 三种原语 + 委托/路由/消息/Git 等工程抽象。
编排
状态机、角色驱动、对话驱动三种哲学各有适用域，生产系统趋向"混合架构"。
通信
从自然语言文本 → 结构化消息 → 潜在空间 KV Cache，通信效率和稳定性持续提升。
